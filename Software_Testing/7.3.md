7.3 是关于**自动化跑测试的机制**——也就是"谁来负责跑测试？什么时候跑？跑失败了会发生什么？"。这是把你前面学的所有东西（test framework, lifecycle hooks）串起来，接入到真实团队协作的工作流程里。

## 1. High-Level Map：这个概念在整个体系里的位置

回想一下 [[7.1]]，我们学的是 test framework 怎么帮你 discover、跑、report 测试；[[7.2]] 教你怎么用 hooks 管理每个测试的 setup/teardown。但到目前为止，这些测试都是**你自己手动在电脑上跑**的（比如你在 terminal 打一个指令）。

7.3 要解决的问题是：**如果测试要靠人手动去跑，人是会偷懒、会忘记的**。所以我们需要一个系统，在特定时机**自动帮你跑测试**——这就是 **Continuous Integration (CI)**。

这个概念坐落在"软件工程流程" (software engineering workflow) 这个更大的框架里：写代码 → 测试 → 合并代码 → 部署。CI 负责的是中间"测试"这一段的自动化。

## 2. Fundamentals First：为什么需要 CI？

想象一个团队，5 个工程师同时在改同一个项目的代码。如果没有自动化测试：

- 工程师 A 改了一个功能，*他自己*测过没问题，就把代码合并 (merge) 进主分支 (main branch)
- 但他的改动其实**悄悄弄坏了**另一个工程师 B 写的功能——A 不知道，因为他没有跑 B 的测试
- 几天后，团队才发现 production 上的功能坏了，但已经很难追溯到底是哪一次改动搞的

这就是"人靠自觉"的失败模式。CI 解决的核心问题是：**每一次代码有改动，系统都自动帮你把所有测试都跑一遍，而且立刻告诉你结果**——不用等人手动去跑，也不会漏掉。

## 3. Progressive Explanation：一个一个概念拆解

### (a) Continuous Integration (CI)

CI 的字面意思是"持续集成"——工程师**频繁地**（比如一天好几次）把自己的代码改动"集成" (integrate) 到共享的主分支里，而不是攒了一大堆改动才一次性合并。

但"集成"本身是有风险的（可能会跟别人的代码冲突、可能弄坏别的功能），所以 CI 系统的核心工作就是：**每次有人要集成代码时，自动跑一遍所有测试，确认没有破坏任何东西**。

你可以把 CI 想成一个**不知疲倦、永远认真、24 小时待命的品管员 (QA)**——只要有新代码进来，它立刻检查。

### (b) Automated Test Run（自动化跑测试）

这就是 CI 实际在做的动作：不需要人手动打指令，系统自己触发 [[7.1]] 学过的 test runner，去跑整个 test suite。

触发的时机通常是：
- 有人 push 代码到某个分支
- 有人开了一个 Pull Request（等一下解释）

### (c) Build Pipeline（构建流水线）

"跑测试"往往不是唯一要做的事。CI 系统通常会按顺序执行一连串步骤，这一整串步骤叫做 **pipeline**（管线/流水线），例如：

```
1. 下载最新代码
2. 安装依赖 (install dependencies)
3. 编译/打包代码 (build)
4. 跑测试 (run tests)   <-- 我们关心的就是这一步
5. (可选) 产生报告
```

之所以叫"pipeline"，是因为这些步骤像工厂的生产线一样**一步接一步**，前一步失败，后面就不会继续跑。

### (d) Fail Fast（快速失败）

这是 CI 的一个重要设计原则：**一旦流水线中有任何一步失败，立刻停下来，马上通知工程师**，不要等到整个流程跑完才说。

为什么这很重要？想象如果第 2 步"安装依赖"就已经失败了，但系统还是傻傻地跑完第 3、4、5 步——这是浪费时间和运算资源。Fail fast 让你**越早知道问题，修复的成本就越低**（这跟你在 debugging 早期学到的原则是一致的：bug 越晚被发现，修起来代价越高）。

### (e) Pull Request Gate（PR 关卡）

这是把 CI 跟团队协作流程结合起来的关键机制。**Pull Request (PR)** 是工程师"请求"把自己的代码改动合并进主分支的动作。

"Gate"（关卡）的意思是：**CI 会被设置成——如果测试没有全部通过，这个 PR 就不准被合并**。

流程通常长这样：

```
工程师提交 PR
      ↓
CI 自动触发，跑所有测试
      ↓
   测试通过？
   ↙        ↘
 是            否
 ↓              ↓
允许合并      拒绝合并，
             显示哪个测试失败了
```

这个机制确保了：**任何进入主分支的代码，都是已经被测试验证过的**——从根本上防止了"没测试就上线"这种情况发生。

## 4. Scope 提醒

这一节**不包括** CD (Continuous Deployment/Delivery)——也就是测试通过之后，代码怎么自动部署到 production 环境。那是下一步的话题，7.3 只专注在"测试怎么被自动触发、自动把关"这一段。

## 5 个你必须完全掌握的核心概念

**1. Continuous Integration (CI)**
- 定义：工程师频繁地把代码改动合并进共享分支，并且每次合并都自动验证不会弄坏东西。
- 为什么重要：让团队协作时不会因为"没人跑测试"而把坏代码带进主分支。
- 检验问题：如果没有 CI，5 个工程师同时改代码，最可能出现什么问题？
  *答案：某人的改动悄悄弄坏了别人的功能，但因为没有自动测试，这个问题要很久之后才会被发现。*

**2. Automated Test Run**
- 定义：不需要人手动操作，系统自己在特定时机触发测试执行。
- 为什么重要：消除"人会忘记跑测试"这个风险，确保每次改动都被检查。
- 检验问题：自动化测试通常在什么时机被触发？
  *答案：当有人 push 代码，或开一个 Pull Request 的时候。*

**3. Build Pipeline**
- 定义：CI 系统按顺序执行的一连串步骤（下载代码、装依赖、build、跑测试等）。
- 为什么重要：把"从拿到代码到验证代码正确"这整个过程标准化、自动化。
- 检验问题：如果 pipeline 的"安装依赖"这一步失败了，后面的"跑测试"步骤还会继续执行吗？
  *答案：不会，因为遵循 fail fast 原则，前面失败就会停止，不浪费资源跑后面的步骤。*

**4. Fail Fast**
- 定义：只要流程中有一步失败，立刻停止并通知，不必等整个流程跑完。
- 为什么重要：越早发现问题，修复成本越低；也避免浪费时间跑注定失败的后续步骤。
- 检验问题：为什么"等所有步骤都跑完才告诉你哪里错了"是一个不好的设计？
  *答案：因为这样会浪费运算资源在注定失败的流程上，而且工程师要等更久才能知道出了什么问题，拖慢修复速度。*

**5. Pull Request Gate**
- 定义：把 CI 测试结果当作"关卡"——测试没过，代码就不能合并进主分支。
- 为什么重要：从制度上保证只有"验证过"的代码才能进入主分支，防止人为疏忽导致坏代码上线。
- 检验问题：如果一个 PR 的测试没有全部通过，系统应该允许它被合并吗？
  *答案：不应该——PR gate 的作用就是阻止测试未通过的代码被合并进主分支。*
