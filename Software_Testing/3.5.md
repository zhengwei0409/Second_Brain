
前面所有的技术（EP、BVA、Decision Table）都假设：**给定相同的输入，程序总是返回相同的输出。**

但有些系统不是这样——**同一个操作，在不同的历史状态下，会产生不同的结果。**

**State Transition Testing 解决的就是：怎么测这类"行为取决于过去发生了什么"的系统？**

## 1. 问题：同一个输入，不同结果

考虑一个交通灯系统。你发出"切换"指令：

- 当前是红灯 → 变绿灯
- 当前是绿灯 → 变黄灯
- 当前是黄灯 → 变红灯

同样是"切换"这个输入，结果完全不同——取决于**系统当前处于什么状态**。

再比如：用户账号系统。

- 账号是 Active 状态 → 登录成功
- 账号是 Locked 状态 → 登录失败（即使密码正确）

同样是"输入正确密码"这个操作，结果因状态而异。

这类系统叫做 **stateful system（有状态系统）**。EP 和 BVA 对这类系统无能为力，因为它们只关注输入值，不关注系统历史。

---

## 2. State 是什么

**State（状态）** = 系统在某一时刻的"模式"或"条件"，决定系统对下一个输入会如何响应。

状态是系统的**记忆**——它记录了过去发生的事情，并影响未来的行为。

**例子：**

| 系统 | 可能的 State |
|------|------------|
| 交通灯 | Red、Green、Yellow |
| 用户账号 | Active、Locked、Suspended、Deleted |
| 音乐播放器 | Stopped、Playing、Paused |
| 订单 | Pending、Paid、Shipped、Delivered、Cancelled |

**在任何时刻，系统只处于一个 State。**

---

## 3. Transition 是什么

**Transition（转换）** = 系统从一个 State 移动到另一个 State 的过程。

Transition 由两部分组成：
- **起始 State**：系统从哪里出发
- **目标 State**：系统到达哪里

```
  [State A] ──────────────────→ [State B]
              (发生了某件事)
```

**例子：**

```
[Active] ──── 输错密码 3 次 ────→ [Locked]
[Locked] ──── 管理员解锁 ────────→ [Active]
```

---

## 4. Event / Trigger 是什么

**Event（事件）** 或 **Trigger（触发器）** = 引起 Transition 发生的动作或条件。

没有 Event，Transition 不会发生——系统会一直停留在当前 State。

```
  [Playing] ──── 用户按暂停键（Event）────→ [Paused]
  [Paused]  ──── 用户按播放键（Event）────→ [Playing]
  [Paused]  ──── 用户按停止键（Event）────→ [Stopped]
```

**State、Event、Transition 三者的关系：**

> **当系统处于 State X，发生 Event E，系统转换到 State Y。**

这是 State Transition Testing 的基本单元。

---

## 5. State Diagram（状态图）

**State Diagram（状态图）** = 用图形表示系统所有 State 和 Transition 的可视化工具。

构成元素：
- **圆圈** = State
- **箭头** = Transition
- **箭头上的标签** = Event / Trigger

**例子：用户账号系统状态图**

```
                    注册成功
  [Start] ──────────────────────────→ [Active]
                                          │
                              输错密码3次  │           管理员解锁
                                          ↓               ↑
                                      [Locked] ───────────┘
                                          │
                                   管理员封禁
                                          ↓
                                    [Suspended]
                                          │
                                   用户删除账号
                                          ↓
                                    [Deleted]
```

状态图让你一眼看清：
- 系统有哪些 State
- 什么 Event 触发什么 Transition
- 从哪个 State 出发，能到达哪些 State

---

## 6. Invalid Transition

**Invalid Transition（无效转换）** = 在某个 State 下，某个 Event 不应该发生或不应该被接受。

测试 Invalid Transition 和测试 Valid Transition 同样重要——你需要验证系统**拒绝了不该发生的事**。

**例子：**

| 当前 State | Event | 是否有效？ |
|-----------|-------|----------|
| Deleted | 登录 | ❌ 无效——已删除的账号不能登录 |
| Locked | 正常使用功能 | ❌ 无效——锁定账号不能操作 |
| Stopped | 暂停 | ❌ 无效——没在播放，无法暂停 |

如果系统没有正确拒绝 Invalid Transition，就是一个 bug。

---

## 7. 完整例子：ATM 取款机

**规格说明：**

ATM 有以下状态和规则：
- 开机后进入 **Idle**（待机）
- 插入银行卡 → 进入 **Card Inserted**
- 输入正确 PIN → 进入 **Authenticated**
- 输入错误 PIN 3 次 → 进入 **Card Captured**（吞卡）
- 完成交易或取消 → 回到 **Idle**

### State Diagram

```
         插卡
[Idle] ──────────→ [Card Inserted]
  ↑                      │              PIN 错误 3 次
  │          正确 PIN     ↓                   ↓
  │        ┌──────── [Authenticated]    [Card Captured]
  │        │               │
  └────────┘        取款/取消
         交易完成
```

### 完整状态转换表

| 当前 State | Event | 目标 State | 是否有效 |
|-----------|-------|-----------|---------|
| Idle | 插卡 | Card Inserted | ✓ |
| Card Inserted | 输入正确 PIN | Authenticated | ✓ |
| Card Inserted | 输错 PIN 3 次 | Card Captured | ✓ |
| Authenticated | 完成交易 | Idle | ✓ |
| Authenticated | 取消 | Idle | ✓ |
| Idle | 输入 PIN | — | ✗ 无效（还没插卡） |
| Card Captured | 继续操作 | — | ✗ 无效（卡已被吞） |

### 测试用例

从状态转换表里，每一行有效 Transition 提取一个测试用例：

| 测试用例 | 步骤 | 期望结果 |
|---------|------|---------|
| TC1 | Idle → 插卡 | 进入 Card Inserted 状态 |
| TC2 | Card Inserted → 输入正确 PIN | 进入 Authenticated 状态 |
| TC3 | Card Inserted → 输错 PIN 3 次 | 进入 Card Captured，卡被吞 |
| TC4 | Authenticated → 完成交易 | 退卡，回到 Idle |
| TC5 | Authenticated → 取消 | 退卡，回到 Idle |
| TC6（Invalid）| Idle → 直接输入 PIN | 系统拒绝，不发生转换 |

---

## 8. 测试用例怎么从状态图里提取

**系统方法：**

**策略一：All States Coverage** — 确保每个 State 至少被访问一次。

**策略二：All Transitions Coverage** — 确保每条有效 Transition 至少被触发一次（更强，推荐）。

**策略三：Invalid Transitions** — 为每个 State 测试那些不应该发生的 Event，验证系统正确拒绝。

实际工作中，**策略二 + 策略三** 是标准做法。

---

## 第3章总结：四种技术的分工

| 技术 | 解决什么问题 | 核心工具 |
|------|------------|---------|
| EP (3.2) | 单个输入范围太大 | 等价类划分，每类选一个代表值 |
| BVA (3.3) | 边界处容易有 off-by-one bug | 测 min-1、min、max、max+1 |
| Decision Table (3.4) | 多个条件组合 | 列出所有 Rule，每列一个测试用例 |
| State Transition (3.5) | 行为依赖历史状态 | 状态图，测所有 Transition + Invalid Transition |

**它们都是 Black-box 技术**，依据规格设计测试，不看源代码。

## Self-Check

**Q1.** State Transition Testing 解决的核心问题是什么？EP 和 BVA 为什么处理不了这类问题？

> **A:** State Transition Testing 处理"行为依赖历史状态"的系统——同一个输入在不同状态下产生不同结果。EP 和 BVA 只关注输入值的选择，假设相同输入总产生相同输出，对有状态系统无效。

---

**Q2.** State、Event、Transition 三者的关系是什么？用一句话描述。

> **A:** 当系统处于某个 **State**，发生某个 **Event**，系统执行 **Transition** 进入新的 State。三者缺一不可——没有 Event，Transition 不会发生；没有当前 State，Transition 没有起点。

---

**Q3.** 为什么 Invalid Transition 也要测？

> **A:** Invalid Transition 测试验证系统是否正确**拒绝**了不该发生的操作。如果系统在错误的 State 下接受了不该接受的 Event（比如已删除的账号能登录），这是 bug。只测有效路径，会遗漏这类安全性和正确性问题。

---

**Q4.** 音乐播放器有三个状态：Stopped、Playing、Paused。请列出所有有效的 Transition。

> **A:**
>
> | 当前 State | Event | 目标 State |
> |-----------|-------|-----------|
> | Stopped | 按播放 | Playing |
> | Playing | 按暂停 | Paused |
> | Playing | 按停止 | Stopped |
> | Paused | 按播放 | Playing |
> | Paused | 按停止 | Stopped |
>
> （共 5 条有效 Transition）

---

**Q5.** 什么是 All Transitions Coverage？它比 All States Coverage 强在哪里？

> **A:** All Transitions Coverage = 每条有效 Transition 至少被触发一次。All States Coverage = 每个 State 至少被访问一次。前者更强，因为你可以访问一个 State 却从未测过它的某些出边（离开该 State 的 Transition）。All Transitions Coverage 确保每条路径都被走过，不会遗漏某些 Transition 上的 bug。
