## 1. 什么是 Test Case？

**Test Case（测试用例）** 是一个 **有完整结构的测试单元**，它描述：

- 在什么条件下（precondition）
- 给程序什么输入（input）
- 程序应该产生什么结果（expected output）
- 以及怎么判断测试是否通过（pass/fail criteria）

一个 test case = 一次对程序特定行为的有组织的验证。

---

## 2. Input（输入）

**Input** 是你传给程序的数据，用来触发你想测试的行为。

```python
# 测试一个计算折扣的函数
price = 100
discount_rate = 0.2

# 这里 price 和 discount_rate 就是 input
result = calculate_discount(price, discount_rate)
```

Input 可以是：
- 函数的参数
- 用户在 UI 上填写的表单值
- 从文件或数据库读到的数据
- HTTP 请求的 body

**注意：** Input 必须具体。"输入一个负数"不够，要写"输入 `-5`"。

---

## 3. Precondition（前置条件）

**Precondition** 是运行测试 **之前**，系统必须处于的状态。

有些测试依赖某个初始状态，如果状态不对，测试结果就没有意义。

**例子：**

> 测试"用户登出"功能

- Precondition：用户必须已经登录（否则根本没有"登出"可测）
- Input：点击"登出"按钮
- Expected Output：用户被跳转到登录页面

如果没有 precondition，你就无法保证测试是在正确的起点上运行的。

**Precondition 的常见形式：**
- 数据库里有某条记录
- 用户已登录
- 某个配置开关是开启状态
- 文件已存在于某路径

---

## 4. Expected Output（预期输出）

**Expected Output** 是根据需求，程序 *应该* 返回的结果。

这是测试里最关键的部分。**没有 expected output，你就不知道测试是否通过。**

```python
# Input: price=100, discount_rate=0.2
# Expected Output: 80.0
```

Expected output 必须：
- **具体** — 不能写"应该返回一个合理的数字"
- **来自需求** — 不是你猜的，是规格说明或产品文档里写的
- **可以验证** — 能被程序自动比较

---

## 5. Actual Output（实际输出）

**Actual Output** 是程序 **实际运行后** 产生的结果。

```python
result = calculate_discount(100, 0.2)
# 假设程序有 bug，实际返回了 20.0（错误地计算成了折扣金额而不是折后价）
# Actual Output: 20.0
```

Actual output 是在运行时才知道的，你事先不知道它是什么。

测试的判断逻辑：
```
if actual_output == expected_output:
    → PASS
else:
    → FAIL
```

---

## 6. Pass / Fail Criteria（通过/失败标准）

**Pass / Fail Criteria** 是判断测试结果的规则。

最简单的标准：`actual output == expected output`

但有时标准更复杂：

| 场景 | Pass/Fail 标准 |
|------|--------------|
| 计算结果 | `result == 80.0` |
| 响应时间 | `response_time < 200ms` |
| 错误信息 | `error_message == "Invalid input"` |
| 状态码 | `HTTP status == 200` |
| 数据库记录 | 数据库里存在新增的那条记录 |

标准必须在 **运行测试之前** 就定好，不能在看到结果后才决定"这算通过还是失败"。

---

## 7. Assertion（断言）

**Assertion** 是在代码里实际执行"比较 expected 和 actual"这个动作的语句。

它就是 pass/fail criteria 的代码实现。

```python
# Python 示例
result = calculate_discount(100, 0.2)
assert result == 80.0  # ← 这就是 assertion
```

如果 assertion 为真（True）→ 测试通过  
如果 assertion 为假（False）→ 测试失败，抛出错误

**Assertion 是测试的"法官"** — 它做出最终判决。

没有 assertion 的"测试"根本不是测试。比如你运行了程序，但没有用 assertion 去检查结果，那你只是"运行了程序"，不是"测试了程序"。

---

## 8. 把所有部分放在一起：完整例子

**场景：** 测试一个计算折后价的函数 `calculate_discount(price, rate)`

| 部分 | 内容 |
|------|------|
| **Precondition** | 无（这个函数是纯计算，不依赖外部状态） |
| **Input** | `price = 100`, `discount_rate = 0.2` |
| **Expected Output** | `80.0`（100 减去 20% 折扣） |
| **Actual Output** | 运行后得到（假设为 `80.0`） |
| **Pass/Fail Criteria** | `actual == 80.0` |
| **Assertion** | `assert result == 80.0` |
| **Result** | PASS ✅ |

**代码形式：**

```python
def test_calculate_discount():
    # Precondition: none
    # Input
    price = 100
    rate = 0.2

    # Run
    result = calculate_discount(price, rate)

    # Assertion (expected output = 80.0)
    assert result == 80.0
```

## Self-Check

**Q1.** 以下哪个是 Precondition，哪个是 Input？

> 测试场景：测试"购物车结算"功能
> - (a) 购物车里已经有 3 件商品
> - (b) 用户点击"立即结算"按钮

**A1.**
- (a) = **Precondition**：这是测试开始前系统必须处于的状态（购物车已有商品）
- (b) = **Input**：这是触发被测行为的操作

---

**Q2.** 一个测试运行了程序，程序没有崩溃，开发工程师说"测试通过了"。这个说法有什么问题？

**A2.** 程序没有崩溃，不等于输出是正确的。没有 **assertion** 就没有判断，测试根本没有比较 expected output 和 actual output。"程序跑完了"≠"程序跑对了"。

---

**Q3.** 为什么 Pass/Fail Criteria 必须在运行测试之前就定好，而不能在看到结果后再决定？

**A3.** 如果在看到结果后才决定标准，你会不自觉地把"程序实际返回的值"当成"正确答案"，测试就失去了意义。标准必须来自需求，而不是来自程序的实际行为。这叫做**确认偏误（confirmation bias）**— 先看结果再定标准，永远不会发现 bug。

---

**Q4.** 下面的测试代码，哪一行是 assertion？

```python
def test_login():
    user = create_test_user(username="alice", password="secret")
    response = login(username="alice", password="secret")
    assert response.status_code == 200
```

**A4.** 第三行 `assert response.status_code == 200` 是 assertion。它比较了 actual output（`response.status_code`）和 expected output（`200`），做出 pass/fail 判决。
