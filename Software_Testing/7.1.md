## 1. 没有 Framework 会怎样？

想象你要测试一个 `add(a, b)` 函数，但没有任何工具，你可能会这样写：

```python
result = add(2, 3)
if result != 5:
    print("测试失败！")
else:
    print("测试通过！")
```

如果你有 100 个测试，你要写 100 次这种 if-else + print，还要**手动运行每一个**、**手动看输出**判断哪个过了哪个没过。没有统一格式，没有汇总报告。

**Test Framework 就是把这些重复劳动自动化的工具。**

---

## 2. Test Framework 是什么？

**Test Framework** = 一整套帮你写、组织、运行、检查测试的软件工具集合。

可以把它想象成一个"测试专用工厂"，里面有几台各司其职的机器：

```
Test Framework
├── Test Runner       — 找到测试 + 执行测试
├── Assertion Library — 判断结果对不对
└── Reporting          — 汇总输出结果
```

这三部分合起来，才构成一个完整的 framework。

---

## 3. Test Runner — 找到并执行测试

**Test Runner** = framework 里负责"找到测试、一个个执行它们"的引擎。

打个比方：如果你的测试是一堆待办事项，test runner 就是那个自动帮你逐条执行、并记录完成状态的助理——你不用自己一个个手动跑。

你通常只需要一个命令（概念上）：

```
run all tests
```

Runner 就会去扫描项目、执行每一个测试、收集结果。

---

## 4. Test Discovery — 自动发现测试

Runner 怎么知道"项目里哪些代码是测试"？靠的就是 **Test Discovery（测试发现）**机制。

Runner 按照约定俗成的规则去扫描项目，比如：
- 文件名包含 `test_` 或 `_test`
- 函数名以 `test` 开头

只要你遵守这些命名规则，runner 就能**自动**找到你的测试，不需要你手动维护一份"测试清单"。

```
你新增一个测试文件 → 命名符合规则 → runner 下次运行自动发现它 → 不用额外配置
```

这也是为什么以后写测试文件时，命名规范很重要——discovery 就是靠这些规则识别"这是测试代码"的。

---

## 5. Assertion Library — 标准化"检查结果"

还记得开头那段笨拙的 if-else 吗？**Assertion Library（断言库）**就是把这个逻辑标准化成一句话，概念上长这样：

```
assert(add(2, 3) equals 5)
```

"Assert" 的意思是"断言/声称"——你在断言"这个结果应该是这样"。

如果 assert 失败，framework 自动记录这是一个失败的测试，你不用自己写 if-else 判断和 print。（回想 [[1.3]]：assertion 就是测试里那句"执行比较"的语句。）

---

## 6. Reporting — 汇总测试结果

所有测试跑完后，你想知道：几个通过？几个失败？哪一行代码出的错？

**Reporting（报告）**就是 framework 生成的总结——通常是一份清晰的列表：
- ✅ 绿色 = 通过
- ❌ 红色 = 失败，并附上失败原因和位置

没有 reporting，你根本不知道 100 个测试里哪几个坏了、为什么坏。

---

## 7. 把四个概念串起来

一次完整的测试执行流程：

```
你写测试代码（用 assertion library 写断言）
        ↓
Test runner 做 test discovery（自动找到这些测试）
        ↓
Test runner 逐个执行测试
        ↓
每个测试的 assert 结果被收集
        ↓
生成 Reporting（告诉你哪里过了、哪里错了）
```

不管以后用 Python 的 pytest、JavaScript 的 Jest、还是 Java 的 JUnit，它们全部遵守这一套逻辑，只是语法长得不一样。**先懂逻辑，语法只是换皮。**

---

## 核心要点总结

```
Test Framework = Test Runner（找 + 跑）+ Assertion Library（判断对错）+ Reporting（汇总结果）

Test Runner    → 自动执行所有测试，不用手动一个个跑
Test Discovery → 按命名规则自动找到测试，不用手动维护清单
Assertion Library → 把"检查结果"标准化成一句话，取代手写 if-else
Reporting      → 汇总 pass/fail，告诉你哪里出了问题
```

## Self-Check

**Q1：** 如果没有 test framework，你要测试 100 个函数，需要自己写多少次"检查 + 打印结果"的逻辑？

<details>
<summary>答案</summary>

100 次，每一个测试都要手动写 if-else 判断逻辑，还要手动运行、手动看输出。这正是 test framework 要解决的问题——把这些重复劳动自动化。

</details>

---

**Q2：** 你新增了一个测试文件，需要手动告诉 runner "去跑这个新文件" 吗？为什么？

<details>
<summary>答案</summary>

不需要。只要新文件遵守 framework 的命名规则（比如以 `test_` 开头），runner 会通过 **test discovery** 机制自动扫描项目并找到它，不需要额外配置。

</details>

---

**Q3：** 为什么测试文件/函数通常要遵守特定命名规则？

<details>
<summary>答案</summary>

因为 test discovery 机制是靠这些命名规则识别"这是测试代码"的。如果不遵守规则，runner 就无法自动发现这个测试，等于它不存在。

</details>

---

**Q4：** Assertion library 解决了什么问题？

<details>
<summary>答案</summary>

它把"检查结果是否符合预期"的逻辑标准化成一句简单的语句（如 `assert result == expected`），取代了手写的 if-else + print 判断逻辑。代码更简洁、更统一，而且 assert 失败时 framework 会自动记录，不需要你手动处理失败逻辑。

</details>

---

**Q5：** Reporting 里的信息，最终来自测试执行流程的哪一步？

<details>
<summary>答案</summary>

来自 **assertion 的结果**。每个测试里的 assert 通过或失败，都会被 runner 收集起来，最终汇总成 reporting 展示给你（哪些绿、哪些红、失败原因是什么）。

</details>
