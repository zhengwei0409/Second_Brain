3.1 说 Black-box 测试依据规格来设计输入。但规格给出的输入空间可能是无限大的——你不可能测每一个数字。

## 1. 问题：为什么不能穷举所有输入？

假设你在测一个函数，接受一个整数作为年龄（0 到 120）。

光是 valid 的输入就有 121 个值。如果加上负数、超大数、小数、字符串……输入空间是无限的。

**穷举（exhaustive testing）= 测所有可能的输入。** 这在现实中几乎永远不可能做到。

那怎么办？

关键洞察：**很多输入对程序来说行为完全一样。**

年龄 25、年龄 30、年龄 50——对一个"检查是否成年"的函数来说，它们都会走同一条代码路径，返回同样的结果。

既然行为一样，测一个就够了，不需要三个。

---

## 2. Equivalence Class 是什么

**Equivalence Class（等价类）** = 一组输入值，它们对被测程序产生**相同的行为**（走相同的代码路径，得到相同类型的输出）。

"Equivalence" = 等价，意思是这些值对程序来说"效果一样"。

**Equivalence Partitioning（等价类划分）** = 把整个输入空间切分成若干个等价类的过程。

```
整个输入空间（无限大）
│
├── 等价类 A：[0, 17]     → 都触发"未成年"逻辑
├── 等价类 B：[18, 120]   → 都触发"成年"逻辑
└── 等价类 C：< 0 或 > 120 → 都触发"无效输入"逻辑
```

每个等价类内部的值，对程序的行为是相同的——所以每类只需要测一个代表值。

---

## 3. Valid vs Invalid Partition

等价类分两种：

**Valid Partition（有效等价类）** = 程序**应该接受**的输入范围，会产生正常的预期输出。

**Invalid Partition（无效等价类）** = 程序**不应该接受**的输入范围，应该触发错误处理（报错、拒绝、返回默认值等）。

两种都必须测。

**为什么 invalid partition 也要测？**

因为程序对非法输入的处理，也是需求的一部分。规格可能说"年龄小于 0 时抛出 ValueError"——你需要验证这个行为是否实现了。

---

## 4. One representative per class

**核心规则：每个等价类只需要选一个代表值来测。**

等价类内的任何一个值都能代表整个类——它们触发的是同一段代码逻辑。

```
等价类 B：[18, 120]
代表值可以选：18、50、99、120 都行
选 50 → 就代表了整个 [18, 120] 区间
```

**Input space reduction（输入空间缩减）** = 这就是 EP 的核心价值——把无限的输入空间，缩减成少数几个有代表性的测试用例。

---

## 5. 完整例子：年龄验证

**规格说明：**

函数 `check_age(age)`:
- `age` 是整数
- 0 ≤ age ≤ 17：返回 `"minor"`
- 18 ≤ age ≤ 120：返回 `"adult"`
- age < 0 或 age > 120：抛出 `ValueError`

### 步骤一：识别等价类

| 等价类 | 范围 | 类型 |
|--------|------|------|
| EC1 | 0 ≤ age ≤ 17 | Valid（未成年） |
| EC2 | 18 ≤ age ≤ 120 | Valid（成年） |
| EC3 | age < 0 | Invalid |
| EC4 | age > 120 | Invalid |

### 步骤二：每类选一个代表值

| 等价类 | 代表值 | 期望结果 |
|--------|--------|---------|
| EC1 | 10 | `"minor"` |
| EC2 | 50 | `"adult"` |
| EC3 | -5 | `ValueError` |
| EC4 | 200 | `ValueError` |

### 步骤三：写测试用例

```python
def test_minor():
    assert check_age(10) == "minor"

def test_adult():
    assert check_age(50) == "adult"

def test_negative_age():
    with pytest.raises(ValueError):
        check_age(-5)

def test_age_too_large():
    with pytest.raises(ValueError):
        check_age(200)
```

**结果：** 从无限输入空间，缩减到 4 个有代表性的测试用例。

---

## 6. Partition Coverage

**Partition Coverage（等价类覆盖率）** = 你已经测到的等价类数量 ÷ 总等价类数量。

```
已测等价类: EC1, EC2, EC3, EC4 = 4 个
总等价类: 4 个
Partition Coverage = 4/4 = 100%
```

目标是达到 **100% partition coverage**——每个等价类至少有一个代表值被测到。

**注意：** Partition coverage 和 Code coverage（4.x 章）是不同的概念。Partition coverage 关注的是输入空间有没有被覆盖，不是代码行有没有被执行。

---

## 7. EP 的局限

EP 告诉你每个等价类选一个代表值，但没告诉你选哪一个。

**如果你总是选等价类的中间值，你会遗漏一类重要的 bug：边界上的 bug。**

例如：函数对 age = 17 返回 `"minor"`，对 age = 18 返回 `"adult"`——但程序员写代码时可能把 `>=18` 误写成 `>18`，导致 age = 18 返回了 `"minor"`。

用 EP 选代表值 10 和 50，永远发现不了这个 bug。

**这就是为什么 3.3 Boundary Value Analysis 紧接在 EP 后面——EP 和 BVA 是配套使用的。**

## Self-Check

**Q1.** Equivalence Partitioning 解决的核心问题是什么？

> **A:** 输入空间是无限大的，不可能穷举所有输入。EP 把输入空间切分成若干等价类，每类里的值对程序行为相同，每类只需测一个代表值，从而把无限输入缩减到有限的几个测试用例。

---

**Q2.** 什么是 Equivalence Class？举一个例子。

> **A:** 等价类是一组输入值，它们让程序产生相同的行为（走相同代码路径，得到相同类型的输出）。例如：函数检查密码长度是否 8~20 位，输入长度 8~20 的所有密码都会通过验证——它们构成一个等价类，只需测其中一个（如长度 12）来代表整个类。

---

**Q3.** 为什么 invalid partition 也要测，不是只测 valid 的输入就够了？

> **A:** 程序对非法输入的处理行为也是需求的一部分。规格可能规定"无效输入要抛出错误"，你需要验证这个行为是否正确实现了。只测 valid 输入，会遗漏对错误处理逻辑的验证。

---

**Q4.** 函数 `set_volume(level)` 接受整数，0~100 是有效音量，其余无效。请列出所有等价类。

> **A:**
> - EC1（Valid）：0 ≤ level ≤ 100（有效音量范围）
> - EC2（Invalid）：level < 0（负数）
> - EC3（Invalid）：level > 100（超出范围）
>
> 代表值可以选：50（EC1），-1（EC2），150（EC3）

---

**Q5.** EP 有什么局限？需要配合什么技术来弥补？

> **A:** EP 只说"每类选一个代表值"，但不指定选哪个值。如果总是选类的中间值，会遗漏边界附近的 bug（例如 `>=18` 误写成 `>18`，只有测 age=18 才能发现）。需要配合 **Boundary Value Analysis（3.3）** 来弥补——BVA 专门测等价类的边界值。
