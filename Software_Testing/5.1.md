## 1. 什么是 Dependency（依赖）？

**Dependency（依赖）** = 你的代码为了完成工作，需要调用的外部东西。

"外部"的意思是：不在你当前被测函数里的东西。

### 常见的 Dependency 类型

| Dependency 类型 | 例子 |
|----------------|------|
| **数据库** | 查询用户表、写入订单记录 |
| **外部 API** | 调用天气 API、支付 API |
| **文件系统** | 读写本地文件 |
| **系统时间** | `datetime.now()`、`Date.now()` |
| **随机数** | `random.random()` |
| **邮件 / SMS 服务** | 发送验证码 |
| **其他模块** | 调用另一个团队写的模块 |

### 代码例子

```python
class OrderService:
    def place_order(self, user_id, product_id):
        user = database.get_user(user_id)      # ← Dependency：数据库
        price = payment_api.get_price(product_id)  # ← Dependency：外部 API
        email_service.send_confirmation(user.email)  # ← Dependency：邮件服务
        return {"status": "success", "price": price}
```

`place_order` 这个函数有 3 个 dependency。如果你要测试它，这 3 个东西都会被触发。

## 2. 真实 Dependency 的三大问题

### 问题一：Slow（慢）

真实的数据库查询、HTTP API 请求需要时间——几百毫秒到几秒不等。

Unit test 的核心价值是**快速反馈**（毫秒级）。如果每个测试都要等真实的数据库或 API 响应，一个项目有几百个 unit test，整个测试套件可能要跑几分钟甚至更久。

```
1 个测试 × 真实 DB 查询 ≈ 200ms
500 个测试 × 200ms = 100 秒 ← 开发者开始跳过测试
```

### 问题二：External / Unreliable（外部 / 不可靠）

外部依赖不在你的控制范围内：

- 第三方 API 可能宕机
- 网络可能断线
- 测试数据库可能没有你需要的数据

如果测试依赖外部服务，那测试失败的原因可能不是你的代码有 bug，而是**外部服务出了问题**。这违背了 unit test 的目的——测试应该只告诉你被测代码是否正确。

### 问题三：Non-determinism（不确定性）

**Non-determinism（非确定性）** = 同样的输入，每次运行可能产生不同的输出。

```python
import datetime

def get_greeting():
    hour = datetime.datetime.now().hour  # ← 每次调用结果不同
    if hour < 12:
        return "Good morning"
    else:
        return "Good afternoon"
```

`get_greeting()` 在上午运行返回 "Good morning"，下午运行返回 "Good afternoon"。

**这对测试是致命的**：

```python
def test_get_greeting():
    assert get_greeting() == "Good morning"  # 上午 pass，下午 fail
```

这个测试在不同时间、不同环境下会产生不同结果——它是一个 **Flaky Test（8.1 会讲）**，完全不可靠。

Non-determinism 的来源：
- 系统时间（`datetime.now()`）
- 随机数（`random.random()`）
- 外部 API 返回的数据
- 数据库里的数据（随时可能被其他测试修改）

## 3. 什么是 Test Double？

**Test Double（测试替身）** = 在测试中，用来**替换**真实 dependency 的假对象。

名字来源于电影里的"替身演员（Stunt Double）"——真正的演员不出场，换一个替身来完成危险动作。测试里也一样：真正的数据库、API 不出场，换一个假的替身来配合测试。

```
真实系统：
OrderService → 真实数据库 → 真实 API → 真实邮件服务

测试时：
OrderService → 假数据库 → 假 API → 假邮件服务
                (Test Double)
```

**Test Double 是一个总称（umbrella term）**，它包含几种不同类型的假对象（5.2 会详细讲）。现在只需要知道：Test Double = 替换真实 dependency 的东西。

---

## 4. Test Double 怎么解决三个问题？

| 问题 | Test Double 的解决方式 |
|------|----------------------|
| **Slow** | 假对象不做任何真实操作，直接返回你指定的值——毫秒级 |
| **External / Unreliable** | 假对象完全在你的控制下，不依赖任何外部服务，永远稳定 |
| **Non-determinism** | 假对象返回固定值（你说返回什么就返回什么），结果完全可预测 |

### 例子：用 Test Double 替换数据库

```python
# 真实代码
class OrderService:
    def __init__(self, db):
        self.db = db

    def get_user_name(self, user_id):
        user = self.db.get_user(user_id)  # ← 调用数据库
        return user.name
```

```python
# 测试时，用假的 db 替换真实数据库
class FakeDatabase:
    def get_user(self, user_id):
        # 不连真实数据库，直接返回固定数据
        return FakeUser(name="Alice")

def test_get_user_name():
    fake_db = FakeDatabase()          # Test Double
    service = OrderService(db=fake_db)  # 把假 db 注入进去
    result = service.get_user_name(1)
    assert result == "Alice"
```

这个测试：
- ✅ 快（FakeDatabase 直接返回，不访问任何真实数据库）
- ✅ 稳定（不依赖外部服务）
- ✅ 确定性（每次运行结果一样）

---

## 5. Isolation Need（为什么隔离是必要的）

Test Double 的存在是为了实现 **Isolation（隔离）**——让 unit test 真正只测试被测代码的逻辑，而不是同时测试数据库、API、邮件服务是否工作。

```
没有隔离的测试失败时，你不知道是：
→ 被测函数的逻辑有 bug？
→ 数据库查询出了问题？
→ 外部 API 宕机了？
→ 网络断了？

有隔离（用 Test Double）的测试失败时，你确定：
→ 一定是被测函数的逻辑有问题
```

**隔离的价值：测试失败时，你能立刻知道问题在哪。**

---

## 核心要点总结

```
Dependency = 被测代码需要调用的外部东西（DB、API、时间、随机数等）

真实 Dependency 的问题：
1. Slow → 拖慢测试，开发者跳过测试
2. External / Unreliable → 测试失败不代表你的代码有 bug
3. Non-determinism → 同样输入，不同时间产生不同结果

Test Double = 在测试中替换真实 dependency 的假对象

解决方式：
→ 假对象直接返回固定值，快速、稳定、确定
→ 实现 Isolation，失败时锁定问题来源
```

**Q3：** 什么是 Non-determinism？为什么它对测试是致命的？

<details>
<summary>答案</summary>

**Non-determinism** = 同样的输入，每次运行可能产生不同的输出。常见来源：系统时间、随机数、外部 API 返回的数据。

它对测试致命，因为一个 non-deterministic 的测试在某些时候 pass，某些时候 fail——与被测代码是否有 bug 无关。你无法依赖这样的测试来判断代码是否正确，它变成了一个 Flaky Test，失去了测试的意义。

</details>

**Q4：** Test Double 是什么？它的名字来自哪里？

<details>
<summary>答案</summary>

**Test Double** = 在测试中用来替换真实 dependency 的假对象，是一个总称（umbrella term），涵盖多种类型（如 Stub、Mock 等）。

名字来源于电影里的**替身演员（Stunt Double）**——真正的演员不出场，由替身完成危险动作。测试里，真实的数据库或 API 不出场，由 Test Double 替代。

</details>

---

**Q5：** 为什么用 Test Double 实现 Isolation 后，测试失败的诊断变得更容易？

<details>
<summary>答案</summary>

没有隔离时，测试失败的原因可能是：被测代码有 bug、数据库出问题、外部 API 宕机、网络断线——你不知道根本原因在哪。

用 Test Double 隔离所有外部依赖后，假对象是你控制的、稳定的。如果测试失败，能确定唯一的原因就是**被测代码的逻辑有问题**，排查范围大幅缩小。

</details>
