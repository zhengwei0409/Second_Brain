# 1.1 — Why Software Fails

## 在大局中的位置

软件测试这门学科，从最基础的问题开始：**为什么软件会出错？**

在你学会怎么写测试之前，你必须先搞清楚你在"测"什么、"防"什么。这就是 1.1 的任务。

```

    └── 软件质量保证（QA）
        └── 软件测试
            └── [1.1] 为什么软件会失败？← 你在这里
```

---

## 本节结构

1. Error（错误）是什么？
2. Fault / Defect（缺陷）是什么？
3. Failure（故障）是什么？
4. 三者的关系：Error → Fault → Failure chain
5. 为什么晚发现缺陷代价更高？

---

## 1. Error — 人犯的错误

**Error** 是 **人类（程序员）** 犯的认知错误。

比如你在写代码时：
- 误解了需求
- 搞错了数学逻辑
- 打错了变量名

Error 是发生在 **人的脑子里** 或 **人的手指上** 的，还没到代码层面。

> 你以为用户年龄必须 `> 18`，但正确要求是 `>= 18`。这个理解上的偏差就是 error。

---

## 2. Fault / Defect — 代码里的缺陷

**Fault**（也叫 **Defect** 或 **Bug**）是 error 的 **结果**，它存在于 **源代码** 里。

程序员犯了 error，写出了有问题的代码，这个有问题的代码片段就叫 fault。

```python
# 正确应该是 >= 18，但程序员写成了 > 18
if age > 18:
    allow_access()
```

Fault 此时静静躺在代码里。**它不一定会被触发。**

> Fault 就像是一颗地雷埋在代码里 — 只有特定的输入踩到它，才会爆。

---

## 3. Failure — 系统在运行时表现出的错误行为

**Failure** 是指程序 **在运行时** 出现了与预期不符的行为。

换句话说：用户或系统看到了错误的结果。

```python
user_age = 18
if user_age > 18:   # fault 在这里
    allow_access()
# 结果：18岁的用户被拒绝访问 ← 这是 failure
```

Failure 是 fault 被触发之后，**对外可见的症状**。

---

## 4. Error → Fault → Failure Chain（链条关系）

这是本节最核心的模型：

```
人类犯认知错误 (Error)
        ↓
导致代码里有缺陷 (Fault / Defect)
        ↓
某个输入触发了它，程序行为出错 (Failure)
```

**重要细节：**

| 步骤 | 发生在哪里 | 谁能看到 |
|------|-----------|---------|
| Error | 人的脑子 / 手 | 只有程序员（也可能自己都不知道） |
| Fault | 源代码 | 代码审查、静态分析工具 |
| Failure | 运行时输出 | 用户、测试、监控系统 |

**关键点：fault 存在 ≠ failure 一定发生。**

一个 fault 可能在代码里藏很久都不被触发 — 只有当某个特定的输入路径经过它，failure 才会出现。

---

## 5. Cost of Late Defects — 晚发现缺陷，代价更大

这是测试存在的经济学理由。

**越晚发现 fault，修复它的成本越高：**

```
[写代码时发现] → 成本最低，直接改
[测试阶段发现] → 成本中等，需要找回去改代码再重新测
[上线后用户发现] → 成本最高，影响用户、损害信誉、可能要紧急修复
```

这个规律叫做 **"1-10-100 Rule"**（各阶段成本的粗略倍数关系）：

| 发现时机 | 相对成本 |
|---------|---------|
| 开发中 | 1x |
| 测试阶段 | 10x |
| 生产环境（上线后） | 100x |

这就是为什么公司愿意花钱雇测试工程师：**提前测试，比事后救火便宜得多。**

## Self-Check

**Q1.** 一个程序员在写代码时误解了需求，把 `>=` 写成了 `>`。请问这个"误解需求"属于 Error、Fault 还是 Failure？这行有问题的代码本身又属于哪个？

**A1.**
- 误解需求 = **Error**（发生在人的认知层面）
- 有问题的那行代码（`> 18`）= **Fault**（error 的产物，存在于源代码中）

---

**Q2.** 一个登录功能有个 bug：当密码恰好是 32 个字符时，系统会错误地拒绝登录。这个 bug 在系统上线一年后，才有用户遇到这个情况并报告了问题。请问在这一年里，代码里存在的是什么？用户报告问题的那一刻，又出现了什么？

**A2.**
- 这一年里：**Fault** 一直存在于代码中（但从未被触发）
- 用户遇到并报告时：**Failure** 发生了（程序运行时出现了错误行为）

---

**Q3.** 以下哪个说法是正确的？

A. Fault 存在，Failure 就一定会发生。
B. Failure 发生，说明一定有人犯了 Error。
C. 一个 Error 一定会立刻导致 Failure。
D. Failure 只在测试环境中才会出现。

**A3.** **B** 是正确的。

- A 错：Fault 不一定被触发，Failure 不一定发生。
- B 对：Failure 的源头追溯到底，必然是某个人犯了认知上的 Error。
- C 错：Error → Fault → Failure 是链条，不是即时的。
- D 错：Failure 可以在任何环境（包括生产环境）中发生，而且生产环境发生代价最高。
