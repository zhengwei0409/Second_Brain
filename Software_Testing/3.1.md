## 1. 为什么需要"设计"测试？

假设你在测一个函数 `calculate_discount(price, member_level)`。

它接受价格和会员等级，返回折后价格。你要怎么测它？

**随便传几个数进去？**

问题：随便传可能永远测不到某些重要的路径。比如：
- 价格是负数会怎样？
- 会员等级无效会怎样？
- 价格是 0 会怎样？

**Test design（测试设计）** = 用系统的方法决定"测哪些输入"，而不是靠感觉随机猜。

Black-Box 和 White-Box 是两种不同的**出发点**来设计这些输入。

---

## 2. Black-Box Testing

**Black-Box Testing（黑盒测试）** = 测试者**不看源代码**，只根据**需求规格（specification）**来设计测试用例。

"Black-box"：把被测系统想象成一个黑色的盒子，你看不到里面，只能：
- 从外面**输入**数据
- 观察**输出**是否符合预期

```
┌─────────────────────────┐
│                         │
│   ??? 内部实现 ???       │  ← 测试者看不到
│                         │
└─────────────────────────┘
        ↑           ↓
      Input       Output
  （测试者控制）  （测试者观察）
```

### 设计依据是什么？

Black-box 的测试用例来自 **specification（规格说明）**——也就是需求文档、API 文档、用户故事等描述"这个功能应该做什么"的文件。

**例子：**

规格说明：`calculate_discount(price, member_level)` 的行为：
- 普通会员：9折
- 金牌会员：8折
- 价格必须 > 0，否则抛出 ValueError

Black-box 测试者根据这段规格，设计测试用例：

| 测试用例 | price | member_level | 期望输出 |
|---------|-------|--------------|---------|
| 普通会员折扣 | 100 | "normal" | 90 |
| 金牌会员折扣 | 100 | "gold" | 80 |
| 价格为负数 | -10 | "normal" | ValueError |
| 价格为零 | 0 | "normal" | ValueError |

注意：测试者完全不需要看函数内部怎么实现折扣逻辑。

### 别名

Black-box testing 也叫 **Specification-based testing（基于规格的测试）**，因为它的设计完全依赖规格文档。

---

## 3. White-Box Testing

**White-Box Testing（白盒测试）** = 测试者**看源代码**，根据**内部结构**来设计测试用例，目的是覆盖代码里的每一条路径。

"White-box"（或 Glass-box）：盒子是透明的，你能看到里面所有的逻辑。

```
┌─────────────────────────────────────────┐
│  if member_level == "gold":             │
│      return price * 0.8                 │  ← 测试者能看到
│  elif member_level == "normal":         │
│      return price * 0.9                 │
│  else:                                  │
│      raise ValueError("invalid level") │
└─────────────────────────────────────────┘
```

### 设计依据是什么？

White-box 的测试用例来自**代码结构**——每一个 `if`、`elif`、`else`、循环、异常处理都是需要被测到的路径。

测试者的目标：让每一行代码、每一个分支都至少被执行一次（这就是后面 4.x 章会讲的 **Code Coverage**）。

**例子：**

看到上面的代码，白盒测试者会设计：
- 进入 `"gold"` 分支的用例
- 进入 `"normal"` 分支的用例
- 进入 `else`（ValueError）分支的用例

这跟黑盒测出来的结果可能相似，但**出发点完全不同**——不是看规格，而是看代码有没有被跑到。

### 别名

White-box testing 也叫 **Structure-based testing（基于结构的测试）**，因为它的设计依赖代码的内部结构。

---

## 4. 两者的核心区别

| | Black-Box | White-Box |
|-|-----------|-----------|
| **设计依据** | 需求规格（specification） | 源代码结构 |
| **测试者需要看代码吗？** | 不需要 | 需要 |
| **目标** | 验证行为符合需求 | 确保所有代码路径被执行 |
| **能发现什么 bug？** | 需求没被满足的 bug | 代码路径里藏着的 bug |
| **发现不了什么？** | 代码里多余的死路径 | 缺少功能（没写进代码的需求） |
| **别名** | Specification-based | Structure-based |
| **谁来做？** | QA、测试工程师、甚至产品 | 开发者（知道代码结构） |

### 关键洞察

**Black-box 能发现规格被违反，White-box 能发现代码路径没被覆盖。两者各有盲区。**

- 只用 Black-box：可能代码里有分支永远不会触发，你不知道
- 只用 White-box：可能某个需求根本没有写进代码，你测的全是已有代码，发现不了缺失

---

## 5. Gray-Box Testing

**Gray-Box Testing（灰盒测试）** = 测试者对内部实现有**部分了解**，结合两者来设计用例。

不是完全黑盒（完全不看代码），也不是完全白盒（完整读代码），而是中间状态。

**实际例子：**
- 测试者知道后端用 PostgreSQL，所以特意测 SQL 注入场景
- 测试者知道有缓存层，所以测缓存失效后的行为
- 测试者知道有限制的某个业务规则，针对性设计边界测试

**实际工作中，大部分测试工程师都处于灰盒状态**——他们不会逐行读源代码，但会通过架构文档、PR review、团队沟通了解系统的关键实现细节。

---

## 6. 什么时候用哪个？

| 场景 | 用哪个 | 原因 |
|------|--------|------|
| 验收测试（产品上线前检查） | Black-box | 从用户角度验证功能 |
| 代码审查后补测试 | White-box | 确保新分支被覆盖 |
| 写 unit test | White-box 为主 | 开发者看着代码写 |
| 写 E2E test | Black-box 为主 | 模拟用户，不关心内部 |
| 安全测试 | Gray-box | 需要了解部分架构才能设计攻击场景 |

**一句话总结：**
> Black-box 验证"你做了该做的事"，White-box 验证"你做的事都能跑到"。两者结合才完整。

## Self-Check

**Q1.** Black-box testing 的设计依据是什么？White-box 的设计依据是什么？

> **A:** Black-box 依据**需求规格（specification）**——不看代码，只看功能应该做什么。White-box 依据**源代码结构**——看 if/else、循环等分支，确保每条路径都被测到。

---

**Q2.** 一个 QA 工程师拿到 API 文档，在不看任何源代码的情况下写了一套测试用例。这是 black-box 还是 white-box？

> **A:** Black-box。设计依据是 API 文档（specification），不涉及内部实现。

---

**Q3.** Black-box testing 有什么盲区？

> **A:** 它发现不了"代码里多余的路径"问题，也无法保证代码的每个分支都被执行过。如果代码里有一段死代码或者某个分支从未被触发，black-box test 不会发现。

---

**Q4.** White-box testing 有什么盲区？

> **A:** 它发现不了"缺失的功能"——如果某个需求根本没有写进代码，white-box 测的都是已有的代码路径，永远不会发现那个需求缺失了。

---

**Q5.** Gray-box testing 是什么？举一个实际例子。

> **A:** Gray-box = 测试者对内部实现有部分了解，结合黑盒和白盒。例子：测试者知道系统用了 Redis 缓存，于是专门设计一个测试：先请求数据（存入缓存），然后直接修改数据库绕过缓存，再次请求，检查返回的是旧的缓存数据还是新数据。这个测试用到了对内部架构的了解，但不需要逐行读源代码。
