## 1. 五种 Test Double 类型总览

| 类型 | 一句话描述 | 核心用途 |
|------|-----------|---------|
| **Dummy** | 占位符，从不被真正使用 | 满足参数要求，测试不关心它 |
| **Stub** | 返回固定数据 | 控制 dependency 的输出，影响被测代码的行为 |
| **Fake** | 有真实简化实现的替身 | 替换太重的 dependency（如内存数据库替换真实 DB） |
| **Spy** | 记录调用信息的 Stub | Stub 的功能 + 记录被调用了几次、传了什么参数 |
| **Mock** | 预设期望并验证调用 | 验证被测代码是否正确地调用了 dependency |

记忆顺序：**Dummy → Stub → Fake → Spy → Mock**，复杂度依次递增。

---

## 2. Dummy — 占位符

**Dummy** = 一个什么都不做的假对象，只是为了满足函数签名的参数要求。

测试根本不关心这个参数，但函数要求你传它，所以你随便传一个。

### 例子

```python
def send_report(data, logger):
    result = process(data)
    logger.log(f"Done: {result}")  # 测试不关心日志
    return result

def test_send_report():
    dummy_logger = None  # 或者一个空对象，反正不会被用到
    result = send_report([1, 2, 3], dummy_logger)
    assert result == 6
```

`dummy_logger` 就是 Dummy——它存在只是因为函数签名要求有 `logger`，但测试对日志行为完全不在乎。

**特征：** 传进去了，但永远不会被真正调用。

---

## 3. Stub — 提供固定数据

**Stub** = 一个返回预设（hardcoded）数据的假对象，用来控制 dependency 给被测代码的输入。

Stub 解决的问题是：被测代码需要从 dependency 拿数据（比如数据库查询），但测试不想连真实数据库——Stub 直接返回你指定的假数据。

### Stub 的方向

```
Test → 被测代码 → Stub（dependency）
                  Stub 返回假数据给被测代码
```

Stub 的数据流是**向被测代码输入**。

### 例子

```python
class UserService:
    def __init__(self, db):
        self.db = db

    def get_display_name(self, user_id):
        user = self.db.get_user(user_id)  # 需要从 db 拿数据
        return f"{user['name']} ({user['email']})"
```

```python
class StubDatabase:
    def get_user(self, user_id):
        return {"name": "Alice", "email": "alice@example.com"}  # 固定返回

def test_get_display_name():
    stub_db = StubDatabase()
    service = UserService(db=stub_db)
    result = service.get_display_name(1)
    assert result == "Alice (alice@example.com)"
```

`StubDatabase` 不管传入什么 `user_id`，都返回同一份假数据。测试只关心 `UserService` 拿到数据后，格式化逻辑是否正确。

**特征：** 你告诉它返回什么，它就返回什么。测试关注的是**被测代码对数据的处理**。

---

## 4. Fake — 有简化实现的替身

**Fake** = 有真实可用逻辑的替身，但实现比真实 dependency 简单很多。

Stub 只返回固定值，Fake 实际上"工作"——只是用更轻量的方式工作。

### 最经典的例子：In-Memory Database

```python
class FakeDatabase:
    def __init__(self):
        self.store = {}  # 用内存字典代替真实数据库

    def save_user(self, user_id, user):
        self.store[user_id] = user

    def get_user(self, user_id):
        return self.store.get(user_id)
```

这个 FakeDatabase：
- 不连任何真实数据库
- 但确实能 save 和 get（有真实逻辑）
- 只是数据存在内存里，测试结束就消失

**Stub vs Fake 的区别：**

| | Stub | Fake |
|-|------|------|
| 有真实逻辑？ | ❌（只返回固定值） | ✅（有简化实现） |
| 适合场景 | 简单的数据返回 | 需要 dependency 真的"工作"（如多步操作） |

**特征：** Fake 能做真实 dependency 能做的事，只是更轻量。常见于集成测试或需要多步操作的场景。

---

## 5. Spy — 记录调用的 Stub

**Spy** = 一个 Stub，额外记录自己被调用了几次、传了什么参数。

Spy 本身也能返回假数据（Stub 的功能），但它还偷偷记录所有被调用的信息，供测试事后检查。

### 例子

```python
class SpyEmailService:
    def __init__(self):
        self.sent_emails = []  # 记录所有发出的邮件

    def send(self, to, subject, body):
        self.sent_emails.append({"to": to, "subject": subject})
        # 不真的发邮件

def test_order_sends_confirmation():
    spy_email = SpyEmailService()
    order_service = OrderService(email_service=spy_email)

    order_service.place_order(user_email="bob@example.com", product="Book")

    # 事后检查 Spy 的记录
    assert len(spy_email.sent_emails) == 1
    assert spy_email.sent_emails[0]["to"] == "bob@example.com"
```

**特征：** 测试结束后，你可以问 Spy："你被调用了几次？传了什么参数？"

## 6. Mock — 验证交互行为

**Mock** = 预先设定期望（expectations），测试结束时自动验证这些期望是否被满足。

这是与 Stub 最容易混淆、也最重要的区别。

### Mock 的方向

```
Test → 被测代码 → Mock（dependency）
Mock 在测试结束时反问：
"你有没有按我预期的方式调用我？"
```

Mock 的关注点是**被测代码的行为**（调用了什么、怎么调用的），而不是返回值。

### 例子

```python
from unittest.mock import MagicMock

def test_order_notifies_user():
    mock_email = MagicMock()  # Python 内建 Mock
    order_service = OrderService(email_service=mock_email)

    order_service.place_order(user_email="bob@example.com", product="Book")

    # Mock 在这里验证：send 有没有被调用？参数对不对？
    mock_email.send.assert_called_once_with(
        to="bob@example.com",
        subject="Order Confirmed"
    )
```

如果 `place_order` 里根本没调用 `email_service.send()`，或者调用时参数错了，这个测试就会 **fail**。

**特征：** 测试前设定期望，测试结束自动验证调用是否符合预期。

## 7. Stub vs Mock — 核心区别

这是本节最重要的概念。

| | Stub | Mock |
|-|------|------|
| **关注点** | 给被测代码提供数据（输入方向） | 验证被测代码是否正确调用了 dependency（输出方向） |
| **验证什么** | 验证被测代码对数据的处理结果 | 验证被测代码的调用行为（调没调、怎么调） |
| **测试 assertion 在哪** | 在测试的 assert 里，检查返回值 | 在 Mock 的 verify 里，检查调用记录 |
| **失败原因** | 被测代码处理数据的逻辑错了 | 被测代码没有按预期调用 dependency |

### 一句话记忆

> **Stub：** 我给你数据，你去处理，我只检查你处理的结果。  
> **Mock：** 我不在乎你的结果，我只在乎你有没有按规定来找我。

### 场景对比

**用 Stub 的场景：** 测试"当用户存在时，显示名称格式是否正确"
→ Stub 提供假的用户数据，测试检查格式化结果

**用 Mock 的场景：** 测试"下单后，系统有没有发确认邮件给用户"
→ Mock 记录邮件服务有没有被调用，用什么参数调用

---

## 核心要点总结

```
五种 Test Double：
Dummy  → 占位符，不被使用
Stub   → 返回固定数据，控制依赖的输出
Fake   → 有简化真实逻辑的替身（如内存 DB）
Spy    → 记录调用信息的 Stub
Mock   → 预设期望并验证调用行为

最重要的区别：
Stub  → 提供数据给被测代码（State verification）
Mock  → 验证被测代码的调用行为（Behavior verification）
```

## Self-Check

**Q1：** 你有一个函数需要传入一个 `config` 对象，但测试根本不关心 config 的内容。应该用哪种 Test Double？

<details>
<summary>答案</summary>

**Dummy。** 只需要一个占位符满足函数签名要求，测试不关心它的内容或行为。可以传 `None` 或一个空对象。

</details>

---

**Q2：** Stub 和 Fake 都能返回假数据，它们的核心区别是什么？

<details>
<summary>答案</summary>

- **Stub** 只返回固定的预设值，没有真实逻辑。不管输入是什么，永远返回同一个值。
- **Fake** 有真实可用的简化逻辑。例如 In-Memory Database——它真的能 save 和 get 数据，只是存在内存里而不是真实数据库。

区别：Fake 会根据输入做实际处理，Stub 不会。

</details>

---

**Q3：** 什么情况下用 Stub，什么情况下用 Mock？

<details>
<summary>答案</summary>

- **用 Stub：** 被测代码需要从 dependency 获取数据，你想控制那份数据是什么，然后验证被测代码对数据的处理结果（state verification）。
- **用 Mock：** 你想验证被测代码是否正确地调用了某个 dependency（有没有调用、调用几次、传了什么参数），而不是关心返回值（behavior verification）。

</details>

---

**Q4：** Spy 和 Mock 都能检查调用信息，它们的区别是什么？

<details>
<summary>答案</summary>

- **Spy** 是被动记录的——它把调用记录存下来，测试结束后由你手动写 assertion 去检查（`assert len(spy.calls) == 1`）。
- **Mock** 是主动验证的——你在测试开始前预设期望（expect），Mock 在测试结束时自动检查期望是否满足，不满足则直接 fail。

简单说：Spy 记录，你来验；Mock 自己预设，自己验。

</details>

---

**Q5：** 以下测试用的是 Stub 还是 Mock？为什么？

```python
def test_login():
    fake_db = StubDatabase(returns={"id": 1, "password_hash": "abc"})
    auth_service = AuthService(db=fake_db)
    result = auth_service.login("user@example.com", "correct_password")
    assert result == True
```

<details>
<summary>答案</summary>

**Stub。** 原因：
- `StubDatabase` 提供固定数据给 `AuthService`
- 测试关注的是 `AuthService.login()` 的返回值（`True`）
- 测试验证的是被测代码对数据的处理结果（State verification）
- 没有验证 `fake_db` 被调用了几次或传了什么参数

如果是 Mock，会在最后加 `fake_db.get_user.assert_called_once_with("user@example.com")`。

</details>
