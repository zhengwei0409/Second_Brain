## 1. Statement Coverage（语句覆盖）

**Statement（语句）** = 代码里的一条可执行指令，通常对应一行代码。

**Statement Coverage** = 测试套件执行了多少比例的语句。

```
Statement Coverage % = (被执行的语句数 / 总语句数) × 100
```

这就是 4.1 里讲的最基础的 coverage 概念。

### 例子

```python
def process(x):
    result = x * 2        # 语句 1
    if x > 10:
        result = result + 1  # 语句 2
    return result         # 语句 3
```

测试：`process(5)`（x = 5，不大于 10）

| 语句 | 执行了？ |
|------|---------|
| 语句 1：`result = x * 2` | ✅ |
| 语句 2：`result = result + 1` | ❌（x=5，跳过 if 块） |
| 语句 3：`return result` | ✅ |

Statement Coverage = 2/3 = **67%**

如果再加一个测试 `process(15)`（x = 15，大于 10），语句 2 也会被执行：

Statement Coverage = 3/3 = **100%**

## 2. Branch Coverage（分支覆盖）

**Branch（分支）** = 代码中每一个判断点（Decision Point）的每一条可能的走向。

**Branch Coverage** = 测试套件覆盖了多少比例的分支路径。

```
Branch Coverage % = (被执行的分支数 / 总分支数) × 100
```

### 什么是 Decision Point？

**Decision Point（决策点）** = 代码中让程序"做选择"的地方，会产生至少两条分支。

常见的 Decision Point：

```python
if condition:      # → True 分支 + False 分支
    ...

while condition:   # → 继续循环 + 退出循环

for item in list:  # → 有元素继续 + 结束循环

x = a if cond else b  # → True 分支 + False 分支（三元表达式）
```

每个 Decision Point 产生 **2 条分支（True / False）**。

### 例子（用同一段代码）

```python
def process(x):
    result = x * 2        # 语句 1
    if x > 10:            # ← Decision Point，产生 2 条分支
        result = result + 1  # 语句 2（True 分支）
    return result         # 语句 3
                          # （False 分支：直接跳到 return）
```

这个函数的所有分支：

| 分支 | 描述 |
|------|------|
| 分支 A | `x > 10` 为 **True** → 进入 if 块 |
| 分支 B | `x > 10` 为 **False** → 跳过 if 块 |

测试：只有 `process(5)`（x = 5）

| 分支 | 执行了？ |
|------|---------|
| 分支 A（True） | ❌ |
| 分支 B（False） | ✅ |

Branch Coverage = 1/2 = **50%**

要达到 100% Branch Coverage，需要同时测试：
- `process(5)` → 覆盖 False 分支
- `process(15)` → 覆盖 True 分支

## 3. 为什么 Branch Coverage 比 Statement Coverage 更强？

这是本节最关键的一点。

看这个例子：

```python
def process(x):
    result = x * 2
    if x > 10:
        result = result + 1
    return result
```

测试：`process(5)` 和 `process(15)` 两个测试同时运行。

**Statement Coverage：**
- 所有 3 条语句都执行过了 → **100%**

**Branch Coverage：**
- True 分支：`process(15)` 覆盖了 ✅
- False 分支：`process(5)` 覆盖了 ✅
- → **100%**

现在换一个更极端的例子：

```python
def check(a, b):
    if a > 0:             # Decision Point 1
        print("a positive")
    if b > 0:             # Decision Point 2
        print("b positive")
```

总语句数：4（两个 if + 两个 print）
总分支数：4（每个 Decision Point 各 2 条）

测试：只有 `check(1, 1)`（a=1, b=1，两个都是正数）

| 语句 | 执行了？ |
|------|---------|
| `if a > 0` | ✅ |
| `print("a positive")` | ✅ |
| `if b > 0` | ✅ |
| `print("b positive")` | ✅ |

Statement Coverage = 4/4 = **100%** ← 看起来完美！

| 分支 | 执行了？ |
|------|---------|
| a > 0 为 True | ✅ |
| a > 0 为 False | ❌ |
| b > 0 为 True | ✅ |
| b > 0 为 False | ❌ |

Branch Coverage = 2/4 = **50%** ← 实际上有一半没测到！

**结论：**

> Statement Coverage 100% 不代表 Branch Coverage 100%。  
> 你可以在所有语句都执行过的情况下，仍然有很多分支没被测试。

**Branch Coverage 更强，因为它强制你测试每个判断条件的两种结果（True 和 False）。**

## 4. Untested Branch 的危险

**Untested Branch（未测试的分支）** = 测试从来没有走过的代码路径。

这很危险，因为那条路径里的 bug 完全不可见。

### 真实场景

```python
def withdraw(balance, amount):
    if amount > balance:
        raise ValueError("Insufficient funds")  # ← 如果这个分支没被测试
    return balance - amount
```

如果测试只测了 `withdraw(100, 50)`（金额足够），那 `amount > balance` 为 True 的分支从没被执行过。

意味着：
- `raise ValueError` 这行代码里如果有 typo 或逻辑错误，测试发现不了
- 用户真正余额不足时，程序可能崩溃或产生意外行为

---

## 核心对比总结

| | Statement Coverage | Branch Coverage |
|-|-------------------|-----------------|
| **衡量对象** | 每条语句是否被执行 | 每个判断点的每条分支是否被走过 |
| **粒度** | 行级别 | 决策点级别 |
| **100% 保证** | 每行都执行过 | 每个 if/while 的 True 和 False 都执行过 |
| **强度** | 弱 | 强（Branch > Statement） |
| **盲区** | 漏掉 if-else 的某个分支 | 漏掉复杂条件的子条件（4.3 讨论） |

```
Branch Coverage 100% → Statement Coverage 一定 100%
Statement Coverage 100% → Branch Coverage 不一定 100%
```

## Self-Check

**Q1：** 以下代码有多少个 Decision Point？总共有多少条分支？

```python
def classify(n):
    if n > 0:
        return "positive"
    elif n < 0:
        return "negative"
    else:
        return "zero"
```

<details>
<summary>答案</summary>

- **2 个 Decision Point**：`n > 0` 和 `n < 0`
- **总共 4 条分支**：
  - `n > 0` → True
  - `n > 0` → False（进入 elif）
  - `n < 0` → True
  - `n < 0` → False（进入 else）

要达到 100% Branch Coverage，至少需要测试 3 个值：正数、负数、零。

</details>

---

**Q2：** 一个测试套件的 Statement Coverage = 100%，可以得出 Branch Coverage = 100% 吗？反过来呢？

<details>
<summary>答案</summary>

- **Statement 100% → Branch 100%：不能得出。** 可能每行都执行了，但某些分支（如 if 的 False 路径）从没被走到。
- **Branch 100% → Statement 100%：可以得出。** 所有分支都走过了，必然所有语句都执行过了。

</details>

**Q3：** 以下代码，测试 `process(0)` 后：Statement Coverage 是多少？Branch Coverage 是多少？

```python
def process(x):
    if x > 0:           # 行 1
        x = x * 2      # 行 2
    return x            # 行 3
```

<details>
<summary>答案</summary>

测试 `process(0)`：x=0，不大于 0，所以行 2 不执行。

- **Statement Coverage：** 行 1 ✅，行 2 ❌，行 3 ✅ → 2/3 = **67%**
- **Branch Coverage：**
  - True 分支（x > 0）❌
  - False 分支（x <= 0）✅
  → 1/2 = **50%**

</details>

---

**Q4：** 为什么 Untested Branch 是危险的？举一个具体场景。

<details>
<summary>答案</summary>

Untested Branch 是测试从没走过的代码路径，里面的 bug 完全不可见。

例如：一个 `if balance < amount` 的错误处理分支，如果测试只测了正常扣款的情况，那余额不足时的错误处理逻辑（如抛出异常）从没被验证过。用户真正触发时，程序可能崩溃或产生意外行为。

</details>

**Q5：** 以下代码有多少条分支？要达到 100% Branch Coverage，最少需要几个测试用例？

```python
def fee(age, member):
    if age < 18:
        return 0
    if member:
        return 5
    return 10
```

<details>
<summary>答案</summary>

- **Decision Point 1：** `age < 18` → True / False（2 条分支）
- **Decision Point 2：** `member` → True / False（2 条分支）
- **总计：4 条分支**

要达到 100% Branch Coverage，最少需要 **3 个测试用例**：
1. `fee(10, True)` 或 `fee(10, False)` → 覆盖 age < 18 的 True 分支（member 不影响结果，但 Decision Point 2 不会被评估）
2. `fee(20, True)` → 覆盖 age < 18 False + member True
3. `fee(20, False)` → 覆盖 age < 18 False + member False

</details>
