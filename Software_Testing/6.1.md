## 1. 什么是 TDD？

**Test-Driven Development（TDD，测试驱动开发）** = 一种开发方法，核心规则是：**在写功能代码之前，先写测试。**

传统开发顺序：
```
写代码 → 写测试（如果有时间的话）
```

TDD 顺序：
```
写测试（失败）→ 写代码（让测试通过）→ 改善代码
```

TDD 不只是一种测试技术，它是一种**设计工具**——先写测试，迫使你在动手写代码之前先想清楚这个函数应该做什么、接受什么、返回什么。

---

## 2. Red — 写一个会失败的测试

**Red** = 在功能代码存在之前，先写一个描述期望行为的测试，然后运行它——它一定会失败（因为代码还没写）。

测试结果显示红色（fail），所以叫 Red。

### 为什么要先让它失败？

这步看起来多余，但它有一个重要目的：**确认你的测试是真的在测试你想测的东西。**

如果一个测试在你还没写任何代码的情况下就 pass 了，说明这个测试有问题——它根本没在验证任何东西（回想 4.3：Coverage Without Assertions）。

### 例子

需求：写一个 `add(a, b)` 函数，返回两数之和。

第一步，先写测试（此时 `add` 函数根本不存在）：

```python
def test_add_two_numbers():
    result = add(2, 3)
    assert result == 5
```

运行测试 → **NameError: name 'add' is not defined** → 红色 ✅

这个红色是预期的、正确的。

---

## 3. Green — 写最少的代码让测试通过

**Green** = 写**刚好足够**让失败测试通过的代码，不多写一行。

测试结果变绿色（pass），所以叫 Green。

**关键原则：只写让测试通过所需的最少代码。**

### 例子（接上）

```python
def add(a, b):
    return a + b
```

运行测试 → **PASSED** → 绿色 ✅

注意"最少代码"的含义：不要在这一步顺手加 logging、加 error handling、加其他功能。那些留给下一个 Red-Green 循环。

**为什么只写最少？**
- 防止过度设计（写了没有测试覆盖的代码）
- 保持每步的变化量小，出错时容易定位

---

## 4. Refactor — 在测试保护下改善代码

**Refactor（重构）** = 在不改变行为的前提下，改善代码的结构、可读性、性能。

测试在这里充当**安全网**——只要重构后测试还是绿色，说明行为没有被破坏。

### Refactor 的例子

假设 Green 阶段写出了这样的代码（为了快速通过而写的丑代码）：

```python
def calculate_discount(price, is_member):
    if is_member == True:
        x = price * 0.2
        final = price - x
        return final
    else:
        return price
```

Refactor 阶段，在测试保护下改善：

```python
def calculate_discount(price, is_member):
    if is_member:
        return price * 0.8
    return price
```

重构后立刻运行测试——仍然绿色 ✅，说明行为没变，只是代码变干净了。

**Refactor 的范围：**
- 消除重复代码
- 改善命名
- 简化逻辑
- 拆分过长的函数

**不做的事：** 在 Refactor 阶段不添加新功能，不改变行为。

---

## 5. Short Cycle — 为什么每步要小

**Short Cycle（短循环）** = 每一个 Red-Green-Refactor 循环只处理一个很小的行为，整个循环时间通常在几分钟内完成。

```
Red (几秒) → Green (1~5 分钟) → Refactor (几分钟) → 下一个 Red...
```

### 为什么循环要短？

| 长循环的问题 | 短循环的好处 |
|------------|------------|
| 改动多，出错时难以定位 | 每步改动小，出错立刻知道原因 |
| 红了很久，压力大 | 频繁见到绿色，建立信心 |
| 测试覆盖不了所有代码 | 每一行代码都由测试驱动产生 |

### 完整循环示例

需求：`multiply(a, b)` 返回两数之积。

**循环 1：** 处理正整数相乘
```
Red:    test_multiply(3, 4) == 12  → fail
Green:  return a * b               → pass
Refactor: 代码已够简单，跳过
```

**循环 2：** 处理乘以 0
```
Red:    test_multiply(5, 0) == 0   → fail（如果前面实现有问题才会 fail）
Green:  （已经通过，`a * b` 天然处理 0）→ pass
Refactor: 无需修改
```

**循环 3：** 处理负数
```
Red:    test_multiply(-2, 3) == -6  → fail（如需要）
Green:  （`a * b` 天然处理负数）    → pass
Refactor: 无需修改
```

每个循环只推进一小步。

---

## 6. TDD 的价值

TDD 的好处不只是"有测试"，它从根本上改变了写代码的方式：

| 好处 | 原因 |
|------|------|
| **100% 测试覆盖** | 每一行代码都是为了让某个测试 pass 而写的 |
| **设计更清晰** | 先写测试迫使你想清楚 API 设计（函数叫什么、接受什么、返回什么） |
| **重构有保障** | 已有测试是安全网，随时可以改善代码结构 |
| **快速定位 bug** | 循环短，出错范围小，立刻知道问题在哪 |
| **防止过度设计** | 只写让测试通过所需的代码，不写没用到的功能 |

---

## 核心要点总结

```
TDD = 测试先行的开发方法

Red-Green-Refactor 循环：
  Red      → 先写测试，让它失败（确认测试真的在测东西）
  Green    → 写最少代码让测试通过（不多写）
  Refactor → 在测试保护下改善代码结构（不改行为）

Short Cycle：每个循环处理一个小行为，几分钟内完成
Test-first 的价值：设计更清晰 + 自动获得高 coverage + 重构有保障
```

## Self-Check

**Q1：** TDD 和传统开发的顺序有什么不同？

<details>
<summary>答案</summary>

**传统开发：** 先写功能代码，再写测试（如果有时间的话）。

**TDD：** 先写测试（Red）→ 写最少代码让测试通过（Green）→ 改善代码（Refactor）→ 重复。

核心差异：TDD 测试先行，功能代码是为了让测试通过而写的，而不是测试是为验证代码而写的。

</details>

---

**Q2：** Red 阶段，测试为什么必须先失败？如果一个测试在还没写功能代码时就 pass 了，说明什么？

<details>
<summary>答案</summary>

测试必须先失败，是为了**确认这个测试真的在测试你想测的行为**。

如果还没写任何功能代码，测试就 pass 了，说明：
- 测试没有 assertion（测试根本没验证任何东西）
- 或者测试的逻辑写错了（它不是在测你以为在测的东西）

一个从不失败的测试没有任何价值——它无法区分代码正确和代码错误的情况。

</details>

---

**Q3：** Green 阶段为什么强调"写最少的代码"？多写一点功能有什么问题？

<details>
<summary>答案</summary>

**最少代码原则**防止两个问题：

1. **过度设计：** 写了没有测试覆盖的代码，意味着那部分代码的行为没有被验证过，可能藏有 bug。
2. **难以定位问题：** 如果 Green 阶段改动太多，后续测试 fail 时不知道哪一行代码出了问题。

TDD 的规则：只有当有测试要求时，才写对应的代码。没有测试的需求 → 不写代码。

</details>

---

**Q4：** Refactor 阶段，测试的作用是什么？

<details>
<summary>答案</summary>

测试在 Refactor 阶段充当**安全网**。

重构的定义是"不改变行为，只改善结构"。测试保证了这一点：重构后立刻运行测试，如果仍然全绿，说明行为没有被破坏；如果有测试变红，说明重构改变了某个行为，需要修正。

没有测试的保护，重构是高风险操作；有测试的保护，重构可以大胆进行。

</details>

---

**Q5：** 为什么 TDD 的循环要保持"短"？长循环有什么问题？

<details>
<summary>答案</summary>

**短循环的价值：**
- 每步改动小，测试 fail 时立刻知道是刚才哪一行代码导致的
- 频繁见到绿色，建立信心，保持节奏

**长循环的问题：**
- 改动多，出错时难以定位根本原因
- 在红色状态停留太久，压力大、方向感弱
- 容易写出没有测试覆盖的代码

TDD 的节奏是：快速 Red → 快速 Green → 快速 Refactor → 下一个。

</details>
