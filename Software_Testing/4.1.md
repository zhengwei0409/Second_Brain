## 1. 什么是 Code Coverage？

**Code Coverage（代码覆盖率）** = 你的测试套件实际上执行了多少比例的代码。

想象一下：你有一段代码，里面有 100 行。你的测试运行完之后，只有 80 行真正被执行过，另外 20 行从来没被碰到。那你的 Code Coverage 就是 **80%**。

简单说：
> **Code Coverage 回答的问题是：我的测试有没有"跑到"某段代码？**

这是一个**白盒（White-Box）测量**——你需要看代码内部才能知道哪些行被执行了。

---

## 2. Coverage 怎么计算？

最基础的公式：

```
Coverage % = (被测试执行到的代码单元数 / 总代码单元数) × 100
```

**"代码单元"** 可以是：
- 一行代码（line / statement）
- 一个分支（branch）
- 一个函数（function）

在 4.1，我们先停在最直觉的概念：**执行到的行数 vs 总行数**。

### 例子

```python
def classify_age(age):
    if age < 0:           # 行 1
        return "invalid"  # 行 2
    elif age < 18:        # 行 3
        return "minor"    # 行 4
    else:
        return "adult"    # 行 5
```

假设你只写了一个测试，传入 `age = 25`：

- 行 1：✅ 执行了（判断 `25 < 0`）
- 行 2：❌ 没执行（25 不小于 0）
- 行 3：✅ 执行了（判断 `25 < 18`）
- 行 4：❌ 没执行（25 不小于 18）
- 行 5：✅ 执行了（走到 else，return "adult"）

执行了 3 行，总共 5 行 → Coverage = **3/5 = 60%**

---

## 3. Coverage Percentage 代表什么？

| Coverage % | 代表的意思 |
|-----------|-----------|
| 100% | 每一行代码都被至少一个测试执行过 |
| 80% | 有 20% 的代码从没被测试执行过 |
| 0% | 测试完全没运行到这段代码 |

**高 Coverage 的意义：**
- 低 Coverage 是警告信号——有代码"死角"完全没被测试触碰过，里面藏着 bug 你完全不知道
- Coverage 越高，你对代码的信心越高（但不等于没有 bug，4.3 会讲这个陷阱）

**低 Coverage 例子：**

```python
def divide(a, b):
    if b == 0:
        raise ValueError("division by zero")  # 如果测试从没传 b=0，这行永远 0% coverage
    return a / b
```

如果你只测试了 `divide(10, 2)`，那 `raise ValueError` 那行的 coverage = 0%。
这意味着如果用户真的传入 `b=0`，你完全不知道会发生什么。

## 4. Coverage Report 长什么样？

Coverage Tool 运行完之后，会生成一份 **Coverage Report（覆盖率报告）**，告诉你每个文件、每个函数、每一行的执行情况。

### 终端输出（文字版）

```
Name              Stmts   Miss  Cover
-------------------------------------
calculator.py        10      2    80%
utils.py             20      5    75%
-------------------------------------
TOTAL                30      7    77%
```

- **Stmts** = 总语句数
- **Miss** = 没被执行的语句数
- **Cover** = 覆盖率百分比

### HTML 报告（视觉版）

Coverage Tool 还能生成 HTML 页面，用颜色标注每一行：

```
  1  def classify_age(age):
  2      if age < 0:           ← 绿色（执行了）
  3          return "invalid"  ← 红色（从没执行）
  4      elif age < 18:        ← 绿色（执行了）
  5          return "minor"    ← 红色（从没执行）
  6      else:
  7          return "adult"    ← 绿色（执行了）
```

红色的行 = 你需要补测试的地方。

---

## 5. Coverage Tool 是怎么工作的？

Coverage Tool 的工作原理叫做 **Instrumentation（插桩）**：

1. **Before：** 你的原始代码
2. **Instrumentation：** 工具在每一行代码前后插入隐形的"记录器"
3. **Run tests：** 测试跑的时候，记录器标记哪些行被执行了
4. **Report：** 工具汇总所有记录，生成报告

你不需要修改你的源代码，Coverage Tool 在背后自动处理。

### 常见 Coverage Tool

| 语言 | 工具 |
|------|------|
| Python | `coverage.py`（`pytest-cov`） |
| JavaScript | `Istanbul / NYC`（`Jest` 内建） |
| Java | `JaCoCo` |
| Go | `go test -cover`（内建） |

### 运行方式（Python 例子）

```bash
# 安装
pip install pytest-cov

# 跑测试同时收集 coverage
pytest --cov=your_module tests/

# 生成 HTML 报告
pytest --cov=your_module --cov-report=html tests/
```

## 核心要点总结

```
Code Coverage = 测试执行了多少比例的代码

计算：已执行代码行数 / 总代码行数 × 100%

低 Coverage → 有代码死角 → bug 可能藏在那里
Coverage Report → 红色行 = 没被测试触碰的代码
Coverage Tool → 用 Instrumentation 自动追踪
```

## Self-Check

**Q1：** 一个函数有 10 行代码，测试运行后有 3 行从未执行。Code Coverage 是多少？

<details>
<summary>答案</summary>

(10 - 3) / 10 × 100% = **70%**

</details>

---

**Q2：** 以下代码，如果测试只传入 `score = 85`，哪些行的 coverage 是 0%？

```python
def grade(score):
    if score >= 90:
        return "A"       # 行 A
    elif score >= 70:
        return "B"       # 行 B
    else:
        return "C"       # 行 C
```

<details>
<summary>答案</summary>

- `score = 85` → 进入 `elif score >= 70` → 执行行 B
- 行 A（需要 score >= 90）：**0% coverage**
- 行 C（需要 score < 70）：**0% coverage**

</details>

---

**Q3：** Coverage Report 中，一个文件显示 `Miss = 5, Stmts = 20`。这意味着什么？

<details>
<summary>答案</summary>

有 5 行代码从未被测试执行过，总共 20 行。  
Coverage = (20 - 5) / 20 × 100% = **75%**。  
这 5 行是潜在的测试盲区。

</details>

---

**Q4：** Coverage Tool 是怎么知道哪些代码行被执行了？它需要你修改源代码吗？

<details>
<summary>答案</summary>

Coverage Tool 使用 **Instrumentation（插桩）** 技术——它在运行时在你的代码中自动插入隐形记录器，追踪哪些行被执行。  
**不需要**修改你的源代码，工具在背后自动处理。

</details>

---

**Q5：** Coverage = 100% 是否代表程序没有 bug？

<details>
<summary>答案</summary>

**不代表。**  
100% coverage 只意味着每一行代码都被执行过至少一次。  
但执行过不等于验证了结果是否正确——如果测试没有 assertion，或 assertion 不够精确，bug 依然可以存在。  
这个陷阱在 4.3 会深入讨论。

</details>
