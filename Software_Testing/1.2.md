## 1. 软件测试的定义

**Software Testing** 是一个 **收集证据** 的过程 — 用来判断软件的行为是否符合预期。

注意这个定义里的关键词：**证据（evidence）**。

测试不是"证明软件没有 bug"，而是"通过运行软件，收集它行为是否正确的证据"。

> 你写了一个计算器程序。测试就是：输入 `2 + 3`，看输出是不是 `5`。这个过程就是在收集"加法功能是否正确"的证据。

---

## 2. Verification vs Validation

这是两个经常被混淆的词，区别在于你问的问题不同：

| 术语 | 问的问题 | 关注点 |
|------|---------|-------|
| **Verification（验证）** | "我们把东西做对了吗？" | 符合规格说明（specification） |
| **Validation（确认）** | "我们做的是对的东西吗？" | 满足用户的真实需求 |

**用一个例子来区分：**

你的需求文档说："用户点击删除按钮后，弹出确认框"。

- **Verification**：点击删除，确认框确实弹出了。✅ 符合规格。
- **Validation**：但用户实际上希望的是"撤销"功能，而不是"确认删除"。❌ 不满足真实需求。

软件可以通过 verification 但失败于 validation — 这就是为什么需要两种视角。

---

## 3. Expected vs Actual Behavior

测试的核心机制就是比较两件事：

- **Expected behavior（预期行为）**：根据需求，程序 *应该* 做什么
- **Actual behavior（实际行为）**：程序 *实际上* 做了什么

```
Expected: 输入 -1 作为年龄，程序应该返回错误信息 "Invalid age"
Actual:   程序接受了 -1，没有任何报错

→ Expected ≠ Actual → 测试失败（Test Fail）
```

如果 expected == actual，测试通过（Pass）。
如果 expected ≠ actual，测试失败（Fail），说明可能存在 fault。

**没有 expected behavior，就没有办法判断测试是否通过。** 这就是为什么测试必须事先定义"正确答案"。

---

## 4. Testing vs Debugging

这两个词经常被新手混用，但它们是完全不同的活动：

| | **Testing（测试）** | **Debugging（调试）** |
|-|-------------------|---------------------|
| **目的** | 发现 failure 是否存在 | 找到并修复 fault |
| **问题** | "程序有没有出错？" | "程序在哪里出错、为什么出错？" |
| **输出** | Pass / Fail 的结论 | 修复后的代码 |
| **由谁做** | 测试工程师 / 自动化工具 | 开发工程师 |

**流程是这样的：**

```
Testing → 发现 Failure
              ↓
         Debugging → 找到 Fault → 修复 → 再次 Testing
```

测试告诉你"有问题"，调试告诉你"问题在哪、怎么修"。

---

## 5. "Testing Shows Presence of Bugs"

这句话来自计算机科学先驱 **Edsger Dijkstra**，是软件测试领域最重要的一句话：

> **"Testing can show the presence of bugs, but never their absence."**
> 测试只能证明 bug 存在，但永远无法证明 bug 不存在。

**为什么？**

因为你不可能穷举所有输入。

```
一个接受整数输入的函数：
可能的输入有 2^32 种（约 42 亿个）

就算你测试了 100 万种，
剩下的 41 亿种你没测过
→ 你不能说"没有 bug"
```

**实际意义：**

- 测试通过 ≠ 软件没有 bug
- 测试的目标是 **降低风险**，不是 **消灭所有风险**
- 这也是为什么测试需要聪明地选择测试用例（后面的 3.x 章节会讲）

## 6. Quality Assurance（QA）

**Quality Assurance（质量保证）** 是一个比"测试"更宽泛的概念。

测试是 QA 的一部分，但 QA 还包括：

```
QA（质量保证）
├── Testing（测试）← 你现在学的
├── Code Review（代码审查）
├── Static Analysis（静态分析工具）
├── Process Standards（开发流程规范）
└── Documentation Review（文档审查）
```

简单说：
- **Testing** = 运行软件，观察行为，收集证据
- **QA** = 所有用来提高软件质量的活动，包括测试

在很多公司，"QA Engineer"和"Test Engineer"这两个职位名字经常混用，但严格来讲，QA 的范围更广。

## Self-Check

**Q1.** 一家公司按照需求文档开发了一个工资计算系统。测试发现所有功能都符合文档要求。但上线后，HR 说："这不是我们想要的，我们需要的是能自动生成报表的功能。"

请问：这个系统通过了 Verification 还是 Validation？哪个失败了？

**A1.**
- **Verification 通过了**：系统符合需求文档（做对了文档里写的东西）
- **Validation 失败了**：系统不满足用户的真实需求（没做对的东西）

---

**Q2.** 测试工程师运行了一个测试，输入用户名为空，期望系统返回 `"Username cannot be empty"`，但系统实际返回了 `"Error 500"`。

请问：Expected behavior 是什么？Actual behavior 是什么？这个测试 Pass 还是 Fail？

**A2.**
- Expected behavior：系统返回 `"Username cannot be empty"`
- Actual behavior：系统返回 `"Error 500"`
- Expected ≠ Actual → **测试 Fail**

---

**Q3.** 开发工程师说："我已经对这个功能测试了 500 个用例，全部通过，所以这个功能肯定没有 bug。" 这个说法有什么问题？

**A3.** 这个说法违反了 Dijkstra 的原则："Testing can show the presence of bugs, but **never their absence**."

500 个测试用例通过，只能说这 500 种情况下没有发现 bug，**不能证明其他未测试的情况下没有 bug**。软件的可能输入通常是无限的，测试只能降低风险，不能消灭所有风险。

---

**Q4.** Testing 和 Debugging 有什么区别？以下哪个描述属于 Testing，哪个属于 Debugging？

- (a) 发现程序在输入负数时崩溃
- (b) 在代码第 47 行找到了导致崩溃的除以零的错误，并修复它

**A4.**
- (a) = **Testing**：运行程序，发现了 failure（崩溃）
- (b) = **Debugging**：定位到 fault（第 47 行），并修复它
