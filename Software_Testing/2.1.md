## 1. 什么是 Unit？

**Unit（单元）** 是软件里 **可以独立测试的最小代码块**。

在大多数语言里，一个 unit 通常是：
- 一个 **函数（function）**
- 一个 **方法（method）**
- 有时候是一个 **类（class）**

```python
def calculate_discount(price, rate):
    return price * (1 - rate)
```

这个函数就是一个 unit — 它有明确的输入、明确的输出、功能单一，可以被独立测试。

---

## 2. 什么是 Unit Testing？

**Unit Testing（单元测试）** 是 **针对单个 unit 写的测试**，目的是验证这个 unit 的逻辑是否正确。

```python
def test_calculate_discount():
    # Arrange
    price = 100
    rate = 0.2

    # Act
    result = calculate_discount(price, rate)

    # Assert
    assert result == 80.0
```

这个测试只关心 `calculate_discount` 这一个函数是否算对了。它不管：
- 这个函数在哪里被调用
- 数据从哪里来
- 结果会被怎么用

---

## 3. Isolation（隔离）

**Isolation（隔离）** 是 unit testing 最核心的概念：**被测的 unit 必须和它的依赖隔离开来。**

### 什么是依赖？

**Dependency（依赖）** 是指被测 unit 需要调用的外部东西：

```python
def get_user_discount(user_id):
    user = database.get_user(user_id)   # ← 依赖：数据库
    email = email_service.send(user)    # ← 依赖：邮件服务
    return calculate_discount(user.tier)
```

`get_user_discount` 依赖了数据库和邮件服务。

### 为什么要隔离？

如果不隔离，测试会有问题：

| 问题 | 原因 |
|------|------|
| 测试很慢 | 真实数据库查询需要几百毫秒 |
| 测试不稳定 | 数据库里的数据可能变，今天通过明天失败 |
| 测试复杂 | 需要先搭一个真实的数据库环境 |
| 失败原因不明 | 是函数逻辑错了？还是数据库挂了？ |

**隔离的目标：当测试失败时，你能确定是被测 unit 本身的逻辑有问题，而不是某个依赖在捣鬼。**

### 怎么隔离？

用 **Test Double（测试替身）** 来替换真实的依赖 — 这是 5.x 章节会详细讲的内容。现在只需要知道：unit testing 要求你把被测函数和外部世界隔开。

---

## 4. Fast Feedback（快速反馈）

Unit testing 最重要的优势之一就是 **速度快**。

因为：
- 没有真实数据库连接
- 没有网络请求
- 没有文件读写
- 只是纯粹的函数调用

一个 unit test 通常在 **毫秒级** 完成。一个项目可以有几千个 unit test，全部跑完可能只需要几秒。

**为什么速度重要？**

```
开发者改了一行代码
        ↓
立刻运行所有 unit tests（几秒）
        ↓
立刻知道有没有破坏现有功能
        ↓
错了就立刻修，没有错就继续写下一行
```

这个循环越快，开发者就能越早发现问题。如果测试要跑 20 分钟，开发者就不会频繁运行它，问题就会积累。

## 5. Function-level Testing（函数级测试）

Unit testing 专注在 **函数这一层**。你测的是：

- 给定这个输入，函数返回正确的值吗？
- 给定边界值（比如 0、负数、空字符串），函数处理正确吗？
- 给定无效输入，函数是否正确地抛出错误？

```python
# 测正常情况
def test_discount_normal():
    assert calculate_discount(100, 0.2) == 80.0

# 测边界：折扣为 0
def test_discount_zero_rate():
    assert calculate_discount(100, 0.0) == 100.0

# 测边界：全额折扣
def test_discount_full_rate():
    assert calculate_discount(100, 1.0) == 0.0

# 测无效输入
def test_discount_negative_price():
    with pytest.raises(ValueError):
        calculate_discount(-10, 0.2)
```

每个函数的不同行为场景，各自写一个独立的测试。

---

## 6. Independence（测试独立性）

**Independence** 是指：**每一个 unit test 都必须能独立运行，不依赖其他测试的执行顺序或结果。**

**错误的做法：**

```python
cart = None  # 全局变量

def test_add_item():
    global cart
    cart = ShoppingCart()
    cart.add_item("apple", 3.0)
    assert cart.item_count() == 1

def test_get_total():
    # 依赖 test_add_item 先跑过！
    total = cart.get_total()
    assert total == 3.0
```

如果 `test_add_item` 没有先跑，`test_get_total` 会因为 `cart` 是 `None` 而崩溃。

**正确的做法：**

```python
def test_add_item():
    cart = ShoppingCart()   # 自己负责 Arrange
    cart.add_item("apple", 3.0)
    assert cart.item_count() == 1

def test_get_total():
    cart = ShoppingCart()   # 自己负责 Arrange
    cart.add_item("apple", 3.0)
    total = cart.get_total()
    assert total == 3.0
```

每个测试自己 Arrange，自己 Act，自己 Assert。

**为什么 independence 重要？**

- 测试可以以任意顺序运行
- 可以只运行其中一个测试
- 一个测试失败不会连带影响其他测试

## 7. 完整例子

**被测代码：**

```python
def is_eligible_for_senior_discount(age):
    if age < 0:
        raise ValueError("Age cannot be negative")
    return age >= 65
```

**Unit Tests：**

```python
def test_senior_eligible():
    # Arrange
    age = 65
    # Act
    result = is_eligible_for_senior_discount(age)
    # Assert
    assert result == True

def test_not_senior():
    # Arrange
    age = 64
    # Act
    result = is_eligible_for_senior_discount(age)
    # Assert
    assert result == False

def test_negative_age_raises_error():
    # Arrange + Act + Assert
    with pytest.raises(ValueError):
        is_eligible_for_senior_discount(-1)
```

注意：
- 每个测试只测一种场景（Single Behavior Focus）
- 每个测试自己 Arrange，完全独立（Independence）
- 没有依赖数据库、网络等外部系统（Isolation）
- 三个测试加起来跑完不超过 1 毫秒（Fast Feedback）

## Self-Check

**Q1.** 以下哪个最符合"unit"的定义？

- (a) 整个用户登录流程（从输入密码到跳转首页）
- (b) 一个验证密码格式是否合法的函数
- (c) 前端、后端、数据库加在一起的注册功能

**A1.** **(b)** 是 unit。它是一个功能单一、有明确输入输出、可以独立测试的最小代码块。(a) 和 (c) 涉及多个组件的交互，属于更高层次的测试范畴。

---

**Q2.** 一个函数 `send_invoice(order_id)` 会查数据库取订单、生成 PDF、发送邮件。你想对它做 unit testing，但发现测试运行很慢而且经常因为数据库连接问题失败。这违反了 unit testing 的哪个原则？

**A2.** 违反了 **Isolation（隔离）** 原则。测试没有把被测函数和它的依赖（数据库、PDF 生成器、邮件服务）隔离开，导致：(1) 测试慢，因为要真实连接数据库；(2) 测试不稳定，因为依赖外部系统的状态。

---

**Q3.** 以下两个测试有什么问题？怎么修？

```python
shared_user = None

def test_create_user():
    global shared_user
    shared_user = User(name="alice")
    assert shared_user.name == "alice"

def test_user_has_default_role():
    assert shared_user.role == "viewer"
```

**A3.** 违反了 **Independence（独立性）** 原则。`test_user_has_default_role` 依赖 `test_create_user` 先执行并设置 `shared_user`。如果单独运行 `test_user_has_default_role`，它会因为 `shared_user` 是 `None` 而崩溃。

修法：让每个测试自己在 Arrange 阶段创建所需的对象：

```python
def test_create_user():
    user = User(name="alice")
    assert user.name == "alice"

def test_user_has_default_role():
    user = User(name="alice")   # 自己 Arrange
    assert user.role == "viewer"
```

---

**Q4.** Unit testing 的 Fast Feedback 为什么重要？如果一个项目的全套测试要跑 30 分钟，会带来什么实际问题？

**A4.** 如果测试要跑 30 分钟，开发者不会每次改完代码就跑一次，他们可能只在提交代码前才跑。这意味着一个 bug 可能在被发现前已经积累了几小时的代码改动，调试时很难定位到底是哪一行引入了问题。Fast Feedback 让开发者保持"改一行，跑一次"的习惯，问题立刻暴露，立刻修复。
