8.2 是关于**怎么"闻出"写得不好的测试**——也就是"这个测试虽然能跑、也会过，但它其实埋了一个隐患，只是你现在还没发现而已"。这是 8.1 的自然延伸：8.1 教你识别"结果不稳定"的测试，8.2 教你识别"结果稳定但写法有问题"的测试。

## 1. High-Level Map：这个概念在整个体系里的位置

回想一下 [[8.1]]，flaky test 的症状很好抓——它会**忽好忽坏**，你一眼就能发现"这个测试怪怪的"。

但 8.2 讲的问题更隐蔽：**测试今天能跑、结果稳定、CI 也一直是绿色的 (passing)**——表面上完全健康。但这个测试的**写法本身**有问题，导致：
- 以后代码稍微改一下，这个测试就无缘无故炸了（即使功能其实没坏）
- 或者这个测试其实**没有真的在测试什么东西**，只是看起来有测试而已
- 或者没有人敢去改这个测试，因为没人看得懂它在测什么

这些"闻起来不对劲，但还没真正出事"的写法模式，就叫 **test smell**（测试异味）。这个概念是从更广为人知的 "code smell"（代码异味）借来的——**smell 不等于 bug**，它是一种"警讯"，代表"这里以后大概率会出问题，最好先处理"。

## 2. Fundamentals First：为什么"能过"的测试也可能有问题？

先纠正一个初学者常有的误解：**"测试是绿色的（通过）" 不等于 "这个测试写得好"**。

一个测试可以：
- 通过 ✅
- 但其实**测试内容跟你的代码逻辑没什么关系**（比如它只是断言 `true === true`）
- 或者**写法太脆弱**，代码只要稍微重构 (refactor)，就算功能完全没变，测试也会跟着失败

这种测试是有害的，因为它给你一种**假的安全感**——你以为有测试保护，实际上它形同虚设，或者它反而在拖慢你开发的速度（因为你要一直花时间去"哄"这些脆弱的测试通过）。

8.2 教你的，就是训练你的"嗅觉"，去发现这些隐患。

## 3. Progressive Explanation：一个一个概念拆解

### (a) Test Smell（测试异味）

这是这一节的总称概念：**测试代码里，某种"写法模式"，虽然不会马上导致测试失败，但代表这个测试的品质或可维护性有问题**。

跟 "smell" 这个字的意象一样——闻到臭味不代表东西已经坏了，但代表"你最好去检查一下，不然它迟早会坏"。

下面四个，就是这一节要你认识的四种具体的 test smell。

### (b) Fragile Test（脆弱测试）

**定义**：这个测试对代码的"无关改动"过度敏感——明明你只是重构了程式内部的写法（逻辑完全没变），这个测试却失败了。

举个例子：

```javascript
test("计算总价", () => {
  const cart = new ShoppingCart();
  cart.addItem("apple", 3);
  cart.addItem("banana", 2);
  expect(cart.internalItemList.length).toBe(2); // 检查了内部实现细节
});
```

这个测试检查的是 `internalItemList`——这是 `ShoppingCart` **内部**怎么存资料的细节。如果哪天你把内部实现从 array 改成 Map（但对外的行为、总价计算完全没变），这个测试就会莫名其妙失败。

**关键点**：好的测试应该只关心**外部行为/结果**（比如"总价对不对"），而不是**内部怎么实现**的。检查内部实现细节的测试就是 fragile——它把测试跟"实现方式"绑得太紧。

### (c) Over-mocking（过度模拟）

先解释 **mock** 是什么：mock 是一个"假的"替代品，用来取代测试中不想真正调用的东西（比如真的数据库、真的 API）。适度使用 mock 是合理的（比如你不想在每次测试时都真的发一封 email）。

但 **over-mocking** 指的是：**mock 用得太多、太细，导致这个测试已经不是在测试"真实行为"，而是在测试"你自己写的假货 (mock) 有没有照你的预期运作"**。

举个极端例子：如果你把一个函数里**所有**用到的东西都 mock 掉，那这个测试基本上只是在验证"我写的 mock 确实会回传我叫它回传的东西"——这句话听起来就很荒谬，因为它没有测到任何**真实**的逻辑。

**关键点**：Over-mocking 会让测试通过，但这个"通过"没有意义——它没有真正验证你的代码在**真实情况下**能不能正常运作。

### (d) Duplicated Setup（重复的准备代码）

这个概念直接呼应你在 [[7.2]] 学过的 setup/beforeEach。

**定义**：多个测试案例里，都各自写了一模一样（或高度相似）的准备代码，而不是用 `beforeEach` 或共用的 fixture 统一处理。

```javascript
test("测试 A", () => {
  const user = { name: "test", age: 20, email: "a@test.com" }; // 重复
  // ...
});

test("测试 B", () => {
  const user = { name: "test", age: 20, email: "a@test.com" }; // 重复
  // ...
});

test("测试 C", () => {
  const user = { name: "test", age: 20, email: "a@test.com" }; // 重复
  // ...
});
```

**为什么这是 smell**：如果以后 `user` 这个 fixture 需要多加一个栏位（比如 `phoneNumber`），你就要跑去改**每一个**测试案例，很容易漏改、容易出错。这跟 [[7.2]] 提到"用 beforeEach 避免重复"的初衷是同一件事——只是 8.2 让你学会**在既有代码里发现**这种"该用 hook 却没用"的坏味道。

### (e) Unclear Intent（意图不清）

**定义**：光看这个测试的代码（或它的名字），**看不出来它到底想验证什么**。

```javascript
test("test1", () => {
  const result = process(data, true, false, 3);
  expect(result).toBe(42);
});
```

这个测试完全没告诉你：
- 这个函数在测什么场景？
- 那些 `true`、`false`、`3` 是什么意思？
- 为什么预期结果是 `42`，这个数字代表什么？

**为什么这是 smell**：几个月后（甚至可能几天后），当这个测试失败时，没有人（包括写这段代码的你自己）能立刻看懂"这代表哪里坏了"。一个好的测试应该像一份**文件 (documentation)**——光看测试名字和内容，就能看懂这个函数应该要有什么行为。

对比一个"意图清楚"的写法：

```javascript
test("当用户是 VIP 且购买满 3 件商品时，应套用九折优惠", () => {
  const result = calculateDiscount({ isVip: true, itemCount: 3 });
  expect(result.discountRate).toBe(0.9);
});
```

## 4. Scope 提醒

这一节**只教你怎么"发现"这些坏味道**（识别警讯）。**该怎么动手修复它们**（比如怎么重构一个 fragile test，怎么正确使用 mock）是下一步的内容，8.2 目前不涉及具体的 refactoring 策略。

## 5 个你必须完全掌握的核心概念

**1. Test Smell**
- 定义：测试代码中的某种写法模式，虽然测试目前仍然通过，但代表测试的品质或可维护性有隐患。
- 为什么重要：帮助你在问题真正爆发（测试无故失败、没人敢改代码）之前，提早发现并处理。
- 检验问题：一个测试目前是绿色的（通过），是否代表它没有任何问题？
  *答案：不一定。测试通过只代表"目前没有失败"，但它仍然可能存在 test smell，比如脆弱、意图不清等，这些问题会在未来造成麻烦。*

**2. Fragile Test**
- 定义：对代码的无关改动（比如内部重构）过度敏感，导致功能没坏但测试却失败的测试。
- 为什么重要：它会让工程师不敢放心重构代码，因为担心"改了内部写法，测试又要跟着炸"。
- 检验问题：一个测试检查了函数内部变量的实现细节（而不是函数对外的行为），这样写有什么风险？
  *答案：只要内部实现方式改变（即使外部行为完全没变），这个测试就可能失败，这就是 fragile test 的典型症状。*

**3. Over-mocking**
- 定义：过度使用 mock（假替代品），导致测试实际上只是在验证"假货有没有照预期运作"，而不是验证真实逻辑。
- 为什么重要：会造成测试通过、但没有真正验证代码在现实情况下是否正确的假象。
- 检验问题：如果一个函数里所有依赖的东西都被 mock 掉了，这个测试还能验证什么？
  *答案：几乎无法验证任何真实逻辑，因为所有会产生"真实行为"的部分都被假货取代了，测试变得没有意义。*

**4. Duplicated Setup**
- 定义：多个测试案例中重复写了相同或高度相似的准备代码，而没有用 beforeEach 或共用 fixture 统一处理。
- 为什么重要：一旦需要修改准备逻辑，必须逐一修改每个测试，容易遗漏且难以维护。
- 检验问题：如果 10 个测试都各自写了同样的一段"建立测试用户"的代码，之后要幫这個用户新增一個欄位，會有什麼麻煩？
  *答案：必须找到并修改全部 10 个地方，很容易漏改，正确做法应该是把这段代码抽到 beforeEach 或共用 fixture 里统一管理。*

**5. Unclear Intent**
- 定义：光看测试的代码或名字，无法看出它到底想验证什么场景与预期行为。
- 为什么重要：好的测试应该像文件一样，能让人一眼看懂代码该有的行为；意图不清的测试在失败时难以判断问题所在。
- 检验问题：为什么 `test("test1", ...)` 这样的命名方式是一种 test smell？
  *答案：因为这个名字完全没有传达这个测试在验证什么场景或预期结果，当它失败时，没有人能立刻理解问题出在哪里。*
