## 1. Test Pyramid 是什么

**Test Pyramid（测试金字塔）** 是一个视觉模型，描述三层测试的**理想数量比例**。

```
        /\
       /  \
      / E2E \          ← 少量（个位数到几十个）
     /--------\
    /Integration\      ← 中等数量
   /--------------\
  /   Unit Tests   \   ← 大量（可以有几百上千个）
 /------------------\
```

- **底层最宽** = Unit test 最多
- **中间** = Integration test 适量
- **顶层最窄** = E2E test 最少

这个形状不是随便定的，背后有严格的逻辑。

---

## 2. 为什么是这个形状？Speed vs Confidence

每一层测试有两个核心属性，而这两个属性是**反向的**：

| 属性 | 含义 |
|------|------|
| **Speed（速度）** | 这层测试跑一次要多久 |
| **Confidence（信心）** | 通过这层测试，我对软件有多少把握 |

### 三层对比

| 层次 | Speed | Confidence | 原因 |
|------|-------|-----------|------|
| Unit test | 极快（毫秒级） | 低（只验证一个函数） | 不连数据库、不启动服务，纯逻辑 |
| Integration test | 中等（秒级） | 中 | 需要启动部分服务、连数据库 |
| E2E test | 慢（十秒到分钟级） | 高（模拟真实用户） | 启动完整系统，浏览器渲染，等网络 |

### 这个矛盾怎么解决？

**用数量来补偿。**

Unit test 单次信心低，但你可以写 500 个，覆盖所有函数的所有场景。500 个 unit test 可能在 10 秒内跑完。

E2E test 信心高，但跑一次要 2 分钟。如果你写了 200 个 E2E test，CI 要等 7 小时——完全不可接受。

**结论：速度越快的层，写得越多；速度越慢的层，只保留最关键的。**

---

## 3. Cost per level — 每一层的代价

"代价" 不只是运行时间，还包括：

| 代价类型 | Unit | Integration | E2E |
|----------|------|-------------|-----|
| **运行时间** | 毫秒 | 秒 | 十秒~分钟 |
| **维护成本** | 低（逻辑变了才改） | 中 | 高（UI 改个 class name 就坏） |
| **环境依赖** | 无 | 需要数据库/服务 | 需要完整系统 |
| **调试难度** | 低（直接定位函数） | 中 | 高（不知道是哪层出问题） |
| **编写难度** | 低 | 中 | 高 |

**金字塔的本质：把大部分测试预算花在 cost 最低的层（unit），少量预算花在 cost 最高的层（E2E）。**

---

## 4. Ice-cream cone anti-pattern — 反面教材

**Ice-cream cone（冰淇淋锥）** 是 Test Pyramid 的反面——一个常见的错误分布。

```
     /--------\
    /          \
   /    E2E     \      ← 大量（团队只信任"手动测试或跑全流程"）
  /--------------\
 /  Integration   \    ← 中等
/------------------\
        ||
        ||            ← Unit test 几乎没有（"太麻烦了"）
        ||
```

形状像冰淇淋球放在甜筒上，倒过来了。

### 为什么有团队会走到这里？

- 开发者觉得写 unit test 麻烦，跳过了
- QA 团队主要负责手动测试或 E2E 测试
- 项目初期没有测试文化，后来补 test 直接写 E2E

### 后果

| 问题 | 表现 |
|------|------|
| CI 极慢 | 每次 push 要等 1~2 小时才知道结果 |
| Flaky tests 多 | E2E 依赖太多外部因素，随机失败 |
| 调试地狱 | E2E 失败，不知道是哪个函数出问题 |
| 反馈循环断裂 | 开发者不愿意等，开始跳过测试 |

**Ice-cream cone 是 anti-pattern（反模式）**——它描述一个你应该避免的常见错误结构。

---

## 5. 实际团队怎么用

Test Pyramid 不是硬性规定，是指导原则。实际团队会根据产品类型调整比例。

### 典型比例

```
E2E:           5~10 个（只覆盖最核心的 happy path）
Integration:   20~50 个（覆盖关键接口和数据库交互）
Unit:          200~500+ 个（覆盖所有业务逻辑函数）
```

### 什么情况调整？

| 情况 | 调整方向 |
|------|----------|
| 纯 API 后端（没有 UI） | E2E 可以稍多（直接调 API，不需要浏览器） |
| 业务逻辑复杂（金融、医疗） | Unit test 比例更高，确保每个计算正确 |
| UI 密集型产品 | Integration test 可以更多（测组件渲染） |

---

## 6. 全图：三层 + 金字塔总结

```
                 /\
                /  \
         慢    / E2E \    信心最高
              /  少量  \
             /----------\
            / Integration \
           /    中等数量   \
          /----------------\
         /    Unit Tests    \
        /      大量          \   速度最快
       /----------------------\
       
       快 ←——————————————→ 慢
       低信心 ←————————→ 高信心
       低成本 ←————————→ 高成本
```

**记住这条规则：**
> 越往上，越贵，越少写。越往下，越便宜，越多写。

## Self-Check

**Q1.** Test Pyramid 解决的是什么问题？

> **A:** 它告诉你三层测试（Unit / Integration / E2E）应该以什么比例分配。核心答案是：Unit test 最多，Integration test 适量，E2E test 最少。

---

**Q2.** 为什么 Unit test 要写最多，E2E test 要写最少？

> **A:** Unit test 运行极快（毫秒级），成本低，可以大量写来覆盖所有场景。E2E test 运行慢（分钟级），维护成本高，大量写会导致 CI 极慢、反馈循环断裂，所以只保留最关键的流程。

---

**Q3.** Speed 和 Confidence 在三层测试里是什么关系？

> **A:** 反向关系。Unit test 速度最快但单次信心最低（只测一个函数）；E2E test 速度最慢但信心最高（模拟真实用户全流程）。Test Pyramid 用"数量"来弥补 unit test 信心低的问题。

---

**Q4.** 什么是 Ice-cream cone anti-pattern？为什么它是个问题？

> **A:** Ice-cream cone 是 Test Pyramid 的倒置——E2E test 最多，Unit test 几乎没有。问题在于：CI 运行极慢（全是慢速 E2E test），flaky test 多（E2E 依赖外部因素），调试困难（E2E 失败无法定位具体哪个函数出错），最终开发者开始跳过测试。

---

**Q5.** 一个团队的 E2E test 每次都需要 3 小时才能跑完，这违反了 Test Pyramid 的什么原则？应该怎么改？

> **A:** 违反了"E2E test 数量要少"的原则（高 cost per level 的测试不应该大量堆积）。改法：把大量 E2E test 下沉——能用 integration test 覆盖的改成 integration test，能用 unit test 覆盖的改成 unit test，只保留少量 E2E test 覆盖最核心的 user flow。
