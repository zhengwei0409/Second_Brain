8.1 是关于**测试本身不可靠的问题**——也就是"同一个测试，同一份代码，为什么有时候过、有时候不过？"。这是一个新的大章节（第 8 章：Test Quality & Maintenance），意味着我们要从"怎么写/怎么跑测试"进阶到"怎么判断测试写得好不好、健不健康"。

## 1. High-Level Map：这个概念在整个体系里的位置

回想一下 [[7.3]]，我们学的是 CI 会自动跑测试，测试没过就当作 **Pull Request Gate**——挡住代码合并。这个机制的前提假设是：**测试的结果是可信的**。如果测试通过，代表代码真的没问题；如果测试失败，代表代码真的有问题。

但如果这个前提被打破了呢？如果测试**本身就不可靠**——同一份代码，这次跑过，下次跑却失败——那 PR gate 就会开始"乱挡人"，工程师会开始不信任测试，甚至养成"测试失败就重跑一次，反正是它的问题不是我的问题"的坏习惯。

8.1 就是在教你**识别**这种不可靠的测试，因为它是团队信任 CI 系统的头号杀手。

## 2. Fundamentals First：什么是"不可靠"的测试？

先建立一个基本概念：一个好的测试应该是**确定性的 (deterministic)**——意思是，只要你测试的代码没有改变，这个测试**每一次跑，结果都应该一模一样**（要嘛永远通过，要嘛永远失败）。

但有一种测试，代码明明没有任何改动，它却：
- 这次跑：通过 ✅
- 下次跑（同样的代码，什么都没改）：失败 ❌
- 再跑一次：又通过了 ✅

这种"时好时坏、结果无法预测"的测试，就叫 **flaky test**（不稳定测试/闪烁测试）。"Flaky" 这个词本身的意思就是"一片一片剥落、不稳固"——形容它的结果像会随机剥落一样，抓不住规律。

## 3. Progressive Explanation：一个一个概念拆解

### (a) Flaky Test（不稳定测试）

定义已经在上面讲了：**同一份代码，重复运行同一个测试，结果不一致**的测试。

这跟"这个测试失败了，因为代码真的有 bug"是完全不同的两回事：

| 情况 | 代码有改动？ | 结果 |
|---|---|---|
| 正常失败 | 有（引入了 bug） | 一直失败，直到你修好 bug |
| Flaky test | 没有 | 忽而通过、忽而失败 |

**关键点**：Flaky test 是**测试本身的问题**，不是被测代码 (code under test) 的问题。

### (b) Non-determinism（非确定性）

这是 flaky test 背后的**根本原因**。"Non-deterministic" 的意思是：给定同样的输入，系统不保证每次都产生同样的输出。

写测试的时候，我们默认代码是 deterministic 的（同样输入 → 同样输出）。但如果测试依赖了一些**不受控制的外部因素**（比如系统时间、随机数、网络请求的回应速度），这个测试的结果就会变成 non-deterministic——因为这些外部因素每次执行时都可能不一样。

### (c) Race Condition（竞态条件）

这是造成 non-determinism 最常见的原因之一。**Race condition** 发生在：**两个或以上的操作同时进行，而测试的结果取决于"谁先完成"**——但"谁先完成"这件事本身是不确定的。

举个例子：

```javascript
test("用户资料应该被更新", () => {
  updateUserProfile();      // 这是一个异步操作 (async)，需要时间完成
  expect(getUserProfile().name).toBe("新名字"); // 但这里马上就去检查了
});
```

如果 `updateUserProfile()` 需要 100 毫秒才能完成，但你的 `expect` 检查在 10 毫秒的时候就执行了——这次测试可能失败（因为更新还没做完）。但下一次，如果电脑刚好比较空闲，更新在 5 毫秒内就做完了，测试就会通过。

**这就是 race condition 造成的 flaky test**：两个操作（"更新"和"检查"）在"赛跑"，谁先谁后不固定，测试结果也就跟着不固定。

### (d) Test Order Dependency（测试顺序依赖）

这是造成 flaky test 的另一个常见原因，而且它直接关联到你在 [[7.2]] 学过的 **shared state risk** 和 **test isolation**。

如果 Test B 的成功与否，**取决于 Test A 有没有在它之前跑过**——这就是 test order dependency。

例如：
- Test A 在数据库里创建了一条数据
- Test B 假设数据库里"已经有"那条数据，所以能通过
- 但如果你只单独跑 Test B（Test A 没有先跑），Test B 就会失败
- 或者如果 CI 系统这次决定**用不同的顺序**跑测试（很多框架会这样做，为了效率），Test B 也可能失败

这跟 [[7.2]] 里讲的"没做好 test isolation"是同一个病根，只是 8.1 让你学会**怎么识别**这个症状（结果不稳定），而不只是知道"应该要"做隔离。

### (e) Quarantine（隔离/隔离区）

发现一个 flaky test 之后，**在你还没时间修好它之前**，该拿它怎么办？

如果放着不管，它会持续在 CI 里随机失败，挡住无辜的 PR，让整个团队开始不信任 CI（"啊，这个测试本来就常常莫名其妙失败啦，重跑一次就好"——这种心态一旦养成，CI 就形同虚设了）。

**Quarantine（隔离）** 是一个折衷方案：把已知的 flaky test **暂时从"必须通过"的测试清单里移除**（比如打上一个特殊标记，让 CI 跳过它，或者让它失败也不会挡住 PR），但**不是删除它**——它仍然存在，仍然在跑，只是它的失败暂时不会阻挡别人。这样团队可以先继续正常开发，之后再回头认真修好这个测试的根本问题。

## 4. Scope 提醒

这一节**只教你怎么识别** flaky test（看到"忽好忽坏"的症状，能诊断出是 race condition 还是 test order dependency）。**怎么真正修好并发问题**（比如加锁、正确处理 async/await）不是这一节的范围，是以后更进阶的内容。

## 5 个你必须完全掌握的核心概念

**1. Flaky Test**
- 定义：代码没有任何改动的情况下，重复运行同一个测试，结果却时而通过、时而失败。
- 为什么重要：它会摧毁团队对 CI 系统的信任，让大家养成"失败了重跑就好"的坏习惯，间接让真正的 bug 也被忽略。
- 检验问题：一个测试昨天失败，今天通过，但代码完全没改，这说明什么？
  *答案：这很可能是一个 flaky test，问题出在测试本身（或它依赖的环境），不是被测代码有 bug。*

**2. Non-determinism**
- 定义：给定同样的输入，系统不保证每次都产生一样的输出。
- 为什么重要：它是所有 flaky test 的根本原因——好的测试应该建立在确定性 (deterministic) 的行为上。
- 检验问题：如果一个测试用到了当下的系统时间 (current timestamp) 来做判断，这个测试可能有什么风险？
  *答案：因为系统时间每次执行都不一样，这会让测试的行为变成 non-deterministic，可能造成 flaky test。*

**3. Race Condition**
- 定义：两个或以上的操作同时进行，而结果取决于"谁先完成"，但先后顺序不受保证。
- 为什么重要：这是造成 flaky test 最常见的技术原因之一，尤其在测试异步 (async) 代码时特别容易出现。
- 检验问题：为什么"更新数据后马上检查结果"的测试容易变成 flaky？
  *答案：因为"更新"可能是异步的，需要时间完成，如果检查动作在更新完成前就执行，结果就会因为执行速度的随机性而忽好忽坏。*

**4. Test Order Dependency**
- 定义：一个测试的成功与否，取决于其他测试有没有在它之前运行过。
- 为什么重要：这违反了 [[7.2]] 教的 test isolation 原则，会让测试在不同的运行顺序下产生不同结果。
- 检验问题：如果一个测试只有在"跟其他测试一起跑"时会过，单独跑就失败，这代表什么问题？
  *答案：这代表这个测试存在 test order dependency，很可能依赖了别的测试留下的共享状态 (shared state)，没有做好 test isolation。*

**5. Quarantine**
- 定义：把已知的 flaky test 暂时从"必须通过"的清单移除，但不删除它，让团队能继续正常开发，之后再回头修复。
- 为什么重要：在没有时间立刻修好 flaky test 的情况下，避免它持续挡住无辜的 PR、破坏团队对 CI 的信任。
- 检验问题：发现一个 flaky test 之后，最好的处理方式是直接删除它吗？
  *答案：不是。应该把它隔离 (quarantine)——暂时不让它阻挡 PR，但保留它，之后修好背后的根本原因（如 race condition 或 test order dependency）。*
