# 2.2 — Integration Testing

## 在大局中的位置

上一节（2.1）你测的是单个函数，完全隔离。

但真实软件不是由一堆孤立的函数组成的 — 函数调用函数，模块依赖模块，代码连接数据库。**当这些东西连在一起时，会不会出问题？** 这就是 Integration Testing 要回答的问题。

```
软件测试的层次（从小到大）
├── [2.1] Unit Testing      （测单个函数，完全隔离）
├── [2.2] Integration Testing ← 你在这里（测组件之间的连接）
├── [2.3] System / E2E Testing（测整个程序）
└── [2.4] The Test Pyramid
```

## 1. 为什么 Unit Test 不够？

假设你有两个函数，单独测都通过了：

```python
def get_user(user_id):
    # 从数据库返回用户字典
    return {"id": user_id, "name": "Alice", "tier": "gold"}

def calculate_discount(user):
    # 期望 user 有 "level" 字段
    if user["level"] == "gold":
        return 0.2
    return 0.0
```

- `get_user` 的 unit test：通过 ✅
- `calculate_discount` 的 unit test：通过 ✅

但当你把它们连在一起：

```python
user = get_user(123)
discount = calculate_discount(user)  # 💥 KeyError: 'level'
```

**崩溃了。** `get_user` 返回的是 `"tier"`，但 `calculate_discount` 期望的是 `"level"`。

两个 unit 各自正确，**但它们之间的接口对不上**。Unit test 发现不了这种问题，因为 unit test 会把两个函数隔离开来测。Integration test 就是专门找这类问题的。

---

## 2. 什么是 Integration Testing？

**Integration Testing（集成测试）** 是测试 **两个或多个组件连接在一起时，它们的交互是否正确**。

它不关心单个组件内部的逻辑（那是 unit test 的事），它只关心：

> 这两个东西连在一起，数据能正确地从一个传到另一个吗？

| | Unit Testing | Integration Testing |
|-|-------------|---------------------|
| 测试对象 | 单个函数/方法 | 多个组件的连接 |
| 是否隔离依赖 | 是（用 test double） | 否（使用真实的组件） |
| 目的 | 验证逻辑正确 | 验证组件之间能正确通信 |
| 速度 | 毫秒级 | 较慢（涉及真实依赖） |

---

## 3. Component Interaction（组件交互）

**Component（组件）** 是一个有明确职责的代码单元 — 可以是一个模块、一个类、一个服务、或一个外部系统（如数据库）。

**Component Interaction（组件交互）** 是指组件之间互相调用、传递数据的过程。

```
UserService ──调用──▶ Database
     │
     └──调用──▶ EmailService
```

Integration testing 测的就是这些箭头 — 数据在组件之间传递时有没有问题。

---

## 4. Interface Testing（接口测试）

**Interface（接口）** 是两个组件之间的"约定" — 规定了：
- 调用方传什么数据（格式、类型、字段名）
- 被调用方返回什么数据

**Interface Testing** 就是验证这个约定是否被双方都遵守。

回到前面的例子：

```
get_user() 返回:  {"id": 1, "name": "Alice", "tier": "gold"}
                                                   ↑
calculate_discount() 期望:                       "level"  ← 接口不匹配
```

这个接口断裂（interface mismatch）是 integration test 最典型要捕捉的 bug。

---

## 5. Integration Point（集成点）

**Integration Point（集成点）** 是两个组件连接的位置，也是最容易出问题的地方。

常见的集成点：

| 集成点类型 | 例子 |
|-----------|------|
| 函数调用 | 模块 A 调用模块 B 的函数 |
| 数据库 | 代码读写数据库 |
| HTTP API | 前端调用后端接口 |
| 文件系统 | 代码读取某个路径的文件 |
| 消息队列 | 服务 A 发消息，服务 B 消费 |

Integration testing 的策略就是：**找出所有集成点，逐一测试它们。**

---

## 6. Bottom-up vs Top-down

当系统有多个层次（如下图），你可以选择从哪个方向开始集成测试：

```
Layer 3: UserController（HTTP 处理层）
              ↓
Layer 2: UserService（业务逻辑层）
              ↓
Layer 1: UserRepository（数据库访问层）
```

### Bottom-up（从底层向上）

先测最底层的集成，确认它正确后，再把上一层加进来测：

```
Step 1: 测 UserRepository ↔ Database
Step 2: 测 UserService ↔ UserRepository
Step 3: 测 UserController ↔ UserService
```

**优点：** 底层稳定后再往上构建，问题好定位。  
**常见场景：** 后端开发中最常用。

### Top-down（从顶层向下）

先测最顶层，底层用假的替代品（stub）暂时代替：

```
Step 1: 测 UserController，UserService 先用 stub 代替
Step 2: 把真实 UserService 替换进来，UserRepository 用 stub
Step 3: 把真实 UserRepository 替换进来
```

**优点：** 从用户视角出发，早期就能看到高层功能是否通。  
**常见场景：** 当底层还没开发完时。

---

## 7. Module Contract（模块契约）

**Module Contract（模块契约）** 是一个模块对外的承诺：

> "如果你用这个输入调用我，我保证返回这个格式的输出。"

Integration testing 实际上是在验证这个契约是否成立。

```python
# UserRepository 的契约：
# 输入：user_id (int)
# 输出：{"id": int, "name": str, "tier": str}
# 如果找不到用户：返回 None

def get_user(user_id: int) -> dict | None:
    ...
```

当两个模块的集成出问题，根本原因往往是：**其中一方破坏了契约**（改了返回格式、改了字段名、改了类型）。

---

## 8. 完整例子

**场景：** 测试 `UserService.get_discount(user_id)` — 它调用数据库取用户，再计算折扣。

**被测代码：**

```python
class UserRepository:
    def find(self, user_id):
        # 真实实现：查数据库
        return db.query("SELECT * FROM users WHERE id = ?", user_id)

class DiscountCalculator:
    def calculate(self, user):
        if user["tier"] == "gold":
            return 0.2
        return 0.0

class UserService:
    def __init__(self, repo, calculator):
        self.repo = repo
        self.calculator = calculator

    def get_discount(self, user_id):
        user = self.repo.find(user_id)
        return self.calculator.calculate(user)
```

**Integration Test（使用真实数据库）：**

```python
def test_gold_user_gets_20_percent_discount():
    # Arrange — 准备真实的组件和数据库里的测试数据
    db.insert("users", {"id": 1, "name": "Alice", "tier": "gold"})
    repo = UserRepository()
    calculator = DiscountCalculator()
    service = UserService(repo, calculator)

    # Act — 触发跨组件的调用链
    discount = service.get_discount(1)

    # Assert
    assert discount == 0.2

    # Cleanup
    db.delete("users", id=1)
```

这个测试验证的不是 `UserRepository` 或 `DiscountCalculator` 各自的逻辑，而是 **它们连在一起时，数据能否正确流通**。

## Self-Check

**Q1.** Unit test 全部通过了，是否意味着 integration testing 可以跳过？为什么？

**A1.** 不行。Unit test 把组件隔离测试，发现不了组件之间的接口问题。两个 unit 各自正确，连接时仍然可能因为字段名不匹配、数据格式不一致、调用顺序错误等原因而出错。Integration testing 专门测这些"接缝"处的问题。

---

**Q2.** 以下哪个属于 Integration Point？

- (a) 一个函数内部的 `if/else` 逻辑
- (b) 后端 API 接收前端发来的 JSON 数据
- (c) 一个变量的命名

**A2.** **(b)**。前端和后端是两个独立的组件，它们之间传递 JSON 数据的位置就是一个 Integration Point。(a) 和 (c) 都是单个组件内部的事，属于 unit test 的范畴。

---

**Q3.** `OrderService` 调用 `InventoryService.reserve(item_id, qty)` 来预留库存。你在做 integration testing。你应该测什么？不应该测什么？

**A3.**
- **应该测：** `OrderService` 传给 `InventoryService` 的参数格式是否正确；`InventoryService` 返回的结果 `OrderService` 是否能正确处理；整个调用链执行后系统状态是否正确（库存被正确扣减）。
- **不应该测：** `InventoryService` 内部的库存计算逻辑（那是它自己的 unit test 的事）；`OrderService` 内部的订单定价逻辑（也是 unit test 的事）。

---

**Q4.** Bottom-up 和 Top-down 集成测试各适合什么场景？

**A4.**
- **Bottom-up**：适合底层（如数据库访问层）已经开发完成的情况。从最基础的组件开始测，逐层向上，每一层都有稳定的基础支撑。后端项目最常用。
- **Top-down**：适合底层还没开发完，但想先验证高层逻辑的情况。用 stub 临时替代未完成的底层，等底层完成后逐步替换。当团队并行开发不同层次时常用。
