## 1. 什么是 BDD？

**Behavior-Driven Development（BDD，行为驱动开发）** = 一种把测试写成可读行为描述的开发方法，让开发者、测试人员、业务方能用同一套语言定义软件应该怎么工作。

TDD 问的是："这个函数通过了吗？"  
BDD 问的是："这个系统的行为符合预期吗？"

BDD 的核心思想：**测试不只是技术文档，也是业务规格说明。**

---

## 2. Given-When-Then 语法

BDD 最标志性的特征是用 **Given-When-Then** 结构来描述一个场景。

```
Given  (前置条件)  — 系统处于什么状态？
When   (触发动作)  — 用户做了什么？
Then   (期望结果)  — 系统应该有什么反应？
```

这和 1.4 的 Arrange-Act-Assert 本质上是同一件事，只是语言换成了自然语言。

| AAA（开发者语言） | Given-When-Then（业务语言） |
|-----------------|--------------------------|
| Arrange（准备状态） | Given（给定前置条件） |
| Act（触发动作）    | When（当用户做了某事） |
| Assert（验证结果） | Then（那么结果应该是） |

### 例子：电商购物车

```gherkin
Feature: 购物车结算

  Scenario: 会员用户结算享受折扣
    Given 用户是会员
    And   购物车里有一件价格为 100 元的商品
    When  用户点击结算
    Then  应付金额应该是 80 元
```

这段描述：
- 产品经理能读懂
- 测试人员能验证
- 开发者能实现
- 三方都在说同一件事

### 多个 And

`And` 用来延续上一个 Given / When / Then，表示"还有"：

```gherkin
Given 用户已登录
And   用户的购物车不为空
When  用户点击结算
And   用户选择了信用卡支付
Then  订单应该创建成功
And   用户应该收到确认邮件
```

## 3. Specification by Example（用例子定义规格）

**Specification by Example** = 用具体的例子来定义软件应该怎么工作，而不是用抽象的文字描述。

**抽象规格（模糊）：**
> "会员享有折扣。"

**Specification by Example（清晰）：**
```gherkin
Scenario: 会员折扣 — 单件商品
  Given 用户是会员
  And   商品价格为 100 元
  When  用户结算
  Then  应付金额为 80 元

Scenario: 非会员 — 无折扣
  Given 用户不是会员
  And   商品价格为 100 元
  When  用户结算
  Then  应付金额为 100 元

Scenario: 会员折扣 — 商品价格为 0 元
  Given 用户是会员
  And   商品价格为 0 元
  When  用户结算
  Then  应付金额为 0 元
```

每个 Scenario 是一个具体例子，把边界情况和正常情况全部说清楚。

**好处：** 用例子代替模糊描述，消除理解上的歧义。开发者和业务方讨论"这个例子的结果应该是多少"，比讨论"折扣规则怎么定义"更有效率。

## 4. Ubiquitous Language（统一语言）

**Ubiquitous Language（无处不在的语言）** = 开发团队和业务方共同使用的一套词汇，这套词汇同时出现在业务讨论、代码、测试里。

**没有统一语言时的问题：**

```
业务方说："用户下单"
开发者代码里写：create_transaction()
测试文件里写：test_purchase_flow()
数据库表叫：tb_order
```

四个地方说的是同一件事，但用了四个不同的词——沟通时总在翻译，容易出错。

**有统一语言时：**

```
业务方说："用户下单"（place_order）
开发者代码里写：place_order()
测试文件里写：test_place_order()
数据库表叫：orders
BDD 场景里写：When 用户 place order
```

全部用同一个词，代码、测试、业务文档都能对上。

**Ubiquitous Language 的好处：**
- 减少沟通损耗（不用在不同词汇间翻译）
- 代码更贴近业务逻辑，可读性更高
- 新人加入时，读代码等于读业务规则

---

## 5. Living Documentation（活文档）

**Living Documentation** = BDD 的场景（Given-When-Then）同时作为可执行的测试和人类可读的文档，两者永远保持同步。

传统文档的问题：

```
需求文档 → 开发者实现 → 代码变了 → 文档忘记更新 → 文档和代码不一致
```

BDD 的解法：

```
BDD 场景 = 文档 + 可执行测试

代码变了 → 运行 BDD 测试 → 测试 fail → 必须更新场景
→ 文档（即场景）和代码永远保持一致
```

因为 BDD 场景本身就是测试，代码一旦与场景不符，测试就会失败，团队会立刻知道。文档不会腐烂（rot）。

**Living Documentation 的价值：** 不需要手动维护文档，测试本身就是最新的、最准确的行为说明。

---

## 6. BDD vs TDD 的关系

BDD 不是替代 TDD，它是 TDD 的**延伸和扩展**。

| | TDD | BDD |
|-|-----|-----|
| **视角** | 开发者 | 开发者 + 业务方 |
| **语言** | 代码（assert、expect） | 自然语言（Given-When-Then） |
| **关注点** | 函数行为是否正确 | 系统行为是否符合业务需求 |
| **受众** | 开发者 | 所有人（开发、测试、产品、业务） |
| **核心循环** | Red-Green-Refactor | Given-When-Then → 实现 → 通过 |

**BDD 是 TDD 的"业务语言版本"**——同样是测试先行，只是把测试语言换成了业务方也能理解的格式。

---

## 核心要点总结

```
BDD = 用自然语言描述行为的开发方法，让所有人用同一套语言

Given-When-Then：
  Given  → 前置条件（= Arrange）
  When   → 触发动作（= Act）
  Then   → 期望结果（= Assert）

Specification by Example  → 用具体例子代替模糊描述，消除歧义
Ubiquitous Language       → 业务词汇统一出现在代码、测试、讨论中
Living Documentation      → BDD 场景 = 测试 + 文档，永远同步

BDD vs TDD：BDD 是 TDD 的延伸，语言从代码变为自然语言
```

  Given 账户余额为 500 元

  Then  账户余额应该变为 300 元

**Q2：** 什么是 Specification by Example？它比抽象的文字规格好在哪里？

<details>
<summary>答案</summary>

**Specification by Example** = 用具体的场景和数值来定义软件行为，而不是用模糊的文字描述。

比抽象规格好，因为具体例子**消除了歧义**。"折扣用户享有优惠"这句话，每个人理解可能不同（优惠多少？有上限吗？对所有商品？）；但"会员购买 100 元商品，应付 80 元"只有一种理解。开发者和业务方讨论具体数字比讨论抽象规则更高效、更不容易出错。

</details>

---

**Q3：** 什么是 Ubiquitous Language？没有它会有什么问题？

<details>
<summary>答案</summary>

**Ubiquitous Language** = 开发团队和业务方共同使用的统一词汇，同时出现在业务讨论、代码、测试里。

没有它的问题：同一个概念在业务文档、代码、数据库里用不同的词（如"下单"、`create_transaction()`、`tb_order`），团队沟通时需要在这些词之间不断翻译，容易产生误解，新人也很难快速理解业务逻辑。

</details>

---

**Q4：** 为什么 BDD 场景被称为 Living Documentation？它解决了传统文档的什么问题？

<details>
<summary>答案</summary>

**Living Documentation** = BDD 场景同时是可执行的测试和人类可读的文档，两者永远保持同步。

传统文档的问题：代码变了，文档经常忘记更新，导致文档和实际行为不一致（文档腐烂）。

BDD 的解法：场景本身就是测试。代码一旦与场景不符，测试就 fail，团队立刻知道——文档自动强制保持最新状态，不需要手动维护。

</details>

---

**Q5：** BDD 和 TDD 是竞争关系还是补充关系？区别在哪里？

<details>
<summary>答案</summary>

**补充关系。** BDD 是 TDD 的延伸，不是替代。

两者都是测试先行，核心思路一致。区别在于：
- **TDD** 用代码写测试，受众是开发者，关注函数行为是否正确。
- **BDD** 用自然语言（Given-When-Then）写测试，受众扩展到所有人（开发、测试、产品、业务），关注系统行为是否符合业务需求。

可以理解为：TDD 是开发者内部的工具，BDD 是打通开发与业务沟通的桥梁。

</details>
