## 1. System Testing 是什么

**System Testing** = 把整个软件系统当作一个完整的产品来测试。

之前的测试层次：
- Unit test：一个函数通不通
- Integration test：A 模块跟 B 模块连得上不

System test 不再关心哪个模块、哪个函数——它问的是：

> "这个产品，**作为一个整体**，有没有按照需求规格（requirement specification）工作？"

换句话说，System test 验证的是：整个系统满不满足它被设计出来的目的。

---

## 2. End-to-End (E2E) Testing 是什么

**End-to-End Testing（E2E）** = 模拟真实用户的操作，从应用的入口（前端界面或 API 入口）一路跑到最底层（数据库持久化），完整走一遍。

"End-to-End" 字面意思：**从一端到另一端**——从用户按下按钮，到数据写入数据库，整条链路都要测。

### System Testing vs E2E Testing

它们非常相似，有时候可以互换使用。细微区别：

| | System Testing | E2E Testing |
|--|----------------|-------------|
| 关注点 | 整个系统符合需求规格 | 模拟真实用户完整操作流程 |
| 视角 | QA 工程师按规格验收 | 用户视角，从 UI 到数据库 |
| 范围 | 可以测非功能性（性能、安全） | 主要测功能性用户流程 |

实际工作中，大多数团队说 "E2E test" 就是在说 System test。

---

## 3. User Flow — E2E test 的核心单位

**User Flow（用户流程）** = 用户为了完成一个目标，在应用里按顺序执行的一系列操作。

E2E test 的写法，就是把一个 User Flow 用代码描述出来，然后让测试框架去模拟执行。

**例子：** 电商网站的"购买商品"User Flow：

```
1. 打开首页
2. 搜索 "laptop"
3. 点击第一个搜索结果
4. 点击 "Add to Cart"
5. 进入购物车页面
6. 点击 "Checkout"
7. 填写地址和信用卡信息
8. 点击 "Place Order"
9. 看到订单确认页面，订单号显示正确
10. 数据库里出现一条新的 order 记录
```

这整条路径，就是一个 User Flow。E2E test 会把它跑一遍，每一步都检查有没有出错。

---

## 4. Real Environment — 为什么真实环境很重要

E2E test 必须在 **Real Environment（真实环境）** 下运行。

### 什么是 Real Environment？

| 组件 | 在 E2E test 里 |
|------|---------------|
| 数据库 | 真实的数据库实例（不是内存模拟版） |
| 外部 API | 真实调用（或高度仿真的 staging 服务） |
| 浏览器 | 真实的 Chrome / Firefox（Playwright、Selenium 控制） |
| 服务器 | 完整运行的后端服务 |

### 为什么不能用假的？

Unit test 可以用 mock（假数据库、假 API）来隔离依赖。但 E2E test 的目的恰恰是测**真实组件之间的配合**。

如果用假的环境，可能出现：
- 测试通过，但真实数据库的 SQL 语法不对 → 生产环境出错
- 测试通过，但真实浏览器渲染方式不同 → 用户看到坏掉的界面

**Real environment 的代价**：慢、成本高、需要维护基础设施。这就是为什么 E2E test 数量要少（参考 2.4 Test Pyramid）。

---

## 5. System-level Black-box — 测试者的视角

E2E test 是 **Black-box testing（黑盒测试）** 的典型代表。

**Black-box** = 测试者不看、也不需要看源代码内部实现。测试者只知道：
- 输入什么
- 期望输出什么

就像用户一样——用户不知道按钮背后的代码是什么，他们只知道"我点了登录，我期望进入首页"。

"System-level black-box" 强调：
- **System-level**：测试对象是整个系统
- **Black-box**：从外部观察行为，不关心内部实现

这跟 Unit test（往往是 white-box，测试者知道函数内部逻辑）形成对比。

---

## 6. 具体例子：登录流程的 E2E test

假设我们在测一个 web 应用的登录功能。用 Playwright（一个 E2E 测试框架）写出来大概是这样：

```python
# 这是 E2E test，Playwright 控制真实浏览器执行
def test_user_can_login():
    # Arrange — 启动浏览器，准备测试用户账号
    page = browser.new_page()

    # Act — 模拟用户操作
    page.goto("https://myapp.com/login")         # 用户打开登录页
    page.fill("#email", "user@example.com")       # 填邮箱
    page.fill("#password", "securepassword123")   # 填密码
    page.click("#login-button")                   # 点登录

    # Assert — 检查系统的响应
    assert page.url == "https://myapp.com/dashboard"       # 跳转到首页
    assert page.locator("h1").text_content() == "Welcome!" # 欢迎信息显示
```

这个测试经过的路径：
```
浏览器 → 前端 HTML/JS → 后端 API → 数据库（验证密码）→ 返回 token → 前端跳转
```

整条链路都被覆盖到了。

## 7. 优点与代价

### 优点

| 优点 | 原因 |
|------|------|
| **最高信心** | 测试场景和真实用户操作完全一样 |
| **发现集成缺陷** | Unit / Integration test 遗漏的系统级问题在这里暴露 |
| **验收测试** | 可以直接用来回答 "这个功能上线了吗" |

### 代价

| 代价 | 原因 |
|------|------|
| **慢** | 要启动浏览器、连真实数据库、等网络请求 |
| **脆弱（Brittle）** | UI 改个按钮的 ID，test 就坏掉了 |
| **贵** | 需要完整的环境，维护成本高 |
| **调试难** | 失败时不知道是哪一层出问题 |

这就是为什么在 Test Pyramid（2.4）里，E2E test 放在最顶端，数量最少。

## Self-Check

**Q1.** System test 和 Unit test 最根本的区别是什么？

> **A:** Unit test 测的是单个函数/模块的逻辑，完全隔离依赖。System test 测的是整个系统作为一个整体是否满足需求，不关心内部实现细节。

---

**Q2.** 下面这条测试描述是 Unit test 还是 E2E test？
> "打开注册页面 → 填写用户名和密码 → 点击注册按钮 → 检查数据库里出现了新用户记录"

> **A:** E2E test。它模拟了真实用户的完整操作流程，从 UI 到数据库，经过了所有层。

---

**Q3.** 为什么 E2E test 必须用 Real Environment，而 Unit test 通常用 mock？

> **A:** E2E test 的目的是验证真实组件之间的配合。如果用 mock 替换数据库或 API，就无法发现真实集成点上的问题（如 SQL 语法错误、真实浏览器渲染差异等）。Unit test 用 mock 是为了隔离被测函数，快速定位问题。

---

**Q4.** User Flow 是什么？为什么它是 E2E test 的核心单位？

> **A:** User Flow 是用户为了完成一个目标而按顺序执行的一系列操作（例如：登录 → 搜索商品 → 加入购物车 → 结账）。E2E test 模拟的就是这个流程，所以一个 E2E test 对应一个 User Flow。

---

**Q5.** E2E test 是 black-box 还是 white-box？为什么？

> **A:** Black-box。E2E test 从外部（用户界面）观察系统行为，不需要知道内部代码实现。测试者只关心"输入什么操作"和"期望看到什么结果"，跟真实用户的视角完全一致。
