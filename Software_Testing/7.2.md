7.2 是关于**测试的生命周期管理**——也就是"测试跑之前要准备什么，跑完之后要收拾什么"。我用你熟悉的结构来讲。

## 1. High-Level Map：这个概念在整个体系里的位置

回想一下 [[7.1]]，我们学的是 **test framework**（比如 Jest, Pytest）怎么帮你 discover 测试、跑测试、用 assertion library 判断成功/失败、最后产生 report。而 7.2 关心的是：test runner 在执行每一个测试的过程中，给你哪些"钩子" (hooks) 可以插入你自己的准备/清理逻辑——也就是**每个测试运行的"生命周期" (lifecycle)**。测试不是凭空运行的，它需要：

1. 准备环境（比如：建一个假的数据库连接）
2. 运行测试本身
3. 清理环境（比如：关掉那个假的数据库连接）

这个"准备 → 执行 → 清理"的模式，在几乎所有测试框架（Jest, Pytest, JUnit...）里都存在，只是叫法不同。这是 Software Testing 里最基础、最必学的机制之一。

## 2. Fundamentals First：为什么需要这个东西？

想象你在写一个测试，测试"用户登录功能"。这个测试需要：

- 一个**干净的数据库**，里面有一个测试用户
- 测试完之后，把这个测试用户**删掉**，不然下一个测试跑的时候，数据库里已经有一堆垃圾数据了

如果你在**每一个测试案例里**手动写"建立数据库连接 → 插入测试用户 → 测试 → 删除用户 → 关闭连接"，你会发现：

- 代码严重重复 (repetition)
- 一旦有一个测试忘记清理，就会**污染**后面的测试

所以框架给了你专门的"钩子" (hooks)——在测试执行的**特定时间点**自动帮你调用的函数。

## 3. Progressive Explanation：一个一个概念拆解

### (a) Setup / Teardown（准备 / 收尾）

这是最基本的概念：
- **Setup** = 测试开始前要做的准备工作
- **Teardown** = 测试结束后要做的清理工作

这就像你做实验前要"消毒器材"（setup），做完实验后要"清洗器材放回原位"（teardown）。

### (b) beforeEach / afterEach（每个测试案例前后跑一次）

这是 setup/teardown 在代码里的具体实现方式（以 JavaScript 的 Jest 为例）：

```javascript
beforeEach(() => {
  // 每个 test 开始之前，都会先跑这一段
  db.connect();
});

afterEach(() => {
  // 每个 test 结束之后，都会跑这一段
  db.disconnect();
});

test("用户能成功登录", () => {
  // 这里的 db 已经连接好了，不用自己写连接代码
});

test("用户密码错误会被拒绝", () => {
  // 这个 test 开始前，db 又被重新连接了一次（全新的、干净的）
});
```

**关键点**：`beforeEach`/`afterEach` 会在**每一个** test 前后都跑一次，确保每个测试都是从"干净状态"开始的。

对应的还有 `beforeAll`/`afterAll`——这两个只会在**整个测试文件**的最开始/最后跑**一次**（用于比较"贵"的操作，例如启动一个真的服务器）。

### (c) Fixture（固定的测试数据/环境）

Fixture 就是 setup 阶段**准备好的那份数据/状态**，专门给测试使用。

例如：

```javascript
beforeEach(() => {
  testUser = { username: "test123", password: "abc" }; // 这就是 fixture
});
```

它是"固定不变的初始条件"，让每个测试都在同样的起跑线上。

### (d) Shared State Risk（共享状态的风险）

这是初学者最容易踩的坑。**Shared state** 指的是——如果两个测试案例共用同一份数据/同一个变量，而没有在每次测试前重置它，就会出现：

- Test A 跑完，修改了这个共享变量
- Test B 跑的时候，"以为"变量是初始值，结果其实已经被 Test A 改过了
- Test B 失败，但**不是因为你的代码有 bug**，而是因为测试之间"互相干扰"了

这种 bug 特别难抓，因为它**只在特定运行顺序下才会出现**。

### (e) Test Isolation（测试隔离）

这是解决 shared state risk 的**目标/原则**：每一个测试案例都应该**完全独立**运行，不管：

- 它是第一个跑还是最后一个跑
- 前面有没有其他测试跑过

`beforeEach`/`afterEach` 正是实现 test isolation 的工具——透过每次都重新 setup/teardown，确保测试互不影响。

## 4. Scope 提醒

这一节**不包括** CI (Continuous Integration) 的整合——那是以后的话题，现在只专注在"单个测试文件内部"的生命周期管理。

## 5 个你必须完全掌握的核心概念

**1. Setup / Teardown**
- 定义：测试前的准备工作，和测试后的清理工作。
- 为什么重要：没有它，测试会互相污染，导致"假阳性"或"假阴性"的结果。
- 检验问题：如果你的测试需要一个数据库连接，你应该在哪里建立它，在哪里关闭它？
  *答案：在 setup（如 beforeEach）建立，在 teardown（如 afterEach）关闭。*

**2. beforeEach / afterEach**
- 定义：分别在每一个测试案例执行前、执行后自动运行的函数。
- 为什么重要：让你不用在每个测试里重复写同样的准备/清理代码。
- 检验问题：如果你有 5 个测试案例，`beforeEach` 会被调用几次？
  *答案：5 次，每个测试前各跑一次。*

**3. Fixture**
- 定义：测试开始前准备好的、固定不变的数据或环境。
- 为什么重要：保证每个测试都在相同的、可预测的起点上运行。
- 检验问题：如果你的测试需要一个"已登录的用户"，这个用户对象应该在哪里创建？
  *答案：在 fixture / setup 阶段创建，这样每个测试都能用到一个一致的测试用户。*

**4. Shared State Risk**
- 定义：多个测试共用同一份没有被重置的数据，导致测试互相影响。
- 为什么重要：这是造成"测试顺序不同、结果就不同"这种诡异 bug 的头号原因。
- 检验问题：为什么先跑 Test A 后跑 Test B 会通过，但反过来跑就失败？
  *答案：因为 Test A 修改了某个共享变量，而这个变量没有在测试间被重置，Test B 依赖的初始状态被破坏了。*

**5. Test Isolation**
- 定义：每个测试案例都应该独立运行，不受其他测试影响或执行顺序影响。
- 为什么重要：这是"可信赖的测试"的基石——你才能相信一个测试失败，是代码真的有问题，而不是测试之间打架。
- 检验问题：怎么知道你的测试有没有做到 test isolation？
  *答案：随机打乱测试运行的顺序，如果结果还是一致的（该过的过，该败的败），就说明测试是隔离的。*
