8.3 是关于**怎么真正衡量测试的"战斗力"**——也就是"你的测试不只是有跑、有过，它到底有没有能力抓到真正的 bug？"。这一节其实是在回收一个很久以前挖的坑，我们把它补上。

## 1. High-Level Map：这个概念在整个体系里的位置

回想一下 [[4.3]]（Coverage Pitfalls），我们学到一个很重要的警告：**coverage（代码覆盖率）不能拿来当作"测试品质好"的证明**。一段代码即使被测试"跑过"（有 coverage），也不代表它真的被**验证**过——因为你的测试可能根本没写 `expect`/assertion，或者写的 assertion 太弱，抓不到任何问题。这就是 4.3 讲的 "coverage without assertions" 和 "false confidence"。

但 4.3 当时只告诉你"coverage 靠不住"，没有告诉你："那到底要怎么衡量一个测试到底强不强？" ——它把这个问题留到了 8.3，也就是我们现在要学的 **mutation testing（突变测试）**。

同时，8.3 也跟 [[8.2]] 有关——8.2 教你用"人的直觉"去闻出写得不好的测试（test smell）。而 mutation testing 是用**一个自动化的、量化的方法**去证明"这个测试到底有没有用"，是比"人的直觉"更严谨的做法。

## 2. Fundamentals First：为什么 coverage 不够，我们需要新的方法？

先复习一下问题所在。假设你有这段代码：

```javascript
function isAdult(age) {
  return age >= 18;
}
```

然后你写了这个测试：

```javascript
test("isAdult 应该能运行", () => {
  isAdult(20); // 只是调用了它，但完全没有检查结果对不对
});
```

这个测试的 **coverage 是 100%**——因为 `isAdult` 这一行代码确实被执行到了。但这个测试**完全没有验证任何东西**——就算你把 `>=` 改成 `<=`（一个严重的 bug！），这个测试依然会通过，因为它根本没检查回传值。

**核心问题**：Coverage 只能告诉你"这段代码有没有被执行到"，但它没办法告诉你"如果这段代码有 bug，你的测试抓不抓得到"。

Mutation testing 就是为了回答这第二个问题而设计的。

## 3. Progressive Explanation：一个一个概念拆解

### (a) Mutation Testing（突变测试）

**核心想法非常巧妙**：与其检查"代码有没有被跑过"，不如**故意在你的代码里制造一个 bug**，然后看看：**你的测试有没有发现这个 bug？**

如果你的测试因为这个人为制造的 bug 而**失败**了——太好了，说明你的测试真的有在认真检查东西，它是"有战斗力"的。

如果你的代码被搞出一个 bug，但你的测试**还是通过**——这就说明你的测试是"纸老虎"，它其实什么都没在检查，就像上面 `isAdult` 的例子一样。

Mutation testing 就是一个**自动化工具**，帮你系统性地、大量地做这件"故意制造 bug，看测试抓不抓得到"的实验。

### (b) Mutant（突变体）

**Mutant** 就是上面说的"故意制造出来、带有一个小 bug 的代码版本"。这个名字来自"mutation（突变）"——工具会对你的原始代码做一个微小的改动 (mutation)，产生出一个"变种"版本，这个变种就叫 mutant。

常见的突变方式，例如：

```javascript
// 原始代码
function isAdult(age) {
  return age >= 18;
}

// Mutant 1：把 >= 换成 <=
function isAdult(age) {
  return age <= 18;
}

// Mutant 2：把 18 换成 19
function isAdult(age) {
  return age >= 19;
}

// Mutant 3：把 >= 换成 >
function isAdult(age) {
  return age > 18;
}
```

Mutation testing 工具会自动产生**大量**这样的 mutant（可能一个专案会产生几百、几千个），然后针对**每一个** mutant，都重新跑一次你完整的 test suite。

### (c) Killed / Survived Mutant（被杀死 / 存活的突变体）

针对每一个 mutant，跑测试之后只有两种结果：

- **Killed（被杀死）**：你的测试因为这个 mutant 的 bug 而**失败**了。太好了——这代表你的测试真的有能力抓到这个特定的错误，它成功"杀死"了这个有问题的变种。
- **Survived（存活）**：你的测试**依然通过**，完全没发现代码已经被改坏了。这是一个警讯——代表你的测试**没有能力**侦测这一种错误，这个 mutant"活了下来"。

回到 `isAdult` 的例子：如果你写的测试是空的（像上面那个只 call 函数不检查结果的版本），那**所有的 mutant 都会存活**，因为不管代码被改成怎样，你的测试都不会失败。

反过来，如果你写的测试是：

```javascript
test("18 岁应该算成年", () => {
  expect(isAdult(18)).toBe(true);
});

test("17 岁不应该算成年", () => {
  expect(isAdult(17)).toBe(false);
});
```

这时候，上面举例的 Mutant 1、2、3 全部都会被**杀死**——因为不管哪一种改动，都会让 `isAdult(18)` 或 `isAdult(17)` 回传错的值，导致 assertion 失败。

### (d) Mutation Score（突变分数）

这是把整个实验结果**量化**成一个数字的方式：

$$
\text{Mutation Score} = \frac{\text{被杀死的 mutant 数量}}{\text{总 mutant 数量}} \times 100\%
$$

举例：如果工具总共产生了 100 个 mutant，你的测试杀死了 80 个，存活了 20 个，那 **mutation score 就是 80%**。

**这个分数的意义**：它比 coverage 更诚实地反映"你的测试有多少战斗力"。Coverage 100% 只代表"代码都被跑过"；但 mutation score 80% 代表"如果代码里真的混进了各式各样的 bug，你的测试有 80% 的机率能够抓到"。

### (e) Test Effectiveness（测试有效性）

这是这整节课想教你建立的**心态转变**：评估测试品质的重点，不该只放在"跑过了多少行代码"（coverage 关心的事），而应该放在"**这个测试到底有没有能力真正抓到 bug**"——这就是 test effectiveness（测试有效性）。

Mutation testing 提供的 mutation score，就是量化 test effectiveness 的一种具体方法。它直接呼应了 [[4.3]] 提到的教训：**quality（品质） 比 quantity（数量）更重要**——mutation score 关心的正是"品质"，而不是"你写了多少测试、覆盖了多少行代码"这种"数量"指标。

## 4. Scope 提醒

这一节**只教你概念**——mutation testing 是什么、为什么需要它、怎么解读 mutation score。**具体要用哪个工具（比如 Stryker, PIT 等）、怎么设定参数、怎么在专案里实际跑起来**，是以后更进阶、更贴近特定语言/框架的内容，8.3 目前不涉及。

## 5 个你必须完全掌握的核心概念

**1. Mutation Testing**
- 定义：故意在代码里制造微小的 bug，检验你的测试能不能发现这些 bug，藉此衡量测试的真实能力。
- 为什么重要：它解决了 coverage 的根本盲点——coverage 只看"代码有没有被跑过"，而 mutation testing 看的是"测试有没有真的在检查东西"，这是回应 [[4.3]] 提出的问题的答案。
- 检验问题：一个测试 coverage 是 100%，但完全没写 assertion，这样的测试在 mutation testing 中会有什么表现？
  *答案：几乎所有的 mutant 都会存活（survived），因为没有 assertion 就不会因为代码被改坏而失败，mutation score 会非常低甚至是 0%。*

**2. Mutant**
- 定义：工具对原始代码做一个微小改动（比如把 `>=` 改成 `<=`）后产生的、带有人为 bug 的代码版本。
- 为什么重要：它是 mutation testing 的"测试题"——用来检验你的测试到底能不能侦测出各种具体的错误类型。
- 检验问题：把 `age >= 18` 改成 `age >= 19`，这个改动后的代码版本叫什么？
  *答案：这叫做一个 mutant——原始代码经过一次人为突变后产生的变种版本。*

**3. Killed / Survived Mutant**
- 定义：Killed 代表测试因为这个 mutant 的 bug 而失败（测试成功抓到问题）；Survived 代表测试依然通过，没发现任何异常（测试没抓到问题）。
- 为什么重要：这是判断"你的测试到底有没有用"最直接的证据——每一个 survived mutant，都代表你的测试有一个具体抓不到的漏洞。
- 检验问题：如果一个 mutant 存活 (survived) 了，这代表什么？
  *答案：代表你的测试面对这种特定的代码错误时，完全没有发现异常，即测试对这一类型的 bug 没有侦测能力，是一个需要补强的警讯。*

**4. Mutation Score**
- 定义：被杀死的 mutant 数量除以总 mutant 数量，得出的一个百分比分数。
- 为什么重要：它把"测试到底强不强"这件事，从模糊的直觉变成一个可以量化、可以追踪进步的具体指标。
- 检验问题：Mutation score 80% 是什么意思？
  *答案：代表工具制造的所有人为 bug（mutant）之中，有 80% 被你的测试成功侦测（杀死）了，还有 20% 的 bug 类型是你的测试目前抓不到的。*

**5. Test Effectiveness**
- 定义：衡量测试"是否真的有能力抓到 bug"的品质概念，而不是单纯看测试的数量或代码覆盖率。
- 为什么重要：它是这整节课最核心的心态——把评估测试品质的重点，从"跑了多少行代码"（coverage）转移到"抓不抓得到问题"（真正的效力），呼应 [[4.3]] 里"quality 比 quantity 重要"的教训。
- 检验问题：为什么两个 mutation score 分别是 90% 和 40% 的专案，即使它们的 code coverage 都是 100%，测试品质也可能天差地远？
  *答案：因为 coverage 只反映代码有没有被执行到，不反映测试有没有认真检查结果；mutation score 90% 代表测试能抓到绝大部分人为制造的错误，而 40% 代表测试的检查能力薄弱，很多类型的 bug 都会被放过，即使两者的 coverage 数字看起来一样漂亮。*
