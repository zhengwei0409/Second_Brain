# 4.8 — QA & Test Engineering

---

## 1. High-Level Map(这个概念在大图里的位置)

先拉远镜头,看看我们现在走到哪了。

我们还在 **Chapter 4: Roles & Titles**。上一课 **4.7** 你认识了一整个"面向客户"的角色家族——他们的战场在公司**外面**,在客户的会议室里。这一课我们掉头回到公司**里面**,看一个完全不同的战场:**软件的质量(Quality)**。

想一个很朴素的问题:工程师写完代码,怎么知道这个软件"没有坏"?谁来负责发现"按钮点了没反应"、"付款金额算错了"这种问题?

**答案是:历史上,这曾经是一整个专门团队的工作——QA(Quality Assurance,质量保证)团队。但在过去二十年里,这个领域经历了一场剧烈的变革:角色换了名字、工作方式彻底改变,甚至很多公司把整个 QA 团队都裁掉了。**

用一句话概括这一课的核心:

> 这一课讲的不只是"几个职位名称",而是一个**正在消失中的职业的完整故事**——从"专人手动找 bug",到"写代码的测试工程师(SDET)",再到"测试责任还给开发者本人"。理解这条演化线,你就能看懂为什么今天顶级科技公司的招聘页上,QA 职位越来越少,以及这对你找工作意味着什么。

## 2. Fundamentals First(从地基开始,一个词一个词拆开讲)

### 2.1 先建立地基:为什么软件需要"测试"这件事?

软件和实体产品有个本质区别:一辆车出厂前,你能看到它、摸到它、开一圈试试。但软件是**看不见的逻辑**,一个几十万行代码的系统里,任何一行写错都可能让整个产品在某个特定情况下崩溃——而且你不去主动触发那个情况,就永远不知道它坏了。

所以软件行业发明了 **Testing(测试)** 这个活动:**在用户发现问题之前,先自己有系统地去"折腾"软件,把 bug 找出来。**

而围绕"谁来做测试、怎么做测试",就诞生了这一课的所有角色和概念。

### 2.2 拆解关键术语

**① QA Engineer(质量保证工程师)**

QA 是 **Quality Assurance** 的缩写,意思是"质量保证"。QA Engineer 就是**专门负责验证软件质量的工程师**。

传统上,他们的日常工作长这样:开发工程师(Developer)写完一个功能,把它"扔过墙"交给 QA 团队;QA Engineer 按照事先写好的 **test plan(测试计划)** 和 **test case(测试用例)**,一步一步地操作软件——点这个按钮、输入这些数据、检查结果对不对——把发现的问题记录成 **bug report(缺陷报告)**,退回给开发去修。

注意一个重点:**传统 QA Engineer 的核心技能是"细心、有条理地找问题",而不是写代码。** 很多传统 QA 岗位完全不需要编程能力。

**② Manual Testing vs Automated Testing(手动测试 vs 自动化测试)**

这是理解整个 QA 领域演化的第一条关键分界线。

- **Manual Testing(手动测试)**:由**真人**亲手操作软件来测试。人打开应用、点击、输入、用眼睛看结果对不对。优点:灵活,能发现"感觉不对劲"的问题(比如界面很丑、体验很卡)。缺点:**极慢、极贵、会漏**——每次软件改动,人都要把成百上千个 test case 重新手动跑一遍。
- **Automated Testing(自动化测试)**:**写代码来测试代码**。工程师写一段测试程序,它自动运行软件、自动输入数据、自动检查结果,几分钟内就能跑完人类要花几天才能跑完的所有测试,而且每次都一模一样、不会累、不会漏。缺点:测试代码本身也要人来写和维护,而且它只能检查"你事先想到要检查的东西"。

一个直观的比喻:manual testing 像**雇一个人每天试吃你餐厅的每一道菜**;automated testing 像**给每个灶台装上温度和咸度传感器**——传感器又快又稳定,但"这道菜摆盘不好看"这种问题,只有人能发现。

**③ SDET(Software Development Engineer in Test)**

当行业发现 automated testing 远比 manual testing 高效之后,一个新角色诞生了:**SDET**,全称 **Software Development Engineer in Test**,直译是"测试领域的软件开发工程师"。

这个头衔最早由 Microsoft 推广开来。SDET 的定位很微妙,你可以理解为:

> **SDET = 真正的软件工程师,但写的不是产品功能,而是"测试产品的软件"。**

SDET 的工作包括:写 automated test(自动化测试代码)、搭建 **test framework(测试框架)**——也就是让其他人也能方便地写测试的工具和基础设施、开发模拟大量用户操作的测试工具等等。

关键区分:**QA Engineer(传统型)靠手和眼睛测试,SDET 靠写代码测试。** SDET 的编程能力要求和普通开发工程师基本一样,面试往往也要考算法和系统设计,薪资也接近。

**④ Shift-Left Testing(测试左移)**

这个词有点抽象,我们从"left(左)"是什么意思讲起。

想象软件开发的流程画成一条从左到右的时间线:

```
需求设计 → 写代码 → 测试 → 发布上线
(最左)                        (最右)
```

传统模式下,"测试"发生在流程**偏右**的位置——代码全部写完之后,才交给 QA 团队去测。这有个致命问题:**bug 发现得越晚,修复成本越高。** 一个在"写代码"阶段就能顺手改掉的小错误,拖到"测试"阶段被发现,开发者早就忘了当时的思路,要重新理解代码、重新修改、重新提交、重新测试——成本翻了好几倍;如果拖到"上线后"才被用户发现,可能就是事故和赔偿了。

**Shift-Left Testing(测试左移)** 的意思就是:**把测试活动往时间线的左边挪——越早测越好。** 具体做法包括:开发者一边写功能代码一边写测试代码、代码还没写完就先想好"怎么验证它是对的"、甚至在设计阶段就让测试思维参与进来。

一句话记住它:**"不要等菜全做完了才试吃,每加一种调料就尝一口。"**

**⑤ Decline of Dedicated QA Teams(专职 QA 团队的衰落)**

这是这一课最有"历史感"的一个概念,也是对你的职业规划最有现实意义的一个。

大约在 2010 年代,以 Google、Facebook、Microsoft 为代表的大厂开始了一场组织变革:**大规模缩减甚至取消独立的 QA/测试部门。** 最标志性的事件是 Microsoft 在 2014 年前后重组,把大量 SDET 岗位合并进开发岗;Facebook 则从创立起就几乎没有传统 QA 团队,信奉"**工程师对自己代码的质量负全责**"。

为什么会这样?三个原因叠加:

1. **Automated testing 成熟了** —— 机器能跑的测试,不再需要人手动跑,manual QA 的核心价值被自动化取代。
2. **Shift-left 理念普及了** —— 既然测试要越早做越好,那最早能测的人就是**写代码的开发者本人**,而不是下游的另一个团队。
3. **"扔过墙"模式的弊端暴露了** —— 有专职 QA 团队时,开发者容易产生心态:"质量是 QA 的事,我写完扔过去就行。"取消这堵墙,让开发者自己写测试、自己对质量负责,反而让质量变好了。

结果就是:**"质量"没有消失,但"专门只做质量的人"大量减少了。** 测试从"一个部门的工作"变成了"每个工程师的基本功"。

## 3. Progressive Explanation(串起来,建立完整画面)

现在把这五个概念串成一个跨越三十年的故事,你会看到一条非常清晰的演化线:

> **第一幕(1990s–2000s:手动测试的黄金年代)**
> 一家软件公司的走廊两边,坐着两个泾渭分明的团队:左边是 Developer,右边是 QA。Developer 写完代码,打包扔给 QA;QA 团队几十号人,人手一份打印出来的 test case 清单,一步步点击、输入、核对,把 bug 一条条记进系统退回去。软件每半年发布一个新版本,发布前 QA 团队要没日没夜地"回归测试"(把所有旧功能重新手动测一遍)。**Manual testing 是绝对主流,QA 是一个不需要写代码、门槛相对低的入行岗位。**
>
> **第二幕(2000s–2010s:SDET 登场)**
> 软件越做越大,发布越来越频繁,手动测试跟不上了——几千个 test case,人跑一轮要几个星期。于是 Microsoft 等公司创造了 **SDET** 这个角色:招真正会写代码的工程师,专职把 manual test 转写成 automated test,并搭建 test framework。一时间 SDET 成了热门职位——既是工程师,又专注质量领域。
>
> **第三幕(2010s–现在:墙塌了)**
> 但很快,行业想通了一件更深层的事:**既然测试已经变成了"写代码",那为什么不让写功能代码的人顺手把测试代码也写了?** 这就是 **shift-left** 的逻辑——测试越早做越便宜,而最"左"的位置就是开发者自己的键盘上。于是 Google 提出"让开发者对质量负责",Microsoft 把 SDET 大量并入开发岗,Facebook 干脆从没建过传统 QA 团队。**专职 QA 部门大规模消失,测试变成了每个 Software Engineer 的日常工作的一部分。**

这条演化线可以浓缩成一张表:

| 时代 | 谁负责测试 | 怎么测 | 关键角色 |
|---|---|---|---|
| 第一幕 | 独立 QA 团队 | Manual testing 为主 | QA Engineer(不写代码) |
| 第二幕 | 独立测试团队 | Automated testing 兴起 | SDET(写测试代码) |
| 第三幕 | 开发者自己 | Automated + shift-left | Software Engineer(人人写测试) |

### 这对正在找工作的你意味着什么?

三个非常实际的启示:

1. **不要把"QA 岗位"当成进入顶级科技公司的捷径。** 曾经流传一种说法:"先做不用写代码的 QA,再转开发。"这条路在今天的顶级大厂基本走不通了——纯 manual QA 岗位在这些公司几乎绝迹。
2. **测试能力是你作为 Software Engineer 的必修课,不是选修课。** 因为质量责任已经落到每个开发者头上,面试和日常工作里,"你怎么测试你的代码"是绕不开的问题。
3. **QA/SDET 岗位并没有完全消失。** 在银行、医疗、航空、政府等对错误零容忍的行业,以及很多传统企业里,专职 QA 和 SDET 依然大量存在——只是行业的整体重心已经不可逆转地转向了"开发者自测"。

(至于自动化测试具体怎么在每次代码提交时自动运行——那属于 CI/CD 的内容,我们到 **6.6** 再展开。)

## 4. 五个你必须完全掌握的核心概念

### 1. QA Engineer(质量保证工程师)
**定义:** 专门负责验证软件质量的角色,传统上按照测试计划手动操作软件、发现并报告 bug,通常不需要写代码。
**为什么重要:** 它是整个质量领域的"原点角色",理解它的工作方式,你才能理解后来 SDET 和"开发者自测"分别革掉了它的什么命。
**自测问题:** 传统 QA Engineer 的核心产出物是什么?
**答案:** Bug report(缺陷报告)。他们按 test case 手动操作软件,发现问题后记录成报告退回给开发者修复,自己通常不修代码。

### 2. Manual vs Automated Testing(手动测试 vs 自动化测试)
**定义:** 手动测试是真人亲手操作软件找问题;自动化测试是写代码让机器自动运行并检查软件。
**为什么重要:** 这是整个 QA 行业变革的技术引擎——正因为机器测试又快又稳又便宜,才导致依赖手动测试的传统 QA 岗位大量消失。
**自测问题:** 软件每天要发布好几次新版本,每次都要把几千个测试重跑一遍,应该用哪种测试方式?为什么?
**答案:** Automated testing。人类不可能每天手动重跑几千个测试,自动化测试几分钟就能跑完,而且每次执行都完全一致、不会疲劳出错。

### 3. SDET(Software Development Engineer in Test)
**定义:** 具备完整编程能力的工程师,但专注于编写自动化测试代码和搭建测试框架,而不是开发产品功能。
**为什么重要:** 它代表了 QA 行业的中间进化形态——"测试"从体力活变成了真正的软件工程,SDET 的技能和薪资都向普通开发工程师看齐。
**自测问题:** SDET 和传统 QA Engineer 最大的区别是什么?
**答案:** SDET 用写代码的方式做测试(开发自动化测试和测试框架),编程能力要求与普通开发者相当;传统 QA Engineer 主要靠手动操作软件来测试,通常不要求编程能力。

### 4. Shift-Left Testing(测试左移)
**定义:** 把测试活动移到开发流程中更早的阶段进行,而不是等所有代码写完后才开始测试。
**为什么重要:** bug 发现得越晚,修复成本越高——越早测试越省钱省时间;这个理念直接导致测试责任从下游 QA 团队转移到了写代码的开发者本人身上。
**自测问题:** 为什么"写代码的同时就写测试"比"全部写完交给 QA 团队测"更省成本?
**答案:** 因为 bug 发现得越早修复越便宜——开发者刚写完代码时思路还在,发现问题能立刻修;拖到后期,要重新理解代码、重新走流程,成本成倍增加,拖到上线后被用户发现代价更大。

### 5. Decline of Dedicated QA Teams(专职 QA 团队的衰落)
**定义:** 2010 年代起,顶级科技公司大规模缩减或取消独立 QA 部门,把质量责任转移给开发者本人的行业趋势。
**为什么重要:** 它直接改变了就业市场——纯手动 QA 岗位在顶级大厂近乎绝迹,而"会写测试"变成了每个 Software Engineer 的必备技能,这直接影响你的求职策略。
**自测问题:** 专职 QA 团队衰落了,是不是意味着"软件测试"这件事也不重要了?
**答案:** 不是。测试比以前更重要、做得更多了——只是执行方式从"专门团队手动测"变成了"开发者自己写自动化测试"。消失的是"专职只做测试的岗位",而不是测试本身。

---

## 小结

这一课讲了一个职业的三幕演化史:**传统 QA Engineer 靠手动测试守护质量;SDET 把测试变成写代码的工程活;最终 shift-left 理念让测试责任回到开发者自己手上,专职 QA 团队在顶级科技公司大规模消失。** 对你的直接启示是:测试不再是"别人的工作",而是你未来作为 Software Engineer 的核心基本功之一。

下一课 **4.9 Adjacent Non-Coding Roles** 会带你认识工程师每天并肩工作、但自己不写代码的那些角色——Product Manager、Designer 等等,补全你对一个科技团队全貌的认知。
