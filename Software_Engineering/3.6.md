# 3.6 — Mobile Engineering

---

## 1. High-Level Map（这个主题在大局中的位置）

还记得我们在 [[2.5 — The Mobile & Cloud Era]] 讲过的故事吗？2007 年 iPhone 发布，2008 年 App Store 上线——这两件事直接"炸"出了一个全新的软件工程领域，就是我们今天要讲的 **Mobile Engineering（移动端工程）**。

用一个类比来理解 Chapter 3 的结构：如果把 Software Engineering 想象成"医学"，那 Frontend、Backend、Embedded、Mobile 这些就是"专科"——心脏科、骨科、皮肤科。你已经学过：

- [[3.1 — Frontend Engineering]] → 医生看的是"病人能看到的部分"（浏览器 UI）
- [[3.2 — Backend Engineering]] → 医生看的是"病人看不到的内脏"（server 逻辑）
- [[3.4 — Systems Programming]] / [[3.5 — Embedded Engineering]] → 医生专攻"义肢和芯片"（硬件贴近的代码）

而 **Mobile Engineering** 是一个很特别的专科——它其实是 Frontend 的"表亲"，但因为运行的载体（手机）完全不同，所以独立成了一个领域。今天我们要搞懂：它到底在造什么？为什么 iOS 和 Android 是两个世界？

---

## 2. Fundamentals First（从零开始）

### 2.1 什么是 Mobile Engineering？

**定义**：Mobile Engineering 就是专门写"跑在手机、平板这类移动设备上的 App"的工程领域。

想象一下，你打开手机，点开 Instagram、微信、TikTok——这些 App 背后的代码，就是 Mobile Engineer 写的。

但这里有个关键问题：**为什么不能直接把网页搬到手机上就好？** 毕竟 Frontend Engineer 已经很会做 UI 了。

答案是：手机是一个**完全不同的"世界"**。它有：
- 自己的操作系统（不是 Windows/Mac，是 **iOS** 或 **Android**）
- 有限的电池、内存、网络（网页在电脑上可以很"浪费"，但手机不行）
- 独特的硬件能力：GPS 定位、摄像头、指纹辨识、推送通知（push notification）
- 一个特殊的"守门人"：**App Store 分发机制**——你不能像部署网站一样随便更新，必须经过审核

正因为这些差异巨大到需要一整套不同的知识体系，所以 Mobile Engineering 才独立成了一个 field，而不是 Frontend 的一部分。

### 2.2 为什么是"两个"平台？iOS vs Android

这是理解 Mobile Engineering 的核心矛盾点。故事要从两家公司说起：

- **Apple** 在 2007 年推出 iPhone，用的操作系统是 **iOS**
- **Google** 在 2008 年推出 Android，作为"开放版本"迎战 Apple

这两家公司互相竞争，各自建立了**完全不兼容**的生态系统：
- iOS 用的编程语言是 **Swift**（以前是 Objective-C）
- Android 用的编程语言是 **Kotlin**（以前是 Java）

这就好比两个国家，讲着完全不同的语言，用着不同的"法律"（开发规范）。这也是为什么这个 topic 叫 "Mobile Engineering" 而不是单一语言的领域——因为它天生就是**分裂**的。

## 3. Progressive Explanation（逐层深入）

### 3.1 第一层理解：Native App 开发

最"原生"、最贴近手机硬件的做法，叫 **Native Development（原生开发）**：

- 想做 iOS App → 用 Swift 写，专门为 iOS 优化
- 想做 Android App → 用 Kotlin 写，专门为 Android 优化

**类比**：这就像请两位不同国籍的厨师，一位只做法国菜（iOS），一位只做日本菜（Android）。同样是"做一道菜"（做一个 App 功能），但食材、工具、手法完全不同。

**优点**：性能最好，能完全使用手机的所有功能（摄像头、传感器等），用户体验最流畅。

**缺点**：如果你想同时上架 iOS 和 Android，你得**写两套完全不同的代码**，请两组工程师，维护两个 codebase。这非常贵、非常慢。

### 3.2 第二层理解：Cross-Platform 开发的出现

工程师很快发现："这样太浪费资源了，能不能写一次代码，两边都能跑？"

于是出现了 **Cross-Platform（跨平台）开发**框架，比如：
- **React Native**（Meta 出品，用 JavaScript）
- **Flutter**（Google 出品，用 Dart）

**类比**：这就像雇一位"融合菜"大厨——他学会了一套通用手法，用同一份食谱，稍微调整一下就能同时做出接近法国菜和日本菜风味的料理。不是 100% 正宗，但效率高很多。

**优点**：一套代码，两个平台都能发布，开发速度快、成本低，特别适合 [[5.2 — Startups]] 这种资源有限的团队。

**缺点**：某些极致的性能需求或最新的系统专属功能，跨平台框架可能没办法完美支持，用户体验有时会比 native 稍逊一筹。

### 3.3 第三层理解：App Store 分发机制

这是 Mobile Engineering 独有、Frontend 完全没有的一环。

网页开发时，你改完代码，直接部署到 server，用户刷新页面就能看到最新版本（我们之后在 [[6.6 — CI/CD]] 会细讲这个概念）。

但手机 App 不是这样运作的。你必须：
1. 把 App 打包
2. 提交给 **Apple App Store** 或 **Google Play Store** 审核
3. 等待人工/自动审核通过（可能几小时到几天）
4. 用户才能在自己手机上"下载更新"

**为什么会有这道关卡？** 因为 Apple 和 Google 要确保 App 不含恶意软件、不违反隐私规范、符合他们的设计准则。这也是移动端工程师日常工作中一个很现实的"摩擦点"——你没办法像网站一样说改就改、说上线就上线。

## 4. 串联全局

回顾一下我们怎么走到这里的：

[[2.5 — The Mobile & Cloud Era]]（iPhone 发布）→ 催生了一个新市场 → 这个市场需要专门的工程师 → 这些工程师形成了今天讲的 **3.6 Mobile Engineering** 这个 field。

而这个 field 因为 Apple vs Google 的商业竞争，天生就是"分裂"的（iOS vs Android），后来才有 Cross-Platform 框架尝试把它们重新"缝合"起来。

至于移动端团队具体怎么协作、组织架构长怎样——那是后面 team structure 章节的故事了，我们现在先记住这个 field 的"身份"就好。

## 5. 五个最重要的概念（Top 5 Takeaways）

### ① Mobile Engineering
**定义**：专门开发跑在手机、平板上的 App 的工程领域。
**为什么重要**：全世界有数十亿人每天透过手机使用软件，移动端已经是许多公司最主要（甚至唯一）的用户触点。
**自问自答**：*为什么手机 App 不能直接用网页代码？*
→ 因为手机有独特的硬件资源限制（电池、内存）和特殊能力（GPS、摄像头、推送通知），网页技术无法完全驾驭这些，需要专门的知识体系。

### ② iOS
**定义**：Apple 为 iPhone/iPad 打造的操作系统，用 Swift 语言开发 App。
**为什么重要**：iOS 用户通常付费意愿更高，是许多公司（尤其是欧美市场）优先开发的平台。
**自问自答**：*开发 iOS App 主要用什么语言？*
→ Swift（早期是 Objective-C）。

### ③ Android
**定义**：Google 打造的开源移动操作系统，用 Kotlin 语言开发 App，全球市占率最高。
**为什么重要**：Android 覆盖全球更广泛（尤其亚洲、发展中国家）的用户群，是全球用户规模最大的移动平台。
**自问自答**：*开发 Android App 主要用什么语言？*
→ Kotlin（早期是 Java）。

### ④ Native vs Cross-Platform
**定义**：Native 是为单一平台（iOS 或 Android）分别写专属代码；Cross-Platform（如 React Native、Flutter）是写一套代码同时跑在两个平台上。
**为什么重要**：这是移动端团队最核心的技术决策之一——直接影响开发成本、上线速度、以及最终的用户体验品质。
**自问自答**：*一家资源有限的 startup，通常会优先选择哪种方式，为什么？*
→ 通常会选 Cross-Platform（如 React Native），因为可以用更少的工程师、更快同时覆盖 iOS 和 Android 两个市场。

### ⑤ App Store 分发机制
**定义**：手机 App 必须打包提交给 Apple App Store 或 Google Play Store 审核通过后，用户才能下载或更新。
**为什么重要**：这是 Mobile Engineering 独有的限制——工程师无法像网站那样"随时想上线就上线"，必须把审核时间纳入开发规划。
**自问自答**：*网页应用和手机 App 在"发布新版本"这件事上，最大的差异是什么？*
→ 网页可以直接部署到 server、用户刷新即可看到最新版；手机 App 必须经过 App Store/Play Store 审核流程，才能让用户获取更新。
