# 6.1 — Version Control Workflow

---

## 1. High-Level Map(这个概念在大图里的位置)

Chapter 5 我们逛完了整个"雇主动物园"——从 Big Tech 到 AI lab,八种雇主形态各不相同。但不管你进了哪一家,**入职第一天,你都会遇到同一个问题**:这家公司的代码库(codebase)可能已经被几十、几百、甚至几千个工程师改了十几年。你要在这堆代码里加一个新功能,而**隔壁工位的同事这一秒也在改同一个文件**。

这就是 **Chapter 6: How Engineering Teams Work** 要回答的核心问题:**几十上百个人,怎么同时改同一份代码,还能不天下大乱?**

这一课是整个 Chapter 6 的地基。不管你之后进 5.1 的 Big Tech、5.2 的 startup,还是 5.8 的 AI lab,团队协作的**底层机制**是同一套——这套机制叫 **version control(版本控制)**,而今天几乎全世界用的实现工具叫 **Git**。

用一句话概括这一课:

> **Version control 是一套"记录代码历史 + 协调多人同时修改"的系统。Git 负责记住每一次改动;branching(分支)让每个人可以在自己的"平行宇宙"里改代码而不干扰别人;pull request(PR)是"把我的平行宇宙合并回主线"的正式申请;code review 是合并前的质检关卡;merge conflict 是两个平行宇宙撞车时必须手动摆平的那一刻。**

(范围提醒:这一课只讲协作机制本身——Git、分支、PR、review、冲突。这些 PR 通过后**自动**发布到生产环境的机制,是 6.6 CI/CD 的话题,现在先不管。)

## 2. Fundamentals First(从地基开始,一个词一个词拆开讲)

### 2.1 先建立地基:为什么"共享一个文件"这件事本身就很危险

想象没有任何工具,你和同事共享一个 Word 文档改论文。会发生什么?

- 你改到一半,同事把整个文件覆盖保存了——你的改动**消失了**,没有任何记录。
- 你想知道"上周五这段代码长什么样"——**没有办法**,除非你手动存了副本。
- 你改坏了一个功能,想"倒退回昨天能跑的版本"——**做不到**,除非你手动备份过。

这就是**没有版本控制的世界**:没有历史、没有安全网、多人同时改等于抢答游戏,谁最后保存谁说了算,前面的改动全部作废。

早年间(还记得 [[1.2 — The Software Development Lifecycle (SDLC)]] 提到的开发流程吗)程序员真的靠"复制整个项目文件夹,命名成 `project_v2_final_FINAL.zip`"这种方式管理代码——荒谬,但曾经是现实。**Version control(版本控制)** 就是为了终结这种荒谬而生的系统性解法:

> **一个能记住"文件在每一个时间点长什么样"、并且允许多人在不互相覆盖的前提下同时工作的系统。**

先别急着往下读,停一秒记住这句话的两个关键词:**记住历史**、**允许多人同时工作而不覆盖**。这一课接下来的每一个术语——Git、分支、PR、review、冲突——全都是为了实现这两件事而存在的零件。

### 2.2 拆解关键术语

**① Git —— 记录历史的账本**

**Git** 是目前全世界几乎所有软件团队都在用的 **version control system(版本控制系统,简称 VCS)**。2005 年由 Linux 操作系统的创造者 **Linus Torvalds** 亲自写出来的——当时 Linux 社区几千个开发者在协作,现有工具都撑不住这个规模,于是他干脆自己造了一个。这个出身决定了 Git 的设计目标:**扛得住海量开发者同时协作**。

Git 怎么"记住历史"?核心动作叫 **commit(提交)**:

> 每当你完成一小块有意义的改动(比如"修好了登录按钮的 bug"),你就执行一次 commit——Git 会给这一刻的所有文件拍一张"快照"，并附上你写的一句话说明(commit message)。

把这想象成一部**游戏存档系统**:你每完成一个小任务就存一次档,每个存档点都有时间戳和一句备注。如果后面出问题,你随时可以"读档"回到任意一个历史节点——这就是"记住历史"这个目标的实现方式。整个项目从第一行代码到今天,就是一条由无数个 commit 串起来的**时间线**。

**② Branching(分支)—— 每个人的平行宇宙**

光有历史记录还不够——Git 真正的杀手锏是解决"多人同时工作"这个难题,靠的是 **branch(分支)**。

一句话定义:

> **Branch 是主时间线(通常叫 `main` 分支)的一份"平行宇宙副本"——你可以在自己的分支上随意改动、随意实验、随意 commit,完全不影响 `main` 分支,也不影响正在其他分支上工作的同事。**

打个比方:`main` 分支是这部电影的**正史剧情**——公司线上运行的、所有人都认可的"官方版本"。每当你要开发一个新功能(比如"加一个深色模式"),你就从 `main` 分支上开一个新分支,比如叫 `feature/dark-mode`——这是你自己的**平行宇宙**,你在里面怎么折腾、commit 多少次、甚至写坏代码,都不会影响正史剧情,也不会影响同事在他自己的平行宇宙 `feature/payment-fix` 里做的事。

这解决了什么问题?回到 2.1 的地基:**没有分支的世界里,多人同时改同一份代码 = 抢答游戏。有了分支,每个人都在自己的宇宙里安静工作,互不干扰。** 这是 Git 协作模型里最重要的一次思想飞跃——把"共享一份文件"变成了"共享一条历史,但允许无数条平行分支"。

**③ Pull Request(PR)—— 把平行宇宙合并回正史的正式申请**

你在 `feature/dark-mode` 分支上折腾完了,功能也测好了,现在要做什么?**不能**直接把你的分支的改动偷偷塞进 `main`——那样 `main` 又变回了"谁手快谁说了算"的抢答游戏,前面用分支解决的问题又回来了。

正确做法是发起一个 **pull request(PR,有些平台叫 merge request/MR)**:

> **PR 是一份正式申请,内容是:"我在 `feature/dark-mode` 分支上做了这些改动,请审核后把它合并(merge)进 `main`。"** PR 会展示这个分支和 `main` 之间**所有的差异(diff)**——哪些行被加了、哪些行被删了、一目了然。

把 PR 想象成公司里的**报销申请单**:你不能直接从公司账上拿钱,得填一张单子、附上收据(你的改动),交给主管审批,通过了钱才会正式入账。PR 就是"代码版的报销单"——你不能直接改动 `main`,得提交 PR、附上你的改动、经过审批(下一个概念),`main` 才会正式更新。

现实中几乎所有团队都会给 `main` 分支上锁:**技术上直接禁止任何人跳过 PR 直接推送改动到 main**。这不是信不过你,而是把"必须经过审核"这件事从"靠自觉"升级成"系统强制"——工程纪律,写进了工具的权限设置里。

**④ Code Review(代码审查)—— 合并前的质检关卡**

PR 提交之后,谁来"审批"?答案是**团队里的其他工程师**——这个审批动作就叫 **code review(代码审查)**。

> 一个或多个同事会逐行阅读你 PR 里的改动,检查:逻辑对不对?有没有明显的 bug?风格是否符合团队规范?有没有更简洁的写法?通过评论(comment)的方式在具体某一行代码上提问或提建议。

为什么需要这一关?第一性原理:**写代码的人是最容易忽略自己盲点的人**——这不是能力问题,是人类认知的通病(你盯着自己的文章看十遍也未必挑得出错别字,但换一双眼睛一眼就看到了)。Code review 引入了"第二双眼睛",在代码正式进入 `main`、影响所有人之前,提前抓住问题。

这一步还有一个容易被忽略的价值:**知识传递**。资深工程师通过 review 把经验教给新人,新人也借此了解代码库里"我没写过的那部分"长什么样。所以 code review 不只是质检,也是团队的隐性培训机制。审核通过后,PR 才会被 **merge(合并)**进 `main`——这一刻,你的平行宇宙正式并入正史。

**⑤ Merge Conflict(合并冲突)—— 两个平行宇宙撞车了**

如果你和同事**同时**改了同一个文件的**同一行**,会发生什么?这就是 **merge conflict(合并冲突)**:

> **当 Git 尝试把你的分支合并进 `main` 时,发现同一处代码在两边被改成了不同的样子,它自己无法判断该保留哪一版——于是停下来,让人类工程师手动决定。**

这是 branch 模型唯一没法自动化的角落:大部分时候大家改的是不同文件、不同区域,Git 能自动合并;但一旦两人改到了同一行,机器没有办法替你判断"业务逻辑上应该听谁的"——这本质上是个**语义判断**,只有人能做。

Git 会用特殊符号(像 `<<<<<<<`、`=======`、`>>>>>>>`)把冲突的两个版本都摆在你面前,工程师需要读懂两边的意图,手动改成正确的最终版本,再重新 commit。新手第一次遇到冲突常常会慌——记住:**这不是你的错,这是协作的正常摩擦**,团队规模越大、改动越频繁,冲突出现的概率就越高,这是"多人共享一条历史"这个模型必然付出的代价,不是 bug。

## 3. Progressive Explanation(层层深入:串起完整的一天)

### 3.1 一个完整的工作流,从头走一遍

把五个概念串成一条完整的流程,这就是 2026 年几乎所有软件团队每天在重复的动作:

1. 你从 `main` 拉出一个新分支 `feature/add-search-bar`(**branching**)。
2. 你在这个分支上写代码、多次 **commit**,记录下每一步小改动。
3. 功能写完、自己测过后,你发起一个 **pull request**,请求把这个分支合并进 `main`。
4. 团队成员对你的 PR 做 **code review**——留评论、提问、要求修改。
5. 你根据评论修改代码,推送新的 commit 到同一个分支,PR 自动更新。
6. Review 通过 → PR 被 **merge** 进 `main`。
7. 如果这期间同事也改了同一处代码 → 触发 **merge conflict** → 手动解决 → 重新走一遍 review。

这七步,从 5.1 的 Google 到 5.2 的两人 startup,骨架都是一样的——区别只在于**严格程度**(下一节细讲)。这也是为什么这一课放在 Chapter 6 开篇:它是所有团队协作的**通用语言**,之后 6.6 的 CI/CD、6.9 的大公司团队结构,全都建立在这套流程之上。

### 3.2 公司规模不同,严格程度天差地别

同一套机制,在不同规模的公司执行起来体验完全不同:

- **在 5.2 的小 startup 或 6.8 讲的十几人团队里**:可能一个人 review 你的 PR 就能合并,流程轻、速度快,几分钟内的事。
- **在 5.1 的 Big Tech 里**:一个 PR 可能需要多名 reviewer 批准、跑几十分钟的自动化检查(下节提)、遵守严格的代码规范文档——一个改动从提交到合并可能要几天。
- **在 5.7 讲的政府/国防项目里**:review 流程可能更重,层层审批,呼应 5.7 提过的"procurement 思维"——宁可慢,不能错。

同一套工具,不同的"严格旋钮"——这是你以后换工作时**第一周就要摸清楚**的团队文化信号。

### 3.3 一个重要边界:PR 触发的自动检查,属于下一课

细心的话你会发现,现实中的 PR 页面上,除了人工 review,通常还会看到一排自动跑的检查(测试是否通过、代码格式是否合规)。**这一课先不展开这部分**——PR 合并之后代码如何被自动测试、自动打包、自动发布到线上,是 **6.6 — CI/CD** 的主题,建立在今天这套版本控制流程之上。现在你只需要知道:今天学的是"人类协作的骨架",6.6 会在这个骨架上装上"自动化的肌肉"。

## 4. 五个你必须完全掌握的核心概念

### 1. Git(版本控制系统)
**定义:** 一套记录代码每一次改动历史、并支持多人同时协作而不互相覆盖的系统,2005 年由 Linux 创造者 Linus Torvalds 开发,是目前全球最主流的版本控制工具。
**为什么重要:** 它是现代软件协作的地基——没有它,多人共享一份代码等于"谁手快谁说了算"的抢答游戏;不管你进哪种公司(5.1 到 5.8 任意一种),入职第一天就要用它。
**自测问题:** Git 靠什么机制"记住历史"?这和游戏存档有什么相似之处?
**答案:** 靠 commit——每完成一小块改动就执行一次 commit,Git 会给当下所有文件拍一张快照并附上说明文字。这和游戏存档几乎一样:每个存档点都记录了那一刻的完整状态,出问题时可以随时"读档"回到任意历史节点。

### 2. Branch(分支)
**定义:** 主时间线(`main`)的一份平行宇宙副本,让每个工程师可以独立开发、随意 commit,而不影响 `main` 或其他人的分支。
**为什么重要:** 它是"多人同时改同一份代码却不天下大乱"的核心解法——把"共享一份文件"的抢答游戏,变成了"每人在自己的平行宇宙里安静工作,完成后再合并"的有序流程。
**自测问题:** 如果没有分支,两个工程师同时开发不同功能会发生什么?
**答案:** 他们要么排队等对方做完(效率极低),要么同时改同一份代码互相覆盖对方的改动(丢失工作成果)。分支让两人可以完全并行、互不干扰地工作,直到各自完成后再合并,从根本上消除了这种冲突。

### 3. Pull Request(PR)
**定义:** 一份正式申请,请求把某个分支的改动合并进 `main`,并完整展示改动的差异(diff),是"直接修改 main"和"团队审核后再合并"之间的正式关卡。
**为什么重要:** 它把"代码进入正史"这件事从"靠自觉"变成了"系统强制的审批流程"——几乎所有团队都会给 `main` 上锁,禁止跳过 PR 直接推送,这是工程纪律被写进工具权限的体现。
**自测问题:** 为什么团队要禁止直接把改动推送到 `main`,而必须走 PR?
**答案:** 因为直接推送会让 `main` 退化回"没有版本控制"时代的抢答游戏——谁手快谁的改动生效,没有审核、没有留痕。强制走 PR 保证了每一处改动在进入所有人共享的正史之前,都经过展示和审批,把质量控制从"个人自觉"升级为"系统性保障"。

### 4. Code Review(代码审查)
**定义:** PR 合并前,由团队其他工程师逐行审阅改动、提出问题和建议的过程,通过后才允许合并进 `main`。
**为什么重要:** 它引入"第二双眼睛"抓住写代码的人自己看不出的盲点和 bug,同时也是团队知识传递的隐性机制——新人通过被 review 和 review 别人来学习代码库和团队规范。
**自测问题:** 为什么"写代码的人自己检查一遍"不能替代 code review?
**答案:** 因为盲点是认知层面的问题,不是能力问题——人对自己写的东西容易产生"我知道这是对的"的心理惯性,很难用同一双眼睛发现同一双眼睛制造的错误。必须引入没有这种心理惯性的"第二个人",才能有效抓住被忽略的逻辑漏洞、边界情况或更优写法。

### 5. Merge Conflict(合并冲突)
**定义:** 当两个分支对同一处代码做出了不同修改,Git 无法自动判断该保留哪个版本时触发的状态,需要工程师手动阅读双方改动、决定最终版本。
**为什么重要:** 它是分支协作模型里唯一无法自动化的环节——机器可以自动合并互不重叠的改动,但两处改动语义冲突时,只有人类能判断"业务逻辑上该听谁的"。团队规模越大、改动越频繁,遇到冲突的概率就越高,这是协作的正常摩擦而非异常。
**自测问题:** 为什么 Git 不能像自动合并不冲突的改动一样,自动帮你解决 merge conflict?
**答案:** 因为解决冲突需要理解代码背后的业务意图——例如两人都改了同一行的价格计算逻辑,一个改成了加税前的价格、一个改成了打折后的价格,Git 只能看到"这一行文字不一样",无法判断哪种业务逻辑才是正确的最终版本。这是纯粹的语义判断,超出了工具能自动处理的范围,必须交还给人类决定。

## 小结

这一课是 **Chapter 6: How Engineering Teams Work** 的开篇,也是所有团队协作的地基。一条主线串起五个概念:**Git 用 commit 记住每一步历史;branch 让每个人在平行宇宙里独立工作;PR 是把平行宇宙合并回正史的正式申请;code review 是合并前的人工质检;merge conflict 是两个平行宇宙撞车时,唯一需要人类介入摆平的时刻。** 把这七步串起来(拉分支 → commit → 开 PR → review → 修改 → 合并 → 视情况解决冲突),就是 2026 年几乎所有软件团队每天在重复的工作流——从 5.2 的两人 startup 到 5.1 的 Google,骨架相同,严格程度不同。

细心的你可能注意到,PR 页面上除了人工 review,通常还有一排自动跑的检查——这部分我们故意留白了。下一课 **6.2 — Waterfall Methodology**,我们会先退后一步,看看"团队按什么节奏规划和交付工作"这个更大的问题——从最早的瀑布式开发讲起,再一路走到 6.6 才会回来填上"PR 合并后代码如何自动发布"这块拼图。下一课见。
