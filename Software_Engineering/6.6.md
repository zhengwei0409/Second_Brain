# 6.6 — CI/CD

---

## 1. High-Level Map(这个概念在大图里的位置)

前面几课我们讲的都是**"团队怎么组织工作"**:[[6.3 — Agile & the Agile Manifesto]] 讲价值观,[[6.4 — Scrum]] 和 [[6.5 — Kanban & Lean]] 讲两种把价值观落地的框架。这些课回答的都是"活儿怎么安排、怎么追踪"。

但有一个问题它们都没回答:**当一个功能被写完、code review 也通过了([[6.1 — Version Control & Git]] 和 [[4.2]] 讲的 PR 流程),这段代码到底是怎么从"合并进 main 分支"变成"真正跑在用户手机上"的?**

在软件行业早期,这一步是**人肉手动**的:某个工程师在自己电脑上把代码打包,通过 FTP 上传到服务器,重启服务,然后祈祷没出问题。这个过程慢、易错、而且只有那一个"懂发布流程"的人敢碰——这本身就是 6.5 讲的 Lean 视角下的巨大浪费(等待、瓶颈、缺陷)。

这一课要讲的 **CI/CD**,就是把这整条"从代码到用户"的路径**自动化**的一套实践。用一句话概括:

> **CI/CD 是一套让代码从"被合并"到"跑在用户面前"这条路径尽可能自动化的工程实践:CI(Continuous Integration,持续集成)负责在每次代码改动时,自动把大家的代码合到一起、自动构建、自动跑测试,尽早发现问题;CD(Continuous Delivery / Deployment,持续交付 / 部署)负责把通过了所有检查的代码,自动、安全、频繁地发布到生产环境。整条自动化路径,业界叫它 pipeline(流水线)。**

它和前几课的关系是:**Agile/Scrum/Kanban 让你"快速地做出小批量的改动",而 CI/CD 是让这些小批量改动能"快速、安全地真正上线"的技术底座。** 没有 CI/CD,再敏捷的团队也会卡在"做完了但发不出去"这一步——这也正是 6.5 Lean 要消灭的"未完成的工作"。

(范围提醒:这一课讲的是"代码顺利上线"的自动化路径。至于上线之后系统出了故障怎么办——谁值班、怎么处理线上事故、事后怎么复盘——那是下一课 [[6.7 — On-Call & Incident Management]] 的话题。)

## 2. Fundamentals First(从地基开始,一个词一个词拆开讲)

### 2.1 先建立地基:"集成"到底是什么?为什么它曾经是个大麻烦

要理解 Continuous **Integration**,得先理解 **integration(集成)** 这个词在软件里指什么。

回想 [[6.1 — Version Control & Git]]:一个团队里,每个人都在自己的 **branch(分支)** 上写自己的功能。你在 `feature-login` 上写登录,你同事在 `feature-payment` 上写支付。你们各写各的,一开始互不干扰。

但迟早,你俩的代码都要合回主干(main 分支),变成同一个能一起运行的程序。**"把大家各自分开写的代码,合并成一个能协同工作的整体"——这个动作就叫 integration。**

现在关键来了。想象一种糟糕的做法(这在早期非常普遍):团队约定"大家各自闷头开发,**两个月后**再统一把所有人的代码合到一起"。两个月里,你改了 500 个文件,你同事也改了 500 个文件,你们可能都动过同一个函数、同一个配置。等到第 60 天真正合并时,会发生什么?

**灾难。** 无数的 merge conflict(合并冲突),你的代码和他的代码逻辑上互相打架,合完之后整个程序跑不起来,而且**没人搞得清是谁的哪一行改动导致的**——因为要排查的改动堆积了两个月。这个恐怖的阶段,业界有个专门的黑话叫 **"integration hell(集成地狱)"**。

问题的根源是什么?**是"攒着不合"。** 改动攒得越久,合并时的风险和排查难度就**指数级**上升。看到这里你应该已经嗅到 6.5 Lean 的味道了——攒着不合的代码,就是"未完成的工作"这种库存浪费。

那解药是什么?非常简单粗暴:**别攒。**

### 2.2 拆解核心术语①:Continuous Integration(CI,持续集成)

**CI 的核心思想就一句话:与其攒两个月合并一次,不如让每个人每天都把自己的代码合并进主干——频繁地、小批量地集成,而不是攒一大批再一次性集成。**

"Continuous(持续)"这个词的意思就是:**集成这件事不再是每隔很久发生一次的大事件,而是几乎每次有人改代码、就自动发生一次的日常小事。**

但这里有个直接的反驳:"每天合这么多次,那 2.1 说的合并冲突和'合完跑不起来'的问题,不是发生得更频繁了吗?"

问得好,而这正是 CI 的精髓所在。是的,问题发生得更频繁了,**但每一次的规模都极小**。你今天合并的只是"今天写的这一点点改动",一旦合完出问题,你要排查的范围就只有"今天这几行",几分钟就能定位。这和"排查两个月堆积的改动"是天壤之别。

> **CI 的哲学:把"罕见的、灾难性的大痛苦",拆成"频繁的、微不足道的小麻烦"。** 因为软件问题有个铁律——**发现得越早,修复越便宜**(还记得 [[6.2 — Waterfall]] 讲过这条规律吗)。

但光靠"人自觉地频繁合并"还不够。因为你合进去的代码,得**真的没把东西搞坏**才行。人是会犯错、会偷懒的——你可能忘了本地跑测试,可能改 A 的时候不小心弄坏了 B。所以 CI 必须配一个自动化的"守门员"。这就引出了下面两个概念。

### 2.3 拆解核心术语②:Build(构建)与 Build Pipeline(构建流水线)

**Build(构建)指的是:把人写的源代码(source code),变成计算机能真正运行的东西的这个过程。**

为什么需要这一步?因为你写的代码(比如一份 `.java` 或 `.ts` 文件)本身,计算机是不能直接跑的。中间通常要经过一系列处理:

- **编译(compile)**:把人类可读的代码,翻译成机器能执行的格式(比如 Java 编译成 bytecode)。
- **打包(package)**:把编译后的东西,连同它依赖的各种库,打成一个可部署的整体(比如一个 `.jar` 文件、一个 Docker 镜像)。

**如果 build 这一步就失败了**(比如你代码里有个拼写错误、少了个分号),那说明代码连"能不能变成可运行程序"这关都没过,更别提上线了。所以 build 是自动化检查的第一道关卡。

而 **build pipeline(构建流水线)**,就是把 build 之后一连串的自动化步骤**串成一条线**:

```
代码推送 → ① 构建 → ② 跑测试 → ③ 代码质量检查 → ④ 部署
   (push)   (build)   (test)      (lint/scan)      (deploy)
              │          │            │               │
           失败就红   失败就红     失败就红        全绿才放行
```

之所以叫"流水线(pipeline)",是借用了工厂流水线的比喻:代码像一件产品,**必须依次通过每一道工序**,任何一道工序不合格,整条线就停下来亮红灯,产品不许流到下一站。这个比喻和 6.5 讲的丰田 Lean 是一脉相承的。

现实中执行这条流水线的工具,就是所谓的 **CI/CD 工具**——你大概率会听到这些名字:**GitHub Actions**(和 GitHub 深度集成,现在最主流之一)、**Jenkins**(老牌、开源、可高度定制)、**GitLab CI**、**CircleCI** 等。它们的作用是一样的:**盯着你的代码仓库,一旦有新代码推上来,就自动跑一遍你预先定义好的这条流水线。**

### 2.4 拆解核心术语③:Automated Testing Gate(自动化测试关卡)

这是流水线里**最重要**的一道工序,也是 CI 真正能"守住质量"的关键。

**Automated testing gate 的意思是:在代码被允许合并 / 上线之前,流水线会自动运行一整套测试(还记得 [[4.3]] 讲的 unit test、integration test 吗),只有全部通过,代码才被放行;只要有一个测试失败,这道"门(gate)"就关上,代码被挡在外面。**

"Gate(门 / 关卡)"这个词用得非常精准——它是一个**强制性的、机器执行的、不讲人情的关卡**。这里的"不讲人情"恰恰是它的价值:

- 它不会因为"这个改动看起来很小"就放水。
- 它不会因为"发布很急、老板在催"就跳过检查。
- 它不会疲劳、不会忘记、不会心软。

想想没有这道门之前的世界:代码能不能合并,靠的是 reviewer 人肉判断"这看起来没问题吧",以及作者自己保证"我在本地测过了"。这两者都极不可靠。**Automated testing gate 把"要不要相信这段代码是安全的"这个判断,从'人的主观感觉',变成了'机器客观运行几百上千个测试后的确定结果'。**

在实践中,这道门通常直接挂在 [[6.1]] 讲的 **PR(Pull Request)** 上:你提了 PR,GitHub 会自动触发流水线跑测试,PR 页面上会显示一排 ✅ 或 ❌。很多团队会设一条硬规则:**只要有 ❌,这个 PR 就无法被合并**(GitHub 里叫 "required status check")。这就把 6.5 讲的"质量内建、缺陷尽早暴露"用制度固化下来了。

### 2.5 拆解核心术语④:CD —— 一个缩写,两个含义(这里最容易混淆,请慢读)

代码通过了 build、通过了测试关卡,接下来就是把它送到用户面前——这就是 **CD**。

但 CD 这个缩写有点狡猾:**它可以指两个不同(但相关)的东西**,很多人会搞混,我们把它彻底拆开。

**含义一:Continuous Delivery(持续交付)**

> 代码通过所有自动化检查后,被自动部署到一个**类生产环境**(比如 staging 预发布环境),**随时处于"一按按钮就能上线"的就绪状态**——但真正推到生产环境的那"最后一按",仍然由人来决定。

关键词是:**自动化做到最后一步,但"是否上线"保留人工决策**。适合那些上线需要谨慎、需要挑时机(比如避开业务高峰)、或受合规约束的场景。

**含义二:Continuous Deployment(持续部署)**

> 更彻底一步:**连"最后一按"都自动化了**。代码只要通过了所有自动化检查,就**直接自动上线到生产环境**,全程无需任何人工点击。

关键词是:**全自动,人完全不介入发布动作**。

两者的唯一区别,就在那"最后一步是否需要人点一下":

```
Continuous Delivery(交付):
  build → test → 部署到 staging →〔人工点击〕→ 生产环境

Continuous Deployment(部署):
  build → test → 部署到 staging → 生产环境   (全程无人工)
```

**为什么有的团队敢做到"全自动上线"(Continuous Deployment)?** 因为他们对自己的 automated testing gate 有足够强的信心——测试覆盖得够全、够可靠,以至于"测试全绿"就等于"可以放心上线"。反过来,如果你的测试不够可靠,你就不敢全自动,只能停在 Continuous Delivery,靠人再把一道关。

**这也揭示了一个深刻的因果:一个团队能自动化到什么程度,取决于它的测试有多可信。** 测试是整个 CI/CD 大厦的地基。

### 2.6 拆解核心术语⑤:Deployment Frequency(部署频率)—— 衡量一个团队工程能力的核心指标

**Deployment frequency 指的是:一个团队多久向生产环境发布一次。**

这个指标之所以重要,是因为它几乎是**整个团队工程成熟度的一面镜子**。我们对比两种极端:

- **低频团队**:每三个月才上线一次。每次上线都是一场大工程——攒了三个月的巨量改动一次性推上去(注意,这又是 2.1 的 integration hell + 6.5 的库存浪费!),风险极高,一旦出问题极难定位是哪个改动引起的,所以每次发布都如临大敌、全员加班。
- **高频团队**:一天上线几十次。每次上线只包含**极少量**的改动。

反直觉的地方来了:**上线越频繁,反而越安全。** 为什么?

因为**每次上线包含的改动越少,出问题的概率越低、排查越快**。如果你刚上线的这一版只改了 3 行,然后线上出问题了,那罪魁祸首几乎肯定就在这 3 行里,几分钟就能定位、甚至一键回滚(rollback)。而三个月一次的发布,出了问题你得在成千上万行改动里大海捞针。

所以你会发现,**高频部署和 6.5 讲的 Lean、小批量流动,是同一套哲学的不同侧面**:小批量、快流动、快反馈。业界有一套著名的研究(**DORA metrics**,谷歌主导的《Accelerate》研究)证实了这一点——**部署越频繁的团队,反而故障率越低、故障恢复越快**。快和稳,在良好的工程实践下不是矛盾的,而是同源的。

## 3. Progressive Explanation(把零件拼成完整画面)

### 3.1 一次代码改动的完整旅程(把所有零件串起来)

现在我们像前几课那样,走一遍完整场景,看看你写的一行代码,是怎么一路自动流到用户面前的:

> **① 你写完代码**:在自己的 `feature-x` 分支上改好了,推(push)到 GitHub,并开了一个 **PR**(6.1 学的)。
> **② 流水线自动触发**:GitHub Actions 侦测到这个 PR,自动启动 pipeline——**没有任何人手动点任何东西**。
> **③ Build**:流水线先尝试构建你的代码。假设编译、打包成功,✅。
> **④ Automated testing gate**:流水线自动运行几百个测试(unit + integration)。假设全部通过,✅。同时可能还跑了代码风格检查(lint)、安全扫描,也都 ✅。
> **⑤ PR 上显示一排绿勾**:你的同事看到测试全绿,放心地 review 你的代码逻辑(机器已经帮他确认了"没搞坏东西",他只需专注看"逻辑好不好")。approve,合并进 main。
> **⑥ CD 接管**:代码一进 main,又自动触发一条部署流水线,把新版本部署到 staging 环境。
> **⑦ 上线**:如果团队做的是 Continuous **Delivery**,这里等一个人点一下"发布";如果做的是 Continuous **Deployment**,直接自动推上生产环境。
> **⑧ 你的代码,现在正跑在真实用户面前。** 从你 push 到上线,可能只花了十几分钟,全程你只做了"写代码 + 开 PR"这两件事。

对比一下本课开头讲的"人肉 FTP 上传、重启、祈祷"的时代——你就能真切体会到 CI/CD 到底解放了什么。

### 3.2 一个关键澄清:CI 和 CD 是两件事,可以分开拥有

初学者常把 CI/CD 当成一个不可分割的整体,其实不然。它们是**递进的两个成熟度台阶**:

- 一个团队可以**只有 CI**:每次 PR 自动 build + 测试,守住代码质量——但上线仍然是手动的。**这是绝大多数团队的起点,也是性价比最高的第一步。**
- 在 CI 稳固之后,才谈得上 CD:把"上线"这一步也自动化。

**顺序不能反。** 你不可能在"测试不可靠"的基础上做自动部署——那等于把没检查过的代码自动推给用户,是灾难。所以现实中永远是**先建立可靠的 CI(尤其是测试),再逐步走向 CD**。这再次印证了 2.5 的那句话:**测试的可信度,决定了你能自动化到多远。**

### 3.3 为什么 CI/CD 是前面所有课的"技术兑现"

停下来把这一课放回大图里看,你会发现 CI/CD 其实是前几课那些"理念"落地的技术保障:

- 6.3 Agile 说要"快速交付可工作的软件"——**靠什么快?靠 CI/CD 把发布从几周压缩到几分钟。**
- 6.5 Lean 说要"消除浪费、小批量流动"——**automated testing gate 消除了'等人肉检查'的等待,高频部署实现了'小批量流动'。**
- 6.5 Kanban 说要让工作"持续流动"到用户手里——**Continuous Deployment 就是这个流动的最后一公里的自动化。**

换句话说:**前面几课告诉你"应该快速、小批量、持续地交付价值",而 CI/CD 是让这件事在技术上真正可行的那套机器。** 没有它,那些理念就只能停留在纸面。

## 4. 五个你必须完全掌握的核心概念

### 1. Continuous Integration(CI,持续集成)
**定义:** 一种让团队成员频繁地、小批量地把各自代码合并进主干(而不是攒很久合一次)的实践,且每次合并都自动触发构建和测试,以尽早发现问题。
**为什么重要:** 它根治了 "integration hell"——把"攒很久后一次性合并"那种灾难性的大痛苦,拆解成"每天频繁合并"的一堆微不足道的小麻烦。因为软件问题发现得越早、修复越便宜,频繁小步集成让每次出问题时要排查的范围都极小。
**自测问题:** 既然频繁合并会让"合并冲突/合完出错"发生得更频繁,为什么 CI 反而降低了整体风险?
**答案:** 因为衡量风险的不是"出问题的次数",而是"每次问题的规模和排查成本"。每天合并时,你只引入了极少量改动,一旦出错,排查范围就只有这一小批改动,几分钟可定位;而攒两个月一次性合并,虽然只出一次问题,但要在两个月堆积的改动里大海捞针,风险和成本都指数级更高。

### 2. Build Pipeline(构建流水线)
**定义:** 把代码从"推送"到"上线"之间一连串自动化步骤(构建 → 测试 → 质量扫描 → 部署)串成的一条自动化通道,代码必须依次通过每一道工序,任何一道失败,整条线就停下亮红灯。
**为什么重要:** 它把原本靠人手动执行、极易出错和遗漏的发布过程,变成一条标准化、可重复、机器执行的通道。每次改动都走完全相同的检查流程,消除了"这次发布忘了跑某个检查"这类人为疏漏,是整个 CI/CD 得以自动化的载体。
**自测问题:** 为什么叫"流水线(pipeline)"?这个比喻抓住了它最本质的什么特征?
**答案:** 借用工厂流水线——代码像一件产品,必须依次、有序地通过每一道工序(工序间有先后依赖:必须先构建成功才能跑测试),且任何一道工序不合格,产品就被拦下、不许流到下一站。这个比喻抓住了它"顺序性"和"逐关卡强制放行"这两个本质特征。

### 3. Automated Testing Gate(自动化测试关卡)
**定义:** 在代码被允许合并或上线前,流水线自动运行整套测试,全部通过才放行、有一个失败就把代码挡在门外的强制性机器关卡,通常直接挂在 PR 上作为合并的必要条件。
**为什么重要:** 它把"这段代码安不安全"的判断,从 reviewer 和作者的主观感觉("看起来没问题吧""我本地测过了"),换成了机器客观运行成百上千个测试后的确定结果。而且它不讲人情——不会因为改动小、发布急就放水,把质量保证从"靠自觉"变成了"靠制度"。
**自测问题:** 有人说"发布很急,这次先跳过测试直接上线,回头再补"。从 automated testing gate 的设计意图看,这句话错在哪?
**答案:** 错在它想绕过这道门"不讲人情"的核心价值。这道门存在的全部意义,就是不因为"急"或"改动看起来小"而放水——因为恰恰是"急着上线、跳过检查"的时候最容易出线上事故。跳过测试直接上线,等于把一段没被验证过的代码直接推给用户,一旦出问题,损失远大于多等几分钟跑完测试。

### 4. Continuous Delivery vs Continuous Deployment(持续交付 vs 持续部署)
**定义:** 两者都把"代码通过检查后送向生产"这条路自动化,唯一区别在最后一步——Continuous Delivery 自动把代码准备到"一按按钮即可上线"的就绪状态,但真正上线由人决定;Continuous Deployment 连这最后一按也自动化,通过检查即自动上线,全程无人介入。
**为什么重要:** 它揭示了一个深刻的因果:一个团队敢自动化到哪一步,取决于它的测试有多可信。测试足够可靠,团队才敢做全自动的 Continuous Deployment;测试不够可靠,就只能停在 Continuous Delivery,靠人再把最后一道关。所以测试是整个 CI/CD 的地基。
**自测问题:** 两个团队都实现了"通过测试就自动部署到 staging",但一个是 Continuous Delivery、一个是 Continuous Deployment。它们的差别具体在哪一步?这个差别背后反映了两队什么不同?
**答案:** 差别在"从 staging 推到生产环境"这一步:Delivery 需要一个人点击确认,Deployment 则自动完成。背后反映的是两队对自己 automated testing gate 的信心不同——做 Deployment 的团队相信"测试全绿=可安全上线",做 Delivery 的团队还需要人再判断一次(可能因为测试覆盖不够全,或上线需挑时机/受合规约束)。

### 5. Deployment Frequency(部署频率)
**定义:** 一个团队多久向生产环境发布一次;它是衡量团队工程成熟度的核心指标之一,高频团队一天可上线几十次,低频团队可能几个月才一次。
**为什么重要:** 它反直觉地揭示了"快"和"稳"的统一——上线越频繁,每次包含的改动越少,出问题概率越低、定位和回滚越快,反而越安全。它是 6.5 Lean"小批量、快流动"哲学在发布环节的直接体现,权威研究(DORA)也证实高频部署的团队故障率更低、恢复更快。
**自测问题:** "我们三个月才发布一次,是为了求稳"——这个说法为什么在良好工程实践下站不住脚?
**答案:** 因为三个月一次的发布,意味着每次要一口气推上去三个月堆积的巨量改动,这既是 integration hell、又是巨大的库存浪费;一旦出问题,得在海量改动里排查,风险和恢复成本都极高。真正的"稳"来自高频小批量发布——每次改动少、出错概率低、能快速定位和回滚。低频发布不是求稳,反而是把风险攒到一起放大。

---

## 小结

这一课我们打通了"代码怎么从 PR 变成用户手里的软件"这条自动化路径。**CI(持续集成)解决"合代码"的问题——用频繁小批量的自动集成 + build + 测试,根治 integration hell,把大痛苦拆成小麻烦;CD 解决"发代码"的问题——Continuous Delivery 自动化到"一按即上线"、由人决定发布,Continuous Deployment 连这一按也自动化。串起整条路的是 build pipeline(流水线),守住质量的是 automated testing gate(不讲人情的机器关卡),而 deployment frequency(部署频率)则反直觉地揭示了:上线越频繁,反而越安全。**

记住贯穿这一课的两条主线:一是 **"发现越早,修复越便宜"**,CI 的一切都由它推导而来;二是 **"测试的可信度,决定了你能自动化到多远"**,它解释了为什么 CD 有交付和部署两个台阶。CI/CD 本质上是前几课 Agile / Lean 理念的**技术兑现**——没有它,"快速小批量持续交付价值"就只是口号。
