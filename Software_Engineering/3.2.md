# 3.2 — Backend Engineering

## 1. High-Level Map（这个主题在大局里的位置）

还记得 [[3.1 — Frontend Engineering]] 那个餐厅比喻吗？

- **Frontend** = 前场（dining area）—— 客人看到的桌椅、菜单、服务员
- **Backend Engineering** = **厨房** —— 客人看不到，但食物真正被煮出来、食材真正被储存的地方

你点了一份"炒饭"（这是你在 App 上按下的一个按钮），服务员（frontend）把订单传进厨房，厨房里的厨师（backend）真正去处理：有没有食材？怎么煮？煮好了要怎么装盘送出去？这整套"看不见的处理过程"，就是 Backend Engineering 的世界。

在 **Chapter 3: Fields of Software Engineering** 里，3.1 讲完了"前场"，3.2 现在讲"厨房"。这一节的任务很单纯：**只定义 backend 是什么、它负责做什么**。至于"前场厨房要怎么配合"（那是 3.3 Full-Stack），或是"厨房所在的整栋大楼要怎么维护"（那是之后 infrastructure/operations 的话题），现在都还不需要碰。

## 2. Fundamentals First（从零开始，解释每个字的"为什么"）

### 什么是 "Backend"？

**Backend（后端）= 用户看不到，但在幕后处理数据、逻辑、储存的那一层软件。**

回到 3.1 提过的例子：你在 Instagram 按下爱心（like）按钮。

- 爱心变红、有动画 —— 这是 **frontend**（3.1 教过了）
- 但这个"赞"要被**记住**——下次你或别人再打开这张照片，都要看到这个赞还在——这个"记住"的动作，发生在你的手机之外，在一台**远方的电脑（服务器）**上。这就是 **backend** 的工作。

### 为什么会需要"专门"做 backend 的人？

从第一性原理想：Frontend 的代码是下载到**你自己的设备**上执行的（3.1 讲过）。但这就产生一个大问题——

> 如果所有东西都只存在你自己的手机上，那你换一台手机、或者你朋友想看你发的照片，怎么办？

答案是：必须有一台**大家都能连接到的中央电脑**，来存放数据、处理逻辑、决定"谁能看什么、谁能改什么"。这台电脑，或者说这一整套运行在远方的程序，就叫 **server（服务器）**。而专门写这套"服务器上的程序"的工程师，就是 **backend engineer**。

这也是为什么 backend 又叫 **"server-side"**（服务器端）—— 对应 frontend 的 **"client-side"**（客户端，也就是你自己的浏览器/手机）。

### 核心概念一：Server-side Logic（服务端逻辑）

**Server-side logic = 运行在服务器上、用户完全看不到源代码的那些程序逻辑。**

关键差异：Frontend 的代码你其实"看得到"——在浏览器按 F12 打开开发者工具，任何人都能看到网页的 HTML/CSS/JavaScript 原始码。但 backend 的代码**永远藏在服务器里**，用户只能看到"结果"，看不到"怎么算出来的"。

> 类比：你在餐厅只看到炒饭端上桌（结果），但看不到厨师加了多少盐、用了什么秘方（server-side logic）。

这个"藏起来"的特性很关键，也是为什么密码验证、付款计算这些敏感逻辑一定要放在 backend，绝对不能放在用户看得到的 frontend。

### 核心概念二：APIs（Application Programming Interfaces）

这是 backend 世界最重要的概念之一。先拆解这个词：

- **Application** = 应用程序
- **Programming Interface** = 给程序用的"接口"（不是给人用的按钮，是给另一段代码用的"沟通窗口"）

**API = backend 定义的一套规则，让 frontend（或其他程序）知道该怎么向它"要数据"或"送数据"。**

> 类比：API 就像餐厅的**菜单**。菜单上写着"炒饭 15元"，你（frontend）不需要知道厨房怎么运作，只需要照着菜单点餐（呼叫 API），厨房（backend）就会按照固定流程处理，然后把结果端出来。

举例：Frontend 想显示你的 Instagram 个人资料，它会呼叫类似 `GET /user/profile` 这样的 API，backend 收到这个请求后，去数据库捞资料，再把结果传回给 frontend 显示出来。**API 就是 frontend 和 backend 之间约定好的沟通桥梁。**

### 核心概念三：Databases（数据库）

**Database = 长期、有组织地储存数据的地方。**

为什么不能直接把数据存在程序的"记忆"里就好？因为程序运行的电脑一旦关机、重启，暂存的记忆（memory）就会清空消失。但你的 Instagram 帐号、照片、朋友列表，不可能每次服务器重启就全部消失——所以需要一个**专门为"长期储存 + 快速查找"设计的系统**，这就是数据库。

Backend engineer 的核心工作之一，就是设计"数据要怎么存"（比如：一个用户对应多张照片，要怎么组织这个关系），以及写程序去跟数据库"要数据"或"存数据"。

### 核心概念四：Business Logic Layer（业务逻辑层）

**Business logic = 反映"这个产品的规则"的程序代码。**

举个具体例子：
- "库存少于 0 不能下单" —— 这是一条业务规则
- "用户没有付费订阅，不能看付费内容" —— 这也是一条业务规则
- "转帐金额不能超过帐户余额" —— 这也是

这些规则不是"数据库怎么存"的问题，也不是"UI 长什么样"的问题，而是"**这个产品应该怎么运作**"的问题。这一层代码通常写在 backend，因为如前面所说，backend 代码是"藏起来的"——如果这些规则写在 frontend，任何懂技术的用户都能绕过限制（比如自己改网页代码，让"库存不足"的检查失效）。

## 3. Progressive Explanation 小结：Backend Engineer 到底"拥有"什么？

回到这节的 Learning Objective：**Define the backend field and what it builds.**

一句话总结：**Backend engineer 负责一切"用户看不到，但支撑产品运作"的东西**——处理来自 frontend 的请求（透过 API）、执行业务规则（business logic）、以及管理数据的长期储存与读取（databases）。

他们**不负责**（这些留到 3.1 已经讲过、或未来模块）：
- 画面长什么样、颜色排版怎么设计（那是 frontend / 3.1）
- 服务器本身要怎么维护、扩容、部署（那是 infrastructure/operations，未来模块）

## 4. 五个你必须完全掌握的核心概念

**1. Backend（后端）**
- **定义**：软件中用户看不到，但在服务器上处理数据与逻辑的那一层。
- **为什么重要**：没有 backend，数据就无法在不同装置、不同用户之间共享或长期保存——你的账号、聊天记录、订单，全部都靠 backend 才能"存在"。
- **自测题**：你在 Instagram 按赞后关掉 App，隔天重新打开，赞还在——这个"记住"的动作是 frontend 还是 backend 做的？
  → **答案：Backend**。因为这个数据被存到远方的服务器/数据库里，不是存在你手机的画面显示逻辑（frontend）中。

**2. Server-side Logic（服务端逻辑）**
- **定义**：运行在服务器上、用户无法直接看到源代码的程序逻辑。
- **为什么重要**：敏感操作（如密码验证、付款金额计算）必须藏在用户看不到的地方，否则容易被篡改或盗用。
- **自测题**：为什么密码验证的逻辑不能写在 frontend，一定要写在 backend？
  → **答案：因为 frontend 的代码任何人都能在浏览器打开查看甚至修改，如果验证逻辑在那里，就能被绕过；backend 代码藏在服务器里，用户无法直接看到或篡改。**

**3. APIs（应用程序接口）**
- **定义**：backend 定义的一套规则，让 frontend 或其他程序知道如何请求数据或发送数据。
- **为什么重要**：API 是 frontend 和 backend 之间唯一的沟通桥梁——没有它，两边就是两个互不相通的孤岛。
- **自测题**：用餐厅比喻，API 相当于餐厅里的什么东西？
  → **答案：菜单**。你（frontend）不需要知道厨房内部怎么运作，只需要照菜单（API）点餐，厨房（backend）就会按固定流程处理并回传结果。

**4. Databases（数据库）**
- **定义**：长期、有组织地储存数据的系统。
- **为什么重要**：程序运行时的暂存记忆一旦关机就会消失，数据库能确保用户的数据（帐号、照片、订单）不会因为服务器重启而丢失。
- **自测题**：为什么不能把所有用户数据都只存在程序运行时的"记忆"里，而需要一个数据库？
  → **答案：因为程序运行的电脑重启或关机时，暂存的记忆会被清空；数据库是专门为长期储存和快速查找设计的系统，能确保数据持久保存。**

**5. Business Logic Layer（业务逻辑层）**
- **定义**：反映产品运作规则（例如"库存不足不能下单"）的程序代码，通常存在于 backend。
- **为什么重要**：这些规则决定了产品"应该怎么运作"，放在用户看不到的 backend，才能防止规则被绕过或篡改。
- **自测题**："转帐金额不能超过帐户余额"这条规则，应该写在 frontend 还是 backend？为什么？
  → **答案：Backend**。因为如果这条规则只写在 frontend，用户可以透过修改网页代码绕过限制，造成超额转帐；写在用户看不到、无法篡改的 backend 才能确保规则被真正强制执行。**
