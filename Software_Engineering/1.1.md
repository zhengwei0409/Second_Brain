# 1.1 — Software Engineering vs Programming

## 1. Big Picture：这个知识点在哪里？

想象你在爬一座山，目标是「拿到 top tech company 的 SWE offer」。

整个 `curriculum.md` 就是这座山的地图：
- **第 1 章 Foundations**（我们现在在这里）—— 山脚，回答"这个行业到底是什么"
- 第 2 章 —— 这个行业的历史
- 第 3-9 章 —— 领域、职位、公司类型、团队怎么运作...

而 **1.1 是整座山的第一块地基**。为什么？因为如果你连 "software engineering" 和 "programming" 的区别都说不清楚，你在面试时会很难解释「为什么我要读 CS，而不是只上 YouTube 学 syntax 就好」。

这不是个 trivial 的语义游戏。这是你从「写代码的人」变成「工程师」的第一个 mindset shift。

---

## 2. Fundamentals First：从零开始

### 2.1 什么是 Programming（编程）？

**Programming = 用编程语言，把逻辑写成电脑能执行的指令。**

举个例子：你写一个 Python script，把一个 list 排序、打印结果。这就是 programming。

关键特征：
- 通常是**你一个人**在写
- 通常是**解决一个具体、当下的问题**
- 写完、能跑，任务就算完成了
- 没有人会在 3 年后还要维护这段代码

Programming 关心的问题只有一个：**"这段代码现在能不能跑？"**

### 2.2 什么是 Software Engineering（软件工程）？

**Software Engineering = 用工程化的方法，在一个团队、一个时间跨度、一个真实世界的约束下，构建和维护软件系统。**

这句话信息量很大，我们拆开看。

#### "工程化的方法" 是什么意思？

想象你要盖一座桥。你不会随便找块钢板焊一焊就完事——你要算负重、要有蓝图、要有人检查、要考虑地震、要考虑 20 年后钢材会不会生锈。

这就是"工程"（Engineering）这个词的核心含义：**用系统化、可重复、可验证的方法，去建造一个东西，并且要让这个东西在长时间、在别人手里、在各种意外情况下，依然能正常运作。**

Software Engineering 就是把这套思维方式，用在"造软件"这件事上。

#### 那具体差在哪几个维度？我们逐个拆解。

## 3. Progressive Explanation：一层层建立复杂度

我们从 Programming 出发，一次加一层，看看它怎么"长成" Software Engineering。

### 层 1：加上"规模"（Scale）

一个人写的 100 行 script，你脑子能记住全部逻辑。

但如果是 100万 行代码，被 500 个工程师同时修改呢？你不可能记住全部。这时候你需要：
- 一致的代码风格（不然读别人代码像读天书）
- 清楚的模块划分（不然改一个地方，全公司的功能都炸了）

**Programming 关心"这行代码对不对"。Software Engineering 关心"这一百万行代码，在五百人手里，还能不能对"。**

### 层 2：加上"时间"（Maintainability，可维护性）

你写的代码，今天能跑，很好。

但半年后，需求变了，要加一个新功能。这段代码你还看得懂吗？别人接手，看得懂吗？

**Maintainability（可维护性）** 指的就是：这段代码是否容易被理解、被修改、被扩展，而不会一改就坏。

这就是为什么工程师要写：
- 有意义的变量名（不是 `x1`, `x2`, `temp`）
- 测试（test），确保你改了 A 功能，没有意外弄坏 B 功能
- 文档，让下一个接手的人不用从头猜

Programming 只需要代码"现在"能跑。Software Engineering 需要代码在"未来"也能被安全地修改。

### 层 3：加上"团队"（Team-based development）

一个人写代码，你想怎么写就怎么写。

但团队协作时，出现了新问题：
- 我写的代码，别人要能看懂 → 需要 **code review**（别人检查你的代码）
- 大家同时改同一份代码，怎么不冲突？→ 需要版本控制工具（比如 Git，这是后面 6.1 会讲的主题）
- 谁负责哪一块？怎么分工？→ 需要流程和角色

**Programming 是个人手艺（craft）。Software Engineering 是团队协作的工程学科（discipline）。**

### 层 4：加上"真实世界的约束"

真实产品要面对：
- 用户量暴增，系统会不会崩？（这叫 scalability，可扩展性）
- 出 bug 了，用户投诉，谁负责修？多快修？
- 预算有限、时间有限，怎么权衡"做得好"和"做得快"？

这些都不是"写对一行代码"能解决的问题，而是需要规划、权衡、管理的**工程决策**。

## 4. 一张表总结对比

| 维度 | Programming（编程） | Software Engineering（软件工程） |
|---|---|---|
| 核心问题 | 这段代码能不能跑？ | 这个系统能不能被团队长期维护、扩展、信任？ |
| 时间尺度 | 现在 | 现在 + 未来几年 |
| 规模 | 通常是个人、小型任务 | 通常是团队、大型系统 |
| 关注点 | 逻辑正确性 | 正确性 + 可维护性 + 协作流程 + 风险管理 |
| 类比 | 拧螺丝 | 盖一栋大楼，还要保证 20 年后不会塌 |

## 5. 为什么这个区别在现实世界很重要？

在面试和职场上，公司招的不是"会写代码的人"，而是"能在团队里，长期、可靠地交付软件的工程师"。

这也是为什么：
- 面试会考 **system design**（不只是 "写出正确答案"，而是"这个系统撑不撑得住"）
- 公司在意你写不写测试、代码是否易读，而不只是"跑得动就好"
- Senior engineer 和 Junior engineer 的差距，往往不在"会不会写这一行代码"，而在"你有没有考虑到未来维护、团队协作、系统边界这些工程层面的东西"

理解这一点，会从第一天就重塑你写代码的方式——不是"这样写能跑就行"，而是"这样写，半年后我或别人还能安全地改它吗？"

## 6. 五个你必须掌握的核心概念

### ① Software Engineering Definition（软件工程的定义）
- **一句话定义**：Software engineering is building and maintaining software as a team, over time, in a way that stays reliable and manageable as it grows.
- **为什么重要**：这决定了你在工作中被评价的标准，不只是"代码能跑"，而是"代码能不能被别人安全地接手"。
- **自测问题**：如果你的代码能跑，但没人（包括你自己半年后）看得懂，这算合格的 software engineering 吗？
  **答案**：不算。因为它缺少了 maintainability（可维护性）这个核心要素。

### ② Programming vs Engineering（编程 vs 工程）
- **一句话定义**：Programming is writing code that works now; engineering is building systems that keep working as they scale, change, and are handed off to others.
- **为什么重要**：帮你理解为什么学校教你的"写出正确答案"，和公司要求的"写出好代码"，是两件不完全一样的事。
- **自测问题**：两个人都写出了能通过测试的代码，为什么其中一个会被说"这不是 production-ready"？
  **答案**：因为 production-ready 的代码还要考虑可读性、可维护性、边界情况（edge cases）、以及未来被别人修改的容易程度，而不仅仅是通过眼前的测试。

### ③ Engineering at Scale（规模化工程）
- **一句话定义**：Problems that don't exist with 100 lines of code start to matter a lot when you have millions of lines and hundreds of engineers.
- **为什么重要**：解释了为什么大公司需要那么多"看起来多余"的流程（code review、规范、架构设计），因为规模一大，人脑记不住全部细节，必须靠系统化的方法弥补。
- **自测问题**：为什么一个人的 side project 可以随便写，但大公司的代码库不行？
  **答案**：因为规模变大后，代码会被很多人同时修改和阅读，缺乏结构和规范会导致混乱、冲突和难以排查的 bug。

### ④ Maintainability（可维护性）
- **一句话定义**：Maintainability is how easily code can be understood, fixed, and extended by someone (including future-you) after it's written.
- **为什么重要**：大部分软件的生命周期里，"维护"花的时间远比"最初写出来"花的时间长，所以这是评价代码质量最重要的标准之一。
- **自测问题**：如果你半年后回来看自己写的代码，完全看不懂，这说明了什么？
  **答案**：说明这段代码的可维护性不够好——可能缺少清晰命名、注释、结构，或测试，导致连作者自己都难以安全地修改它。

### ⑤ Team-based Development（团队化开发）
- **一句话定义**：Real-world software is built by many people working on the same codebase at the same time, which requires coordination tools and processes.
- **为什么重要**：这是"个人写代码"和"职业软件工程师"之间最大的分水岭——你需要学会和别人的代码共存、协作、不冲突。
- **自测问题**：如果 500 个工程师同时在改同一个项目，为什么不能大家各写各的，最后拼在一起？
  **答案**：因为会产生大量冲突（conflict）、风格不一致、互相不理解对方代码的问题，所以需要 code review、版本控制（如 Git）、规范等机制来协调。
