# 3.9 — Security Engineering

---

## 1. High-Level Map（这个主题在大局中的位置）

我们已经走过了 Chapter 3 里一连串"造东西"的领域：[[3.1 — Frontend Engineering]] 造用户看得到的界面、[[3.2 — Backend Engineering]] 造服务器逻辑、[[3.7 — Data Engineering]] 整理数据、[[3.8 — Machine Learning Engineering]] 训练智能模型。

这些领域有一个共同的默认假设：**"这个功能会被正常使用。"** 用户会乖乖填正确的表单、乖乖用你设计好的方式点击按钮。

但今天要介绍的 **Security Engineering（安全工程）**，问的是完全相反的一个问题：

> **如果有人不按牌理出牌，故意想办法破坏、偷取、滥用这个系统，会发生什么事？我们要怎么提前挡住他？**

这就是 Security Engineering 存在的理由。它不是 Chapter 3 里"再一个造东西的领域"，而是一个**横跨所有其他领域的守门员**——不管是 Frontend、Backend、Mobile 还是 ML 系统，只要是软件，就有可能被攻击，所以都需要 Security Engineering 的介入。这也是为什么它被放在 Chapter 3 接近尾声的位置：先认识完"各种系统长什么样"，才能理解"这些系统会怎么被攻破"。

## 2. Fundamentals First（从零开始）

### 2.1 先建立一个画面：为什么软件需要"防御"？

想象你的 [[3.2 — Backend Engineering]] 写了一个登入功能，逻辑很单纯：

```
用户输入帐号密码 → 后端检查是否正确 → 正确就放行
```

这个逻辑对"正常用户"来说完全没问题。但世界上有一群人，专门会去想："如果我不老老实实输入密码，而是输入一段很奇怪的文字，会不会骗过这个检查？如果我一秒钟内狂试一百万个密码，系统会不会没有阻挡我？"

**这就是攻击者（Attacker）的思维方式**——他们不是在"使用"你的系统，而是在"测试你的系统有没有漏洞"。而绝大多数工程师在写功能的时候，脑子里想的都是"怎么让这个功能正常运作"，很少有人会同时想"这个功能会不会被恶意滥用"。这个思维上的落差，就是安全问题的根源。

### 2.2 什么是 Security Engineering？

**定义**：Security Engineering 是专门负责"找出软件系统里可能被攻击的弱点，并且提前设计、修补，让系统能抵御恶意攻击"的工程领域。

用一个类比来理解：如果把一间公司的所有系统比喻成一栋大楼，那么 [[3.2 — Backend Engineering]]、[[3.1 — Frontend Engineering]] 这些领域负责的是"把大楼盖起来、把电梯、水电、房间都装修好，让住户能正常生活"。而 Security Engineer 的工作，就是从"小偷、纵火犯"的角度重新检查这栋大楼——门锁够不够坚固？消防通道会不会被人恶意打开当成入侵口？监控系统够不够全面？他们不盖新房间，而是确保已经盖好的房间不会被闯入。

## 3. Progressive Explanation（逐层深入）

### 3.1 第一层理解：Vulnerability（漏洞）

**定义**：系统里一个可以被攻击者利用、用来做出设计者原本没预期的行为（例如偷数据、破坏系统、冒充别人）的弱点。

举一个经典例子——**SQL Injection（SQL 注入）**。假设 Backend 的登入逻辑，直接把用户输入的帐号，原封不动地拼进一段查询数据库的指令里。正常用户输入 `zhengwei`，系统就乖乖去数据库查有没有这个帐号。但如果攻击者不输入正常帐号，而是输入一段精心设计过的文字，这段文字会"骗过"数据库，让它把查询逻辑整个改掉——例如直接绕过密码检查，或是把整张用户表格的数据全部吐出来。

**类比**：Vulnerability 就像一栋大楼里，某扇窗户的锁其实是坏的，但从外表看起来完全正常，连屋主自己都不知道。平常没有人会去推那扇窗户，所以问题一直不会被发现——直到有一天，一个专门"到处推窗户"的人经过，发现了这个弱点。漏洞很少是"故意"被设计出来的，通常是工程师在赶功能、没考虑到"恶意输入"的情况下，无意间留下的破口。

### 3.2 第二层理解：Application Security，AppSec（应用程序安全）

漏洞可以存在于很多层面——操作系统、网络、硬件——但今天要聚焦的是离一般软件工程师最近的一层：**AppSec**。

**定义**：专门聚焦在"应用程序本身"（也就是工程师写的那些代码、功能逻辑）里的安全问题，而不是底层的服务器或网络设施。

常见的 AppSec 关注的问题类型，除了上面提到的 SQL Injection，还有像是：
- **XSS（Cross-Site Scripting）**：攻击者想办法把恶意程式码，偷偷塞进网页里，让其他访问这个网页的用户，在不知情的情况下执行这段恶意程式码。
- **身份验证漏洞（Authentication flaws）**：例如登入功能没有限制密码错误次数，让攻击者可以无限次疯狂尝试密码。

AppSec 之所以重要，是因为 [[3.1 — Frontend Engineering]] 和 [[3.2 — Backend Engineering]] 的工程师，写的每一行处理"用户输入"的代码，理论上都是攻击者可能下手的地方——功能写得越多，潜在的攻击面（Attack Surface）就越大。

### 3.3 第三层理解：Penetration Testing（渗透测试，俗称 Pen Test）

知道了系统"可能"有漏洞，接下来的问题是——**怎么在攻击者找到之前，自己先找到？**

**定义**：由专业人员，主动地、有系统地"扮演攻击者"，尝试攻破一个系统，目的是在真正的坏人下手之前，先找出所有的弱点。

这些专业人员有时被称为 **Ethical Hacker（白帽黑客）**——他们用的技术手法，和真正的攻击者一模一样，差别只在于：他们是被公司**授权**这么做的，找到漏洞之后会写成报告交给公司修补，而不是拿去做坏事。

**类比**：Penetration Testing 就像是一间银行，花钱请专业的"职业窃贼"，在光天化日、经过授权的情况下，真的尝试闯入自己的金库——试试看保险箱撬不撬得开、监控死角在哪里、警卫换班的空档能不能溜进去。银行宁愿付钱让"自己人"先找出弱点，也不愿意等真正的窃贼哪天闯进来才发现问题。

### 3.4 第四层理解：Secure Development Lifecycle，SDLC-Security（安全开发生命周期）

到这里你可能会想：那是不是等整个产品做完了，最后再找人来做一次 Penetration Testing 检查一遍就好？

**这是新手很容易掉入的误区。** 安全工程真正想推动的理念，叫做 **Secure Development Lifecycle**——把"考虑安全"这件事，融入到 [[1.2 — The Software Development Lifecycle (SDLC)]] 的**每一个阶段**，而不是只在最后才补一次。

```
需求阶段：这个功能会不会开放不该被开放的资料存取？
设计阶段：这段资料要怎么加密储存？
开发阶段：工程师写代码的当下，就避开容易出漏洞的写法
测试阶段：除了功能测试，也做安全测试
部署阶段：上线后持续监控是否有异常攻击行为
```

**类比**：如果盖大楼的时候，一直等到大楼盖好、住户都搬进去了，才第一次检查消防安全，这时候如果发现结构有问题，要拆掉重盖会非常昂贵、也非常危险。正确的做法是从设计蓝图阶段，就把消防通道、防火建材规划进去。安全也是一样的道理——**在开发早期修一个漏洞，成本可能只是改几行代码；等到产品上线被攻击了才发现，成本可能是整个公司的用户资料外泄**，这也是为什么这个理念在业界越来越受重视，甚至衍生出"Shift Left"（把安全检查往开发流程的更早期"左移"）这样的说法。

## 4. 串联全局

我们串一下逻辑链：

[[3.1 — Frontend Engineering]]、[[3.2 — Backend Engineering]] 等领域写出的每一个功能，只要有处理用户输入或数据的地方，就可能存在没被发现的 **Vulnerability** → 聚焦在应用程式层面找出这些弱点的领域叫 **AppSec** → 为了在坏人下手前主动找出弱点，专业人员会执行 **Penetration Testing**，模拟真实攻击 → 而真正治本的做法，是不要等到最后才检查，而是把安全思维融入 **Secure Development Lifecycle** 的每一个阶段，从需求、设计一路贯穿到上线后的监控。

Security Engineering 之所以能横跨、守护 Chapter 3 前面介绍的所有领域，是因为不管是处理用户资料的 Backend、储存海量资讯的 [[3.7 — Data Engineering]]，还是训练模型用到的敏感数据的 [[3.8 — Machine Learning Engineering]]，只要牵涉到软件，就一定有被攻击的风险。这也是为什么在往后 [[5.3 — FinTech & Regulated Industries]] 这类对安全要求极高的产业里，Security Engineering 会是决定公司能不能生存下去的关键领域。

## 5. 五个最重要的概念（Top 5 Takeaways）

### ① Security Engineering
**定义**：负责找出软件系统里可能被攻击的弱点，并提前设计、修补，让系统能抵御恶意攻击的工程领域。
**为什么重要**：其他工程领域默认"用户会正常使用系统"，但现实世界一定存在恶意攻击者——没有 Security Engineering，再好的产品功能都可能因为一次资料外泄事件瞬间崩盘。
**自问自答**：*Security Engineer 和一般写功能的工程师（例如 Backend Engineer），思考问题的角度有什么根本差别？*
→ 一般工程师主要思考"怎么让这个功能对正常用户正确运作"；Security Engineer 则要额外思考"一个恶意的人，会怎么滥用或攻破这个功能"，也就是从攻击者的角度反过来检视系统。

### ② Vulnerability
**定义**：系统里一个可以被攻击者利用、做出设计者未预期行为的弱点。
**为什么重要**：漏洞往往是无意间留下的，而且平常不会被发现，一旦被攻击者找到并利用，可能导致资料外泄、系统被破坏或用户被冒充。
**自问自答**：*为什么说漏洞通常不是"故意"被设计出来的，而是"无意间"留下的？*
→ 因为工程师写功能时，注意力通常放在"如何让正常用户顺利完成操作"，很少会同时设想"如果有人刻意输入奇怪、异常的资料会发生什么"，这种思维上的盲点就会在代码里留下没被察觉的破口。

### ③ AppSec（Application Security）
**定义**：专门聚焦在应用程式本身（工程师写的代码与功能逻辑）里的安全问题的领域。
**为什么重要**：功能写得越多，潜在的攻击面就越大，AppSec 确保开发团队自己写的代码不会成为攻击者最容易下手的入口。
**自问自答**：*为什么一个网站的功能越多、越复杂，通常代表它的安全风险也越高？*
→ 因为每一个处理用户输入、存取数据的功能，理论上都是攻击者可能尝试突破的一个入口点；功能越多，代表需要被检查、被防御的"入口"数量也越多，出错的机率自然更高。

### ④ Penetration Testing
**定义**：由专业人员主动、有系统地扮演攻击者尝试攻破系统，目的是在真正的攻击者之前先找出弱点。
**为什么重要**：与其被动等待被攻击才发现问题，不如主动花钱请专业人员先找出弱点，把主导权掌握在自己手上。
**自问自答**：*Penetration Testing 和真正的恶意攻击，最核心的差别是什么？*
→ 使用的技术手法可能完全相同，但 Penetration Testing 是经过公司正式授权进行的，目的是把找到的漏洞写成报告交回给公司修补，而不是拿这些漏洞去造成实际伤害或谋取不当利益。

### ⑤ Secure Development Lifecycle
**定义**：把安全考量融入软件开发生命周期每一个阶段（需求、设计、开发、测试、部署）的理念，而不是只在产品完成后才检查一次。
**为什么重要**：在开发早期发现并修复一个漏洞，成本远低于等到产品上线、被真正攻击之后才处理，后者的代价可能是大规模的用户资料外泄与信任崩盘。
**自问自答**：*为什么"等产品全部开发完，最后再做一次安全检查"不是一个理想的做法？*
→ 因为这时候如果发现重大的安全问题，可能牵涉到系统架构层面的缺陷，修改成本极高、甚至需要打掉重练；而如果从设计阶段就把安全考量进去，很多问题在写代码的当下就能被避免，成本低很多。
