# 3.1 — Frontend Engineering

## 1. High-Level Map（这个主题在大局里的位置）

想象整个 software engineering 的世界是一间餐厅。

- **Frontend Engineering** = 餐厅的「前场」（dining area）—— 客人看到的桌椅、菜单设计、灯光氛围、服务员跟你互动的方式。
- **Backend Engineering**（3.2，还没教）= 厨房 —— 客人看不到，但食物真正被煮出来的地方。
- **Full-Stack**（3.3）= 一个人同时会做前场又会做厨房的活。

你现在在 **Chapter 3: Fields of Software Engineering**，也就是"软件工程这行业，到底分成哪几种专业方向"。3.1 是第一个方向：**Frontend**。它的定位很单纯——**只讲 frontend 这个领域本身是什么、它负责做什么**。Backend 怎么运作、full-stack 怎么结合，都留到之后的模块，这一节完全不需要碰。

## 2. Fundamentals First（从零开始，解释每个字的"为什么"）

### 什么是 "Frontend"？

**Frontend（前端）= 用户直接看得到、摸得到、点得到的那一层软件。**

任何时候你打开一个网站或 App，你的眼睛看到的按钮、文字、图片、颜色、动画——这一切都是 frontend engineer 做出来的。

反过来想：当你在 Instagram 按下"like"这个爱心按钮时——
- 那个爱心图案会变红、会有个小动画 —— 这是 **frontend** 做的
- 你按下之后，服务器把这个"赞"存进数据库 —— 这是 **backend** 做的（下一节才讲）

Frontend engineer 的世界，就是"**浏览器里发生的事**"（browser-based applications）。

### 为什么会需要"专门"做 frontend 的人？

Why 很重要，我们从第一性原理想：软件要能被人用，就一定要有一个"人机接口"（human-computer interface）。这个接口的复杂度，随着互联网发展越来越高：

- 1990 年代网页只是静态文字 + 图片，随便一个人都能写
- 现在的网页要处理：动画、即时更新（比如 WhatsApp 消息秒到）、跨装置适配（手机/平板/电脑都要好看）、无障碍设计（accessibility，给视障用户用的）、效能优化（页面要快，不然用户流失）

这个复杂度高到需要一整个专门的工程领域去处理它 —— 这就是 frontend engineering 存在的原因。

### 核心概念一：User Interface (UI)

**UI = 软件呈现给用户看、给用户操作的所有视觉与互动元素的集合。**

按钮、输入框、下拉菜单、进度条——这些全部叫 UI。Frontend engineer 的核心工作，就是把设计师（UX/UI designer）画出来的设计稿，变成一个真正能点、能用、会动的东西。

> 类比：设计师画的是"建筑蓝图"，frontend engineer 是"把蓝图盖成真实房子"的建筑工人。

### 核心概念二：Browser-based Applications（基于浏览器的应用）

Frontend 的"战场"主要是 **web browser**（Chrome、Safari 这些）。你打开浏览器输入网址，浏览器下载一堆代码，然后**在你自己的电脑/手机上**把这些代码执行、画出画面——这个过程叫 **rendering（渲染）**。

关键洞察：这些代码不是在远方的服务器上跑的，而是**下载到你的设备上，在你的设备上跑**。这就是为什么 frontend 代码几乎都是用一种特定的语言写的——因为浏览器只看得懂这一种语言组合。

### 核心概念三：HTML / CSS / JavaScript —— 前端的三大件

这三个是 frontend 世界的"地基"，浏览器天生只认识这三种语言。用盖房子来比喻最直观：

| 语言 | 角色 | 类比 |
|---|---|---|
| **HTML** (HyperText Markup Language) | 定义页面**有什么内容、结构** | 房子的骨架——这里是墙、这里是门 |
| **CSS** (Cascading Style Sheets) | 定义**长什么样**（颜色、字体、排版） | 装潢——刷什么颜色的漆、铺什么地板 |
| **JavaScript** (JS) | 定义**怎么互动**（点击、动画、逻辑） | 电路系统——开关灯、门铃会响 |

三者缺一不可：
- 只有 HTML → 页面像没上色的黑白草图，能看但很丑，也不能点
- 加上 CSS → 页面变漂亮，但按钮按下去没反应，是"死"的
- 加上 JavaScript → 页面"活"起来，按钮能点、表单能送出、数据能即时更新

### 核心概念四：Frontend Frameworks（前端框架）

**Framework = 别人已经帮你写好一大堆"常用积木"的工具包，让你不用每次都从零造轮子。**

举例：如果每个网站的"下拉菜单"、"弹出视窗"都要工程师自己手写 JavaScript 逻辑，那效率太低，而且每个人写的方式都不一样、容易出 bug。

于是业界发展出 framework，比如：
- **React**（Facebook / Meta 开发）
- **Vue**
- **Angular**（Google 开发）

这些 framework 提供一套"标准化的做法"，让 frontend engineer 用更少的代码、更少的 bug，做出更复杂的互动网页（比如 Gmail 那种即时更新、不用整页刷新的体验）。这也是为什么求职时你常看到"React Developer"这种职称——因为业界几乎已经默认你会用某个 framework，而不是纯手写 HTML/CSS/JS。

## 3. Progressive Explanation 小结：Frontend Engineer 到底"拥有"什么？

回到这节的 Learning Objective：**Define the frontend field and what it builds.**

一句话总结：**Frontend engineer 负责一切"用户在浏览器里看到、点到、感受到"的东西**——从画面结构（HTML）、视觉风格（CSS）、互动逻辑（JavaScript），到用 framework（React 等）把这些整合成一个流畅好用的产品体验。

他们**不负责**（这些留到 3.2 Backend）：
- 数据存在哪里、怎么存
- 服务器怎么处理请求
- 商业逻辑（比如"库存少于 10 件时不能下单"这种规则通常算在 backend）

## 4. 五个你必须完全掌握的核心概念

**1. Frontend（前端）**
- **定义**：软件中用户能直接看到、点击、互动的那一层。
- **为什么重要**：不管产品的后台逻辑多强大，如果 UI 难用、卡顿、丑陋，用户根本不会用——frontend 是用户对产品的"第一印象"和"唯一体验"。
- **自测题**：你按下 Instagram 的爱心按钮，"爱心变红"这个动作是 frontend 还是 backend 做的？
  → **答案：Frontend**。因为这是用户在浏览器/App 画面上直接看到的视觉反应。

**2. User Interface (UI)**
- **定义**：软件呈现给用户操作的所有视觉与互动元素（按钮、菜单、输入框等）。
- **为什么重要**：UI 是设计和工程的交界点——frontend engineer 的日常工作就是把设计稿变成可运行的 UI。
- **自测题**：设计师画的按钮草图和 frontend engineer 写出来的按钮，两者的关系是什么？
  → **答案：设计师画蓝图（视觉设计），frontend engineer 用代码把蓝图变成真正能点击、有功能的按钮。**

**3. Browser-based Applications（浏览器应用）**
- **定义**：代码被下载到用户自己的设备上，由浏览器在本地执行并画出画面的应用程序。
- **为什么重要**：这解释了为什么 frontend 代码有特殊限制——必须是浏览器看得懂的语言，且要考虑不同用户的设备性能、网速。
- **自测题**：当你打开一个网页时，网页的代码是在哪里被执行、画出画面的？在遥远的服务器上，还是在你自己的电脑/手机上？
  → **答案：在你自己的设备上（本地），浏览器下载代码后自行渲染画面。**

**4. HTML / CSS / JavaScript**
- **定义**：浏览器唯一能读懂的三种语言——HTML 管结构，CSS 管样式，JavaScript 管互动逻辑。
- **为什么重要**：这是所有 frontend 工作的地基，不管你之后用什么 framework，最终都会被转换成这三种语言让浏览器执行。
- **自测题**：如果一个网页只有 HTML，没有 CSS 和 JavaScript，会是什么样子？
  → **答案：内容都在，但没有样式（黑白、没排版、很丑），而且按钮点了也不会有任何反应（是"死"的静态页面）。**

**5. Frontend Frameworks（如 React）**
- **定义**：预先写好、可重复使用的代码工具包，让工程师不用从零手写所有互动逻辑。
- **为什么重要**：现代网站的互动复杂度极高，framework 让团队能用标准化、可维护的方式快速开发，也是业界招聘时的常见技能要求。
- **自测题**：为什么现在的公司招聘 frontend engineer 时，常常要求"会 React"而不是只要求"会 HTML/CSS/JS"？
  → **答案：因为纯手写 HTML/CSS/JS 在处理复杂互动（如即时更新、组件复用）时效率低、容易出 bug；React 这类 framework 提供了标准化、更快速且更易维护的开发方式，是业界的实际工作标准。**
