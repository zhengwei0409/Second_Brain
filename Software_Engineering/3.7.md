# 3.7 — Data Engineering

---

## 1. High-Level Map（这个主题在大局中的位置）

先带你回顾一下我们现在走到哪了。在 Chapter 3 里，我们已经逛过好几个"专科"：

- [[3.1 — Frontend Engineering]] → 造用户看得到的东西
- [[3.2 — Backend Engineering]] → 造服务器、API、数据库逻辑
- [[3.6 — Mobile Engineering]] → 造手机上的 App

这些领域有一个共同点：他们都在处理**产品功能**——让用户"做某件事"（点击按钮、下单、发消息）。

但今天要介绍的 **Data Engineering（数据工程）**，是一个完全不同性质的领域。它不是为了"让用户做事"，而是为了回答一个问题：

> **公司产生了海量的数据，但这些数据乱七八糟散落各处，怎么把它们变成"有用、干净、能被分析"的东西？**

这就是 Data Engineering 存在的理由。而在 Chapter 3 的排列顺序上，它是一个转折点——从这里开始，我们要从"造产品的工程师"过渡到"处理数据/智能的工程师"，直接铺路到下一个主题 [[3.8 — Machine Learning Engineering]]。

## 2. Fundamentals First（从零开始）

### 2.1 数据从哪里来？先建立一个画面

想象你在经营一个像 Shopee 这样的电商平台。每一秒钟，系统里都在发生：

- 用户点击了某个商品（Frontend 产生的事件）
- 用户下单付款（Backend 数据库写入一笔订单）
- App 记录了用户在哪个页面停留多久（Mobile 产生的日志）
- 客服系统记录了一通电话（另一个系统的数据）

这些数据分散在**几十个不同的系统**里，格式还都不一样——有的是数据库表格，有的是纯文字 log，有的是 JSON 文件。

**问题来了**：如果老板想问一句"我们上个月在马来西亚的营收成长了多少？"——这个答案散落在五六个不同系统里，没有人能直接回答。

这，就是 Data Engineer 要解决的核心痛点。

### 2.2 什么是 Data Engineering？

**定义**：Data Engineering 是专门负责"搬运、整理、清洗数据"的工程领域，他们建造的是让数据从"混乱的原始状态"变成"干净、可分析状态"的基础设施。

用一个类比来理解：如果把整间公司比喻成一座城市，那么各个系统（Frontend、Backend、Mobile）就是这座城市里不同的"水源"（河流、井、雨水）。而 Data Engineer 的工作，就是建造**自来水管线系统**——把散落各处的水源，收集起来、过滤干净、送到每一户人家的水龙头。

没有这套管线，即使城市里到处都是水（数据），也没有人能安全地喝到一杯干净水（可用的分析结果）。

## 3. Progressive Explanation（逐层深入）

### 3.1 第一层理解：Data Pipeline（数据管道）

Data Engineer 最核心的产出物叫做 **Data Pipeline（数据管道）**。

**定义**：一套自动化流程，负责把数据从"来源"搬运到"目的地"，路上通常还会做一些加工处理。

它通常长这样：

```
[原始数据来源] → [搬运 + 处理] → [干净的存放地]
  (App logs,         (Pipeline)      (Data Warehouse)
   数据库, API...)
```

就像工厂的生产线：原料（原始数据）进入一端，经过一道道工序（清洗、转换、去重），出来的是可以直接卖给客人的成品（干净的数据表）。这套"生产线"，就是 Pipeline。而且这条生产线通常是**自动化、按时跑的**——比如每天凌晨 2 点自动跑一次，把昨天的数据处理好。

### 3.2 第二层理解：ETL / ELT

这是 Pipeline 具体"怎么做"的方法论，拆成三个步骤来理解：

- **E = Extract（提取）**：从原始来源把数据"捞出来"。例如从 App 的数据库、log 文件里把数据抓出来。
- **T = Transform（转换）**：把捞出来的数据整理干净。比如把日期格式统一、把重复的记录去掉、把货币单位统一换算成美金。
- **L = Load（载入）**：把处理好的数据，放进最终的存放地。

那 **ETL** 和 **ELT** 的差别是什么？只是顺序不同：

- **ETL**：先转换（Transform），再载入（Load）——先在半路上把数据洗干净，才存进仓库。
- **ELT**：先载入（Load），再转换（Transform）——先把原始数据整包倒进仓库，之后才在仓库里慢慢清洗。

**类比**：ETL 就像你去市场买菜，先在市场旁边把菜洗好切好，才带回家放进冰箱。ELT 则是你把整篮子生菜直接搬回家堆进大冰箱，之后要用的时候再拿出来洗切。ELT 之所以在近几年更流行，是因为现代的仓库（下面会讲）运算能力很强，可以直接处理原始数据，不需要在"半路"上先加工。

### 3.3 第三层理解：Data Warehouse（数据仓库）

那"最终的存放地"到底是什么？就是 **Data Warehouse（数据仓库）**。

**定义**：一个专门为了"分析"而设计的大型数据存放系统，跟 [[3.2 — Backend Engineering]] 里讲的那种数据库不一样。

这里要澄清一个新手很容易混淆的点：

- Backend 用的数据库（例如存用户订单的那种），是为了**快速处理单笔交易**而设计的——例如"帮用户扣款"这种一次一笔的操作，要求极快、极准确。这叫 **OLTP（Online Transaction Processing）**。
- Data Warehouse 则是为了**一次分析海量历史数据**而设计的——例如"算出过去三年每个月的总营收"，这种要扫过几百万笔资料做汇总的操作。这叫 **OLAP（Online Analytical Processing）**。

**类比**：Backend 数据库像是便利店收银台——一次服务一位顾客，快进快出。Data Warehouse 像是仓储物流中心的大数据分析室——不服务单一顾客，而是一次盘点仓库里所有货物的趋势。两者的设计目标完全不同，所以是两种不同的系统。

### 3.4 第四层理解：Big Data

当数据量大到传统方法处理不了的时候，我们就进入了 **Big Data（大数据）** 的范畴。

**定义**：泛指数据量巨大、增长快速、或格式复杂到传统单台电脑/传统数据库无法有效处理的数据。

常用一个"3V"框架理解：
- **Volume（量）**：数据大到 TB、PB 等级（1 PB = 100 万 GB）
- **Velocity（速度）**：数据产生的速度极快，例如每秒几十万笔点击事件
- **Variety（多样性）**：数据格式五花八门——文字、图片、影片、log

当公司数据达到这个规模，就需要专门的分布式系统（把运算拆到很多台机器上一起处理）来应对，这也是为什么 Data Engineering 会发展出自己专属的一整套工具生态（如 Spark、Hadoop 这类工具，这里先知道名字就好，之后有需要再深入）。

## 4. 串联全局

我们串一下逻辑链：

各个 field（[[3.1 — Frontend Engineering]]、[[3.2 — Backend Engineering]]、[[3.6 — Mobile Engineering]]）在日常运作中，都会产生数据 → 这些数据散落各处、格式混乱 → Data Engineer 建造 **Pipeline**，用 **ETL/ELT** 的方法把数据清洗、搬运 → 最终存进专门为分析设计的 **Data Warehouse** → 当数据量膨胀到 **Big Data** 等级时，需要更专业的基础设施来应对。

而这整套"干净、有结构的数据"，正是下一章 [[3.8 — Machine Learning Engineering]] 存在的前提——没有 Data Engineer 打好的地基，Machine Learning 工程师根本没有干净的数据可以拿来训练模型。这也是为什么这两个领域在 curriculum 里紧紧相连。

## 5. 五个最重要的概念（Top 5 Takeaways）

### ① Data Engineering
**定义**：负责把混乱、分散的原始数据，搬运整理成干净、可分析数据的工程领域。
**为什么重要**：没有干净的数据，公司无法做出正确的商业决策，也无法训练任何 AI 模型——它是所有"数据驱动"工作的地基。
**自问自答**：*Data Engineer 和 Backend Engineer 的核心差别是什么？*
→ Backend Engineer 专注于让产品功能正常运作（例如处理一笔订单）；Data Engineer 专注于把各系统产生的数据整理成可以拿来分析、决策用的干净资料。

### ② Data Pipeline
**定义**：把数据从来源自动搬运、加工到目的地的一整套流程。
**为什么重要**：它让数据处理变成自动化、可重复的流程，不需要每次都靠人工手动整理数据。
**自问自答**：*为什么 Pipeline 通常要"自动化定时跑"，而不是需要的时候才手动跑一次？*
→ 因为数据每天都在持续产生，若靠人工处理既慢又容易出错；自动化能保证数据永远是最新、一致、可靠的。

### ③ ETL / ELT
**定义**：Extract（提取）、Transform（转换）、Load（载入）三个步骤的组合，差别在于"转换"这一步发生在载入仓库之前（ETL）还是之后（ELT）。
**为什么重要**：这是数据从"脏"变"干净"的具体方法论，决定了整个 Pipeline 怎么设计。
**自问自答**：*ELT 为什么在近几年比 ETL 更受欢迎？*
→ 因为现代 Data Warehouse 运算能力大幅提升，可以直接吞下大量原始数据，之后再依需要弹性地做转换，比起提前固定好转换逻辑更灵活。

### ④ Data Warehouse
**定义**：一个专门为大规模数据分析设计的存放系统，区别于日常交易用的数据库。
**为什么重要**：它是公司做商业分析、生成报表、训练 AI 模型时真正会去读取数据的地方。
**自问自答**：*为什么不能直接拿 Backend 那种处理订单的数据库来做数据分析？*
→ 因为那种数据库（OLTP）是为"快速处理单笔交易"设计的，用来扫描几百万笔历史资料做汇总分析会非常慢、甚至拖垮线上服务；Data Warehouse（OLAP）才是为大规模分析优化过的系统。

### ⑤ Big Data
**定义**：数据量、产生速度、或格式复杂度大到传统方法无法有效处理的数据规模（常用 Volume、Velocity、Variety 三个维度描述）。
**为什么重要**：数据规模一旦达到这个等级，公司必须投资专门的分布式基础设施和专业人才，这也是 Data Engineering 作为独立高薪领域存在的重要原因之一。
**自问自答**：*一家小型 startup 每天只有几百笔用户数据，需要马上建置 Big Data 基础设施吗？*
→ 不需要。Big Data 工具是为了解决"传统方法处理不了"的规模问题而存在的；数据量还小的时候，用简单的数据库和基本的 Pipeline 就足够，过早引入复杂的大数据系统反而是浪费资源。
