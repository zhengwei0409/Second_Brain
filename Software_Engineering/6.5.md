# 6.5 — Kanban & Lean

---

## 1. High-Level Map(这个概念在大图里的位置)

上一课 [[6.4 — Scrum]] 我们讲了 Agile 最主流的落地框架:三种角色(Product Owner 决定做什么、Scrum Master 清障、Development Team 决定怎么做)+ 四场仪式(sprint planning、daily standup、sprint review、retrospective),核心骨架是 **sprint** ——一个长度固定的短周期,时间到了就必须停下来、展示成果、重新规划。

但上一课结尾我们也留了一个悬念:Scrum 不等于 Agile。Agile 是一套价值观([[6.3 — Agile & the Agile Manifesto]]),Scrum 只是众多实现之一。这一课要讲的,就是**另一条完全不同的实现路线:Kanban**。

它不同在哪?一句话:

> **Scrum 用"固定长度的时间盒(sprint)"来组织工作,Kanban 干脆把时间盒整个拿掉,改用"限制同时进行的工作数量(WIP limit)"来组织工作——不再是"这两周我们要完成这一批",而是"任何时刻,团队最多只能同时手上有 N 件事在做,做完一件才能拉进下一件"。**

而这个思路的源头,根本不在软件行业——它来自 1950 年代的**丰田汽车工厂(Toyota)**。这也是这一课的另一半内容:**Lean(精益)**,一套先在制造业被验证、后来被整个软件行业借走的思维方式。

用一句话概括这一课:

> **Kanban 是一种基于"持续流动(continuous flow)"而非"固定周期"的工作管理方法——用一块可视化的看板(Kanban board)把工作变成看得见的卡片,用 WIP limits 强制限制同时进行的工作量,让工作像水一样一件一件流过团队;它背后的哲学是源自丰田生产系统的 Lean,核心信念是:一切不为客户创造价值的东西,都是浪费(waste)。**

(范围提醒:这一课的目标是理解 Kanban 作为"sprint 之外的另一种选择"到底是什么、怎么运作。至于"我的团队到底该选 Scrum 还是 Kanban"这个决策问题,不是这一课要回答的——先把两种模式本身理解透。)

---

## 2. Fundamentals First(从地基开始,一个词一个词拆开讲)

### 2.1 先建立地基:一个 Scrum 解决不了的问题

先别急着学新术语。我们先看一个真实的痛点——正是这个痛点,让很多团队用不下去 Scrum。

想象你在一个**运维支持团队(support team)**:公司的线上系统出问题了,用户投诉进来了,老板临时要一个数据报表。这些事情有一个共同特点——**它们不会礼貌地等到下个 sprint 才发生**。

现在按 6.4 学的 Scrum 规则来:sprint backlog 在 sprint planning 时就锁定了,原则上 sprint 期间不该往里塞新东西(不然就是 scope creep)。可你的工作性质就是"随时来活、来了就得马上处理"。结果会怎样?

- 要么你严格守规矩 → 线上着火了还说"这个排进下个 sprint",荒谬。
- 要么你天天破例往 sprint 里加东西 → 那 sprint 这个概念就形同虚设,每次 planning 定的计划全被打乱,retrospective 上永远在检讨"为什么又没完成承诺"。

问题出在哪?出在 **Scrum 的核心假设**上:它假设你**能够提前一到两周,把这段时间要做的事大致规划好**。对做产品功能开发的团队,这个假设通常成立;但对工作以**不可预测的、持续到达的单件请求**为主的团队(运维、客服工程、bug 修复、内容审核),这个假设从根上就不成立。

Kanban 的出发点,就是**放弃这个假设**。它不问"这两周我们要完成哪一批",而问一个更朴素的问题:

> **工作总是会源源不断地来。我怎么让它们一件一件、顺畅地流过团队,而不是堵在中间?**

记住这句话,它是理解后面所有术语的地基。

### 2.2 词源:Kanban 到底是什么意思?丰田工厂里发生了什么

**Kanban(かんばん / 看板)是日语,字面意思就是"看得见的板子 / 招牌"**。它的发明地不是软件公司,而是 1950 年代的**丰田汽车工厂**,发明背景是一个非常具体的制造业难题。

当时美国车厂的主流做法是 **push(推动式)生产**:上游拼命生产零件,做好了就往下游推。逻辑是"机器不能闲着,闲着就是浪费"。结果呢?上游做零件的速度比下游装配快,零件就在车间里堆成山——**堆积的库存(inventory)本身就是巨大的浪费**:占仓库、压资金、可能过期、可能设计一改全报废,而且最要命的是——**如果这批零件有质量问题,等下游发现时,已经错做了成千上万个了**。

丰田的大野耐一(Taiichi Ohno)提出了反过来的做法:**pull(拉动式)生产**。

规则很简单:**下游不来要,上游就不许做。** 具体怎么执行?用卡片。下游工位快用完某个零件时,把一张卡片(就是 kanban 卡)送回上游,上游看到卡片才开始补货。**没有卡片 = 没有需求 = 不许生产。**

这个机制的天才之处在于:**卡片的总数,就是库存的上限。** 你不需要复杂的中央计划系统,只要控制卡片数量,整条生产线的在制品数量就被物理地限制住了。

而这个"卡片总数 = 上限"的思路,就是软件业 Kanban 里 **WIP limit** 的直系祖先。丰田这一整套围绕"消除浪费"的生产哲学,后来被西方学者总结成一个词:**Lean(精益)**。

### 2.3 拆解核心术语①:Kanban Board(看板)—— 让工作从"看不见"变成"看得见"

**Kanban board 是一块被划分成若干列(column)的板子,每一列代表工作流程中的一个阶段;每一张卡片(card)代表一件具体的工作,卡片从左往右移动,代表这件工作在流程中往前推进。**

最基础的形态长这样:

```
┌─────────┬──────────────┬─────────────┬──────────┐
│  To Do  │ In Progress  │  In Review  │   Done   │
├─────────┼──────────────┼─────────────┼──────────┤
│ [卡片]  │  [卡片]      │  [卡片]     │ [卡片]   │
│ [卡片]  │  [卡片]      │             │ [卡片]   │
│ [卡片]  │              │             │ [卡片]   │
└─────────┴──────────────┴─────────────┴──────────┘
```

这里有一个**极其重要、但初学者几乎都会搞错的点**,必须现在讲清楚:

> **Kanban board 的列,不是随便起的名字,而是必须如实反映你团队"真实的"工作流程。**

什么意思?比如你的团队实际上每个功能做完后,要先经过 code review(还记得 [[6.1 — Version Control & Git]] 和 [[4.2]] 讲的 PR 流程吗),再经过 QA 测试,才能上线。那你的板子就该有 `In Review` 和 `Testing` 这两列。如果你偷懒只写 `To Do / Doing / Done`,那么所有"卡在等 review"和"卡在等测试"的工作,在板子上都长得一模一样——**你就永远看不见问题堵在哪一环**。

而"看见问题堵在哪一环",恰恰是 Kanban 的全部意义。这就引出了下一个概念。

### 2.4 拆解核心术语②:WIP Limit(在制品限制)—— Kanban 真正的心脏

**WIP = Work In Progress(在制品),指"已经开始、但还没完成"的工作。**

**WIP limit 就是给板子上的某一列,设定一个"最多同时能有几张卡片"的硬性上限。**

```
┌─────────┬──────────────────┬─────────────────┬──────────┐
│  To Do  │ In Progress (3)  │  In Review (2)  │   Done   │
├─────────┼──────────────────┼─────────────────┼──────────┤
│ [卡片]  │  [卡片]          │  [卡片]         │ [卡片]   │
│ [卡片]  │  [卡片]          │  [卡片]  ← 满了 │ [卡片]   │
│ [卡片]  │  [卡片]  ← 满了  │                 │ [卡片]   │
└─────────┴──────────────────┴─────────────────┴──────────┘
```

上图中 `In Progress` 列标了 `(3)`,意思是:**这一列任何时刻最多只能有 3 张卡片。已经满了,你就不许从 To Do 拉新卡片进来——必须先把里面的某一张推到下一列。**

初学者第一反应通常是:"这不是自断手脚吗?限制我们同时做的事,不就做得更少了?"

**恰恰相反,而且这是这一课最反直觉、也最有价值的一个认知。**

想想你自己的经历:同时开着 5 个任务是什么感觉?你在 A 上做一半,想起 B 急,切过去;B 卡住了要等别人回复,切到 C;C 做一半又被 A 的人催……结果一天下来,5 件事都推进了一点,**但一件都没做完**。而且每次切换,你的大脑都要花时间重新加载上下文(**context switching**,上下文切换),这部分时间**纯粹是浪费**——它没有为任何一件工作产生进展。

更关键的是,**没做完的工作,价值等于零**。一个做了 90% 的功能,用户能用吗?不能。它对公司产生的价值,和一行没写的功能,**是一样的**——但它已经消耗了你 90% 的成本。这就是丰田车间里"堆积的半成品零件"在软件世界的完美对应:**未完成的工作 = 库存 = 浪费。**

所以 WIP limit 的真正作用是:**强迫团队"完成"而不是"开始"**。当 `In Progress` 满了,你不能开新活,你唯一能做的事,就是去帮忙把已经在里面的某件事推完。这在实践中会产生一个非常健康的副作用——**团队开始互相帮忙**,而不是各自埋头开新坑。

还有第二个、同样重要的作用:**WIP limit 会让瓶颈(bottleneck)自动暴露出来。**

举个例子:如果 `In Review` 这一列老是满的,卡片总是堵在那儿,而 `In Progress` 的人因为下游堵了、推不过去、也开不了新活——**你不需要任何人写报告,板子本身就在大喊:"code review 是我们团队的瓶颈!"** 可能是因为团队里只有一个人有权限 review,可能是大家觉得 review 别人的代码"不算自己的活"。问题被看见了,才可能被解决。

这正是 6.4 提到的 Scrum Master"清除障碍"职责的另一种实现方式——Scrum 靠一个人主动发现和清障,Kanban 靠**流程设计让障碍自己浮出水面**。

### 2.5 拆解核心术语③:Continuous Flow(持续流动)—— 没有 sprint 的世界长什么样

现在把前两个概念合起来,你就能理解 Kanban 和 Scrum 最根本的区别了。

**Scrum 是"批次(batch)"思维**:攒一批工作(sprint backlog)→ 集中两周做完 → 一起交付 → 开会复盘 → 攒下一批。工作是**一波一波**流出去的。

**Kanban 是"流动(flow)"思维**:没有批次,没有固定周期。一件工作做完了,就立刻可以交付(如果搭配 [[6.6 — CI/CD]] 的自动化流水线,甚至可以立刻上线);然后从 To Do 里拉下一件进来。工作是**一件一件、连续不断**地流出去的。

这就是 **continuous flow(持续流动)**。

打个比方:Scrum 像**公交车**——固定时刻表,每 15 分钟发一班,你错过了这班,就得等下一班;Kanban 像**扶手电梯**——没有时刻表,你到了就上,一直在动。

由此带来几个连锁的差异,值得逐条对比:

| | **Scrum** | **Kanban** |
|---|---|---|
| **节奏** | 固定长度的 sprint(通常 2 周) | 无固定周期,持续流动 |
| **交付时机** | sprint 结束时集中交付 | 单件完成即可交付 |
| **计划锁定** | sprint 内范围原则上锁定 | 任何时刻都可调整优先级 |
| **角色** | 明确规定三种角色 | **不规定任何角色**,沿用团队现有结构 |
| **核心约束** | 时间(时间到就结束) | 数量(WIP 满了就不能开新的) |
| **核心度量** | velocity(每 sprint 完成多少 story point) | **cycle time / lead time**(一件事从开始到完成花多久) |

最后一行的两个新词要解释一下,因为它们是 Kanban 的"体检指标":

- **Cycle time(周期时间)**:一张卡片**从开始做**到**做完**,花了多长时间。
- **Lead time(前置时间)**:一张卡片**从被提出**(进入 To Do)到**做完**,花了多长时间。

区别在于 lead time 包含了"排队等待的时间",而 cycle time 不包含。对客户来说,他们感受到的其实是 **lead time**——"我提了这个需求,多久能用上?"

这里藏着 Kanban 一个很实际的优势:因为它没有 sprint,交付是连续的,所以它天然更适合回答"平均多久能交付一件事"这个问题。而团队优化的目标也变得非常清晰:**让 cycle time 越来越短、越来越稳定**。

### 2.6 拆解核心术语④:Lean —— 藏在 Kanban 背后的那套哲学

前面讲的 Kanban board、WIP limit、continuous flow,都是**具体做法**。而 **Lean(精益)**,是这些做法背后的**思维方式**——它源自丰田生产系统(Toyota Production System),后来被 Mary 和 Tom Poppendieck 夫妇在 2003 年的《Lean Software Development》一书中,系统地翻译到软件行业。

Lean 的核心信念只有一句话:

> **一切不为客户创造价值的活动,都是浪费(waste),都应该被消除。**

在丰田的车间里,"浪费"指的是堆积的库存、多余的搬运、等待的机器、生产过剩的零件。**翻译到软件行业**,浪费长这样(这是理解 Lean 最关键的一步,请慢慢读):

1. **未完成的工作(partially done work)** — 写了一半的功能、开了三个月没合并的分支。占用了成本,却零价值。这直接对应车间里的半成品库存,也正是 WIP limit 要消灭的东西。
2. **多余的功能(extra features)** — 做了但几乎没人用的功能。这是软件业最大的浪费之一,而且它不只是浪费了当初的开发时间——**它会永久增加系统的复杂度、测试成本和维护负担**。
3. **等待(waiting)** — 等 code review、等测试环境、等别的部门审批。你的板子上那些堵住的列,量化的就是这个。
4. **切换(task switching)** — 就是 2.4 讲的 context switching。
5. **交接(handoffs)** — 每次工作从一个人/一个团队交给另一个,信息就会损耗一部分。这正是 [[6.2 — Waterfall]] 那种阶段式流程效率低下的深层原因之一。
6. **缺陷(defects)** — bug 越晚被发现,修复成本越高(还记得 6.2 讲的这条规律吗)。

看到了吗?Lean 提供了一个**统一的视角**,让你重新审视之前学过的所有东西:为什么要 CI/CD?因为它消除"等待"和"缺陷"。为什么要小批量交付?因为它减少"未完成的工作"。为什么 Agile 强调"可工作的软件"?因为没交付的东西价值为零。

**这就是为什么很多人认为 Lean 比 Agile 更底层**——Agile Manifesto 那四句话,某种意义上都可以从"消除浪费、快速交付价值"这个更根本的原则推导出来。

## 3. Progressive Explanation(把零件拼成完整画面)

### 3.1 一天在 Kanban 团队里是什么样的

我们像 6.4 那样,走一遍真实场景,让你看到这些零件怎么协同运转:

> **早上**:你打开板子,你手上那张卡片(实现"用户导出数据"功能)昨天做完了,你把它从 `In Progress` 拖到 `In Review`。→ **一看**:`In Review` 现在有 2 张卡,而这列的 WIP limit 就是 2,**满了**。→ **这意味着什么**:按规则,你现在**不能**从 To Do 拉新卡片开工(因为你推过去后下游已满,再开新活只会加剧堵塞)。你唯一该做的事,是**去 review 别人卡在那里的代码**,帮忙把那一列疏通。→ **疏通之后**:某张卡片流到了 `Done`,`In Review` 空出位置,你的卡片顺利进入,你才从 To Do 拉下一张卡片开始做。→ **中途**:线上突然出了个紧急 bug。Product Owner(如果团队有这个角色的话)直接把这张卡片放到 To Do 的最顶端——**不需要等下个 sprint,因为根本就没有 sprint**。下一个空出手的人立刻拉走它。

对比一下 6.4 那个 Scrum 的循环,你会发现两者的"驱动力"完全不同:**Scrum 由日历驱动(到点了就该开会/交付),Kanban 由完成事件驱动(有人做完了,才触发下一件事被拉进来)。**

### 3.2 Kanban 的四条基本实践(把前面的内容收成一套可执行规则)

前面拆的术语,业界通常总结成这几条实践,你可以当成"怎么开始用 Kanban"的清单:

1. **Visualize the workflow(可视化工作流)** — 建一块如实反映真实流程的板子,让所有工作和它们所处的阶段都看得见。
2. **Limit WIP(限制在制品)** — 给每一列设上限,强迫团队"完成"而不是"开始"。
3. **Manage flow(管理流动)** — 持续观察卡片流动的速度,盯住 cycle time,找出堵塞点并解决。
4. **Continuous improvement(持续改进)** — 日语叫 **Kaizen(改善)**,同样源自丰田:不追求一次性的大改革,而是持续做小幅度的、永不停止的优化。

第 4 条值得多说一句,因为它揭示了 Kanban 一个常被忽略的性质:**Kanban 不要求你推翻现有流程重来。** Scrum 要求你引入三种角色、四场仪式,是一次组织变革;而 Kanban 的起手式是"**先把你现在的流程原样画到板子上**",然后再逐步改进。这种"低侵入性"是它在很多既有团队里更容易推行的重要原因。

### 3.3 一个必须澄清的误解:Kanban ≠ "没有计划的自由散漫"

因为没有 sprint、没有固定承诺,初学者很容易误以为 Kanban 是"松散版的 Scrum"、纪律更弱。

**恰恰相反。** Scrum 的纪律体现在**时间边界**上(sprint 到点必须停);Kanban 的纪律体现在 **WIP limit** 上,而且这条纪律在日常中往往更难遵守——因为它要求你在"手上活满了、但有件新事看起来很急"的时候,**忍住不开新活**。人的本能是"多接一点显得更努力",而 Kanban 要求你反着本能来。

一个没有认真执行 WIP limit 的 Kanban board,本质上只是一块贴满便利贴的白板,**没有任何 Kanban 的实质**——这大概是这套方法在实践中最常见的失败方式。

### 3.4 那 Kanban 和 Scrum 是对立的吗?

不是。它们是 Agile 价值观的**两种不同实现**,而且在现实中经常被混着用——业界有个词叫 **Scrumban**,指的就是保留 Scrum 的部分仪式(比如 retrospective、daily standup),同时引入 Kanban 的看板可视化和 WIP limit。

事实上,绝大多数用 Scrum 的团队,**也**会用一块看板来追踪 sprint 内的任务(比如 Jira 里的 sprint board)。所以"看板"这个工具和"Kanban 这套方法"是两回事——**用了看板不代表在做 Kanban,真正的分水岭是:你有没有设 WIP limit、有没有放弃固定周期改用持续流动。**

至于"什么情况该选哪个",按这一课的 scope,先记住它们各自的运作机制就够了,选择问题留到你真正身处团队时再判断。

## 4. 五个你必须完全掌握的核心概念

### 1. Kanban Board(看板)
**定义:** 一块按"真实工作流程"划分成若干列的板子,每张卡片代表一件工作,卡片从左往右移动代表工作向前推进,让"有哪些工作、分别卡在哪一步"变成所有人都看得见的信息。
**为什么重要:** 工作在软件行业是"看不见"的——你没法像在车间里那样一眼看到零件堆在哪儿。看板把不可见的工作变成可见的卡片,这是后面所有改进(发现瓶颈、缩短 cycle time)的**前提**:看不见的问题,不可能被解决。
**自测问题:** 一个团队实际上每个功能都要经过 code review 和 QA 测试,但他们的板子只有 `To Do / Doing / Done` 三列。这块板子有什么问题?
**答案:** 它没有如实反映真实流程,导致"正在写代码""在等 review""在等测试"这三种完全不同的状态,在板子上全都混在 `Doing` 里、长得一模一样。团队因此永远看不出工作到底堵在哪一环,也就无法定位瓶颈——而定位瓶颈正是看板存在的核心价值。

### 2. WIP Limit(在制品限制)
**定义:** 给板子上某一列设定的"最多同时能有几张卡片"的硬性上限;这一列满了,就不许再往里拉新工作,必须先把里面的某件事推到下一步。
**为什么重要:** 它是 Kanban 真正的心脏,作用有两个:一是**强迫团队"完成"而不是"开始"**,消灭上下文切换的浪费和"做了 90% 等于零价值"的半成品库存;二是**让瓶颈自动暴露**——哪一列老是满的,哪里就是团队的堵点,不需要任何人写报告。
**自测问题:** 为什么"限制同时进行的工作数量",反而能让团队整体产出更多?
**答案:** 因为价值只在工作**完成**时才产生,做了一半的工作价值为零却已消耗成本。同时开太多任务会导致每件都推进一点、一件都完不成,还要为频繁的上下文切换付出纯浪费的时间。限制 WIP 后,团队被迫集中精力把手上的事做完、并互相帮忙疏通堵塞,单位时间内**真正完成**的工作反而更多。

### 3. Continuous Flow(持续流动)
**定义:** 不设固定周期,工作一件一件连续地流过流程——做完一件即可交付,然后拉下一件进来;与 Scrum"攒一批、集中两周做完、一起交付"的批次模式相对。
**为什么重要:** 它让团队能随时响应新到达的优先事项(不必等下个 sprint),因此特别适合工作以不可预测的单件请求为主的场景(运维、支持、bug 修复);同时,交付不再被时间盒切成批次,价值可以持续不断地流向客户。
**自测问题:** Scrum 的驱动力是"日历",Kanban 的驱动力是什么?这个区别带来的最实际的后果是什么?
**答案:** Kanban 的驱动力是"完成事件"——只有当有人做完一件事、腾出 WIP 名额,下一件工作才被拉进来。最实际的后果是:紧急工作可以随时被排到队首、立刻被处理,而不必等待当前 sprint 结束,因此 Kanban 天然适应工作不可预测的团队。

### 4. Pull System(拉动式系统,源自丰田)
**定义:** 由下游的"需求"触发上游"开始工作"的机制——下游不来拉,上游就不开工;与"上游拼命生产、往下游推"的 push 模式相对。丰田用实体卡片实现它(卡片总数即库存上限),软件团队用 WIP limit 实现它。
**为什么重要:** 它是 Kanban 全部机制的历史与逻辑源头,解释了 WIP limit"为什么长这样"。更重要的是它揭示了一个反直觉的原理:**让每个环节都保持满负荷,并不能让整体产出最大;真正决定整体产出的是最慢的那个环节(瓶颈),上游超速生产只会制造堆积。**
**自测问题:** 丰田为什么认为"车间里堆满已经做好的零件"是一种浪费而不是效率高?这在软件团队里对应的是什么?
**答案:** 因为堆积的零件占资金、占空间、可能因设计变更而报废,而且质量问题会被延迟很久才发现——那时已经错做了成千上万个。在软件团队里,它对应的是**未完成的工作**:写了一半的功能、长期不合并的分支、堆积等待 review 的 PR。它们已经消耗了成本,却尚未产生任何价值,而且拖得越久、出问题时的返工代价越大。

### 5. Lean(精益)与它对"浪费"的定义
**定义:** 源自丰田生产系统、后被翻译到软件行业的一套思维方式,核心信念是"一切不为客户创造价值的活动都是浪费,都应被消除";在软件语境下,浪费主要指未完成的工作、多余的功能、等待、上下文切换、交接损耗和缺陷。
**为什么重要:** 它比 Kanban 更底层——Kanban 只是 Lean 思想的一个具体工具。掌握"什么是浪费"这个统一视角后,你能重新解释此前学过的许多实践:CI/CD 消除"等待"与"缺陷",小批量交付减少"未完成的工作",Agile 强调"可工作的软件"是因为未交付之物价值为零。
**自测问题:** "做了但几乎没人用的功能"为什么是一种严重的浪费?它的成本仅仅是当初开发它花掉的时间吗?
**答案:** 不仅如此。当初的开发时间只是一次性成本,而这个功能一旦存在,就会**永久性地**增加系统的复杂度、测试负担和维护成本——每一次未来的改动都要考虑它、绕开它、为它写测试。所以它的浪费是持续累积的,这也是为什么 Lean 主张"能不做的功能就不做"。

---

## 小结

这一课我们学了 Agile 的另一条落地路线。**Scrum 用固定长度的 sprint 组织工作,靠"时间"约束;Kanban 干脆拿掉时间盒,用 Kanban board 把工作可视化、用 WIP limit 限制同时进行的工作量,让工作以 continuous flow 的方式一件一件流过团队,靠"数量"约束。** 它的驱动力不是日历而是完成事件,因此天然适合工作不可预测、随时到达的团队。

而它背后是更底层的 **Lean** 哲学——源自丰田的 pull system 和"消除一切浪费"的信念。这个视角的价值远超 Kanban 本身:未完成的工作就是库存,等待和交接就是损耗,没人用的功能是永久性的负担。记住这一课最反直觉的那条结论:**限制你同时做的事,反而能让你完成得更多。**
