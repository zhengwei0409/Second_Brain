# 3.10 — Game Development

---

## 1. High-Level Map（这个主题在大局中的位置）

Chapter 3 到现在，我们走过了一连串"造东西给人用"的领域：[[3.1 — Frontend Engineering]] 造网页界面、[[3.2 — Backend Engineering]] 造服务器逻辑、[[3.9 — Security Engineering]] 守护这些系统不被攻破。这些领域的产品，本质上都在解决一个"实用型"问题——买东西、传讯息、管理资料。

而今天要介绍的 **Game Development（游戏开发）**，目标完全不一样：它造的不是"帮你完成任务的工具"，而是**"让你自愿花时间投入其中的体验"**。同样是写代码、同样要考虑性能与架构，但游戏开发工程师脑子里想的问题，跟 Backend Engineer 想的问题，从根本上就不一样——不是"这个系统够不够稳定、够不够快",而是"这个画面看起来够不够流畅、这个操作手感好不好玩"。

这就是为什么 Game Development 值得被当成一个**独立的领域**来认识：它虽然也用到编程、也需要架构设计，但它的评判标准、开发文化、甚至工程师每天在烦恼的事情，都跟前面九节介绍的领域有明显的分岔。

---

## 2. Fundamentals First（从零开始）

### 2.1 先建立一个画面：游戏和一般软件，到底哪里不一样？

想象你在用一个记帐 App（一般软件）：你点一个按钮，App 把资料存进数据库，画面更新，结束。这个过程发生得快一点、慢一点（例如 0.1 秒还是 0.5 秒），你大概都不会太在意。

现在想象你在玩一个动作游戏：你按下"跳跃"键，角色必须**在你能感知的瞬间**（通常要求 1 秒里更新画面 30 到 60 次以上）就做出反应。如果反应慢了、卡顿了，你会立刻感觉到"这游戏怪怪的"，甚至因为慢半拍而在游戏里"被打死"。

**这就是游戏开发最核心的特殊性：它对"每一瞬间"的要求，比一般软件严苛得多。** 一般软件工程师追求的是"最终结果正确"，而游戏工程师除了要结果正确，还必须让**每一秒钟发生的过程本身**都流畅、都有回馈感。

### 2.2 什么是 Game Development？

**定义**：Game Development 是专门负责设计与打造电子游戏的工程领域，核心工作是让代码持续、快速地计算画面与逻辑，创造出玩家可以互动、感觉"活起来"的虚拟世界。

用一个类比理解：如果说 [[3.2 — Backend Engineering]] 像是在盖一间"办公大楼"——目标是让每个人进来把事情办好、准时下班；那么 Game Development 就像是在盖一座"游乐园"——目标不是效率，而是让人愿意留下来、一玩再玩、甚至忘记时间。这两种"建筑"用的工具（编程语言、数据结构、算法）有很多是共通的，但盖出来的东西,评判的标准完全不同：办公大楼看"能不能用",游乐园看"好不好玩"。

## 3. Progressive Explanation（逐层深入）

### 3.1 第一层理解：Real-Time Rendering（实时渲染）

**定义**：让电脑在极短时间内（通常一秒钟内要重复 30~60 次以上）持续计算并画出画面，制造出"动作是连续、流畅"的错觉。

你可能会好奇：游戏画面看起来是连续的动作，其实背后是什么原理？答案跟卡通动画一样——**它其实是一张一张的静止图片，只是切换得非常快，快到人眼分辨不出来，于是大脑把它脑补成"流畅的动作"。** 每一张图片叫一个 **Frame（帧）**,一秒钟画出几张,就叫 **FPS（Frames Per Second）**。FPS 越高,画面通常感觉越顺;一旦 FPS 掉下来（俗称"掉帧"）,玩家立刻会感觉到卡顿。

**类比**：想象你在快速翻动一叠画着连续动作的便利贴（像小时候玩的"手翻书"）。翻得够快，看起来就像动画；翻得慢一点或者中间漏翻几张，画面就会"一格一格"跳动，很不自然。Real-Time Rendering 要解决的问题，就是"电脑能不能在极短时间内，一直画出下一张便利贴，永远不漏拍"。这也是为什么游戏对电脑硬件（尤其是显卡 GPU）的要求，通常比一般软件高出很多——因为它要做的计算量，是每秒钟重画整个世界，而不是等你点击才计算一次。

### 3.2 第二层理解：Game Engine（游戏引擎）

知道了游戏需要疯狂地"每秒重画画面",接下来的问题是——**每个游戏公司,难道都要从零开始，自己写这套画面计算的机制吗？**

**定义**：Game Engine 是一套已经写好、专门用来开发游戏的软件框架，里面预先建好了渲染画面、物理碰撞、声音播放、输入侦测（键盘/手把）等游戏共同都需要的底层功能，让开发者不用每次都重新发明轮子。

业界最知名的两套引擎是 **Unity** 和 **Unreal Engine**。你可以把它们想成游戏界的"半成品"——引擎公司已经把"怎么让画面动起来、怎么侦测两个物体有没有撞在一起"这些又难又通用的底层问题解决好了，游戏工作室拿到这套工具之后，专注在"设计关卡、角色、剧情、玩法"这些真正让这个游戏独一无二的部分。

**类比**：这就像 [[3.2 — Backend Engineering]] 里的框架（例如写网站不用从零手刻怎么处理网络请求），Game Engine 就是游戏版本的框架——只是它要处理的"共同麻烦事"更复杂：不只是资料怎么传输，还包括画面怎么渲染、物体怎么模拟重力掉落、碰撞要怎么侦测。

### 3.3 第三层理解：Gameplay Programming（玩法编程）

有了 Game Engine 处理好底层的渲染和物理，游戏公司里的工程师每天实际在写的代码，到底是什么？

**定义**：Gameplay Programming 是专门编写"游戏规则与玩家互动逻辑"的工作——例如"按下这个键角色要跳多高"、"敌人看到玩家之后要怎么追"、"打倒这只怪物会掉落什么道具"。

这跟前面提到的 Real-Time Rendering、Game Engine 底层机制不一样：渲染和引擎解决的是"这个世界能不能被画出来、能不能模拟物理"这种偏技术、偏通用的问题；而 Gameplay Programming 解决的是"这个游戏好不好玩"这种偏设计、偏创意的问题。写 Gameplay 的工程师，某种程度上要同时具备工程能力，又要有一点"设计师的直觉"——因为同一段代码，参数调得不一样（例如角色跳跃的高度差 0.1 秒的延迟），玩家的手感可能天差地远。

**类比**：如果 Game Engine 是电影摄影棚里已经架好的摄影机、灯光、音响系统，那 Gameplay Programming 就是导演跟演员在这个摄影棚里，实际决定"这一幕角色要怎么演、剧情怎么走"。摄影棚（引擎）搭得再好，没有导演（Gameplay Programmer）设计出好玩的互动，这部片子（游戏）还是不会好看。

### 3.4 第四层理解：Crunch Culture（赶工文化）

前面三点都在讲游戏开发的**技术特性**，这一点要讲游戏开发一个很不一样、也常常被讨论的**产业文化特性**。

**定义**：Crunch Culture 指的是游戏产业里，在产品即将上线前，团队被要求长时间、高强度加班（有时长达数周甚至数月）赶工的现象。

为什么游戏产业特别容易出现这个问题？一个重要原因是：**"好不好玩"这件事很难在开发初期就准确评估。** 写一个登入功能，做完了大概就是做完了；但一个游戏关卡"好不好玩"，往往要等到接近完成、可以实际玩起来之后，团队才真正知道，这也导致大量的修改、调整、甚至推翻重做,常常集中在开发后期——也就是最接近上线日期的时候发生，逼得团队不得不长时间加班。这个现象在业界已经引起许多讨论,不少工作室和从业者近年也在推动更健康的开发节奏与劳动权益保障。

**类比**：这有点像办一场婚礼——场地、请帖这些"确定的事项"可以提早规划好，但"整场婚礼流程顺不顺、气氛好不好"，往往要到彩排、甚至现场进行时才真正知道效果，于是常常在婚礼前几天,还在疯狂调整细节到深夜。游戏开发的"好玩与否"就像这种难以提早验证的部分，让赶工在这个行业特别常见。

## 4. 串联全局

我们串一下逻辑链：

游戏之所以能让玩家感觉"世界是活的"，靠的是 **Real-Time Rendering** 让画面每秒重画数十次 → 因为自己从零打造这套渲染与物理机制太困难，游戏公司普遍使用现成的 **Game Engine**（如 Unity、Unreal）打底 → 在引擎处理好底层技术之后，真正决定"这个游戏好不好玩"的，是工程师撰写的 **Gameplay Programming**，也就是玩家互动与游戏规则本身 → 而正因为"好不好玩"很难在早期就验证完成，这个行业衍生出了独特、也备受讨论的 **Crunch Culture**。

Game Development 之所以值得独立于 [[3.1 — Frontend Engineering]]、[[3.2 — Backend Engineering]] 之外被认识，是因为它虽然共享编程这个工具，但追求的目标（体验与乐趣，而非纯粹效率）、技术门槛（实时画面计算）、以及产业文化（Crunch），都自成一格。这也是为什么想进入游戏产业的工程师，往往需要额外学习 Unity 或 Unreal 这类专门工具，而不是只靠一般 Web 开发的技能就能直接转职。

## 5. 五个最重要的概念（Top 5 Takeaways）

### ① Game Development
**定义**：专门设计与打造电子游戏的工程领域，核心是让代码持续计算画面与逻辑，创造玩家可互动的虚拟世界。
**为什么重要**：它的评判标准（好不好玩）跟一般软件（好不好用）完全不同，代表了一整套不一样的技能树、工具链与职涯路径。
**自问自答**：*Game Development 和一般软件开发，最根本的目标差异是什么？*
→ 一般软件追求"帮用户有效率地完成任务"；游戏开发追求"让玩家自愿投入时间、获得乐趣与体验"，评判标准从"实用"变成了"好玩"。

### ② Real-Time Rendering
**定义**：电脑在极短时间内（通常一秒 30~60 次以上）持续计算并画出画面，制造动作连续流畅的错觉。
**为什么重要**：这是游戏"感觉是活的"的技术基础，也是游戏对硬件性能要求远高于一般软件的主要原因。
**自问自答**：*为什么游戏画面掉帧（FPS 下降）会让玩家立刻感觉到不对劲？*
→ 因为游戏的流畅动作其实是靠一张张画面快速切换制造出来的错觉，一旦画面更新速度跟不上，切换就变得不连续，大脑无法再把它脑补成流畅动作，玩家就会感觉卡顿。

### ③ Game Engine
**定义**：预先写好渲染、物理、声音、输入侦测等游戏共同需要的底层功能的软件框架，例如 Unity、Unreal Engine。
**为什么重要**：让游戏工作室不必从零打造极其困难的底层技术，能专注在设计出这个游戏独一无二的玩法与内容。
**自问自答**：*为什么游戏公司大多选择使用 Unity、Unreal 这类现成引擎，而不是自己从零打造？*
→ 因为渲染画面、模拟物理碰撞这些底层机制的技术难度非常高、开发成本极大，使用现成引擎能让团队把精力集中在真正决定游戏是否好玩的关卡设计与玩法上。

### ④ Gameplay Programming
**定义**：编写游戏规则与玩家互动逻辑的工作，例如角色如何移动、敌人如何反应、道具如何掉落。
**为什么重要**：这是决定一款游戏"手感"与"好不好玩"的关键部分,同一套引擎,不同的 Gameplay 设计能做出完全不同体验的游戏。
**自问自答**：*Gameplay Programming 和 Game Engine 负责的工作,分工上有什么不同？*
→ Game Engine 负责通用、技术性的底层机制（画面怎么渲染、物体怎么碰撞）；Gameplay Programming 负责在这个底层之上，设计具体的游戏规则与互动方式，决定这款游戏独特的玩法与手感。

### ⑤ Crunch Culture
**定义**：游戏产业里，产品上线前团队被要求长时间高强度加班赶工的现象。
**为什么重要**：这是游戏开发一个鲜明的产业文化特征，理解它有助于对这个行业的工作节奏与劳动条件建立实际的预期。
**自问自答**：*为什么"好不好玩"很难提早验证，容易导致游戏开发后期容易出现赶工？*
→ 因为许多功能与关卡要等到接近完成、能实际玩起来时，团队才真正知道效果好不好，导致大量调整与修改集中在最接近上线日期的开发后期，压缩了时间因而容易引发长时间加班。
