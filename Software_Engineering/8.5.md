# 8.5 — Agentic Coding & AI-Native Engineering (2024–2026)

## 1. High-Level Map：这个topic在讲什么？

我们继续走在 **Section 8: Evolution to 2026** 这条路上。先快速回顾一下，好让你感受到这个topic是怎么"长出来"的：

- **8.1 DevOps** — 打破Dev跟Ops之间的墙
- **8.2 Cloud-Native** — 基础设施变了（Docker、Kubernetes）
- **8.3 Remote Work** — 工作地点变了（COVID逼大家在家工作）
- **8.4 AI Coding Assistants** — AI开始"辅助"你写代码（Copilot、ChatGPT）
- **8.5（现在这个）Agentic Coding** — AI从"辅助"进化成"**代理/委托**"

上一个topic 8.4的结尾我特别强调了一个界线：2021-2024年的AI，角色是**"副驾驶"（co-pilot）**——它建议、你决定；它生成、你审查。你还是坐在驾驶座上，手握方向盘，AI只是坐在旁边帮你看路。

8.5要讲的，就是这条界线**从2024年开始被打破**。AI不再只是"建议下一行代码"，而是可以：

> 拿到一个任务描述，自己规划步骤、自己写代码、自己执行、自己检查结果，然后回来跟你报告"我做完了，你要不要看一下？"

这就是这个topic标题里"**Agentic**"这个字的意思——AI从一个"工具"，进化成一个有自主行动能力的"**代理人/agent**"。而你现在正在使用的Claude Code，其实就是这个topic正在讲的东西的真实案例。

## 2. Fundamentals First：先搞懂几个基本概念

### 2.1 什么是 "Agent"（代理人）？

在讲"agentic coding"之前，一定要先搞懂 **agent** 这个字。

先用一个生活比喻：假设你要装修房子，你有两个选择——
1. 你自己买建材、自己规划、自己动手做每一步（这是"工具"模式：你用锤子，锤子不会自己想事情）
2. 你请一个**装修承包商（contractor）**，你只告诉他"我要一个北欧风的客厅"，他自己去买建材、自己找工人、自己安排流程、做完了叫你来验收

**Agent** 就是软件工程版本的"承包商"。技术上的定义是：

> 一个AI系统，不只是"回答问题"或"生成一段文字/代码"，而是能够**自主执行一连串的步骤**——观察现状、做决定、采取行动（例如执行代码、读写档案、查资料）、检查结果，然后根据结果决定下一步该做什么，一直重复这个循环直到任务完成。

这跟8.4的ChatGPT/Copilot最大的差别在于：ChatGPT只能"说"，它没办法自己去执行程式码、自己看执行结果、自己修正错误。而agent**可以**。

### 2.2 什么是 "Agentic Workflows"（代理式工作流程）？

**Agentic workflow** 指的是：AI agent处理一个任务时，实际跑过的那一套"感知→决策→行动→检查"的循环流程。

用一个具体例子帮你感受，假设你叫一个coding agent："帮我把这个项目里所有用到过时API的地方都改成新版API"：

1. **感知（Perceive）**：Agent先搜寻整个codebase，找出所有用到旧API的档案
2. **决策（Decide）**：Agent规划：先改A档案、再改B档案，因为B档案依赖A档案的改动
3. **行动（Act）**：Agent实际去修改A档案的代码
4. **检查（Check）**：Agent跑测试(test)，看看改完之后代码还能不能正常运作
5. **循环**：如果测试失败，agent自己看错误讯息、自己修正，再跑一次测试；如果测试通过，才继续处理下一个档案

注意这整个过程中，**人类没有介入每一步**——你只在一开始下了指令，最后agent才回来给你看结果。这跟8.4的Copilot（每打一行字就要你按Tab确认）是完全不同等级的自主性。

### 2.3 什么是 "Vibe Coding"（2025）？

这是一个2025年才流行起来、听起来很轻松但背后有严肃含义的词，你一定要理解清楚，因为坊间对它有很多误解。

**Vibe coding** 字面上的意思是"凭感觉写代码"，具体指的是：

> 工程师(或甚至非工程师)只用自然语言描述"我想要的东西长什么样子、感觉怎么样"，完全依赖AI agent生成、执行、甚至部署代码，自己几乎不看、也看不懂底层代码写了什么，纯粹凭直觉判断"这个东西用起来感觉对不对"。

举个例子：一个不太会写代码的产品经理(PM)想做一个小工具，他直接跟AI说"我要一个网页，使用者可以上传一张照片,然后AI帮忙自动去除背景",AI agent自己写完整个网站、自己测试、自己部署上线,这个PM从头到尾**没有读过一行代码**,只是不断跟AI说"再往这个方向调整一下感觉"。

这个词之所以重要,是因为它把8.4提到的**productivity debate**推向了一个更极端、更两极化的争论:
- **乐观派**认为:这让"写软件"这件事的门槛大幅降低,任何有想法的人都能做出产品,创造力比语法知识更重要了
- **怀疑派**(通常是资深工程师)认为:这种做法在小玩具项目上没问题,但一旦放到正式的商业产品上,会累积大量没人真正理解、没人能安全维护的代码,一旦出现复杂bug或安全漏洞,会是灾难——这也直接呼应你在 **3.9 Security Engineering** 学过的概念:没人理解的代码,就是最大的安全隐患。

### 2.4 什么是 "Human-in-the-Loop Review"（人类参与审查）？

这是整个8.5 topic里,**最重要的一个平衡机制**,你一定要牢牢记住。

正因为agent可以自主执行一大串步骤,业界很快发现一个问题:如果完全不监督,agent很可能会:
- 误解你的需求,做出方向错误的东西
- 自己"发挥创意",改动了你根本不希望它碰的档案
- 生成看起来能跑,但其实有严重bug或安全漏洞的代码

**Human-in-the-loop review** 指的是:即使AI agent能够自主完成一大串工作,业界共识仍然是——**在关键的节点上,必须有真人工程师检查、批准,才能让改动真正生效**(比如你在 **6.1 Version Control** 学过的Pull Request code review,现在变成"审查AI写的代码"而不是"审查同事写的代码")。

你可以把这个理解成:agent是那个"装修承包商",但你不会让承包商拿了钥匙就完全不闻不问——你还是会去验收每一个阶段的成果,而不是等房子完全装修完才第一次去看。

这也是为什么这个topic的Scope特别写"included: 现在的AI状态",而"NOT required yet: 2026年以后的预测"——因为human-in-the-loop这个平衡点,到底会不会随着AI越来越强而慢慢消失,这是一个还在演变、没有定论的问题。

### 2.5 什么是 "Spec-Driven Development"（规格驱动开发）？

**Spec** 是 **specification**(规格/规格说明书)的缩写。

**Spec-driven development** 指的是一种新兴的开发方法论:

> 在叫AI agent动手写代码之前,先花时间把"你到底要什么"写成一份清楚、结构化的规格文件(spec)——包含这个功能要做什么、不做什么、边界情况(edge case)怎么处理、成功的标准是什么——然后让AI agent根据这份spec去执行。

为什么这个方法论在agentic coding时代变得特别重要?因为你在2.1想一下:agent的自主性越高,如果你一开始给的指令越模糊,它自己"发挥"出错的空间就越大。就像你请装修承包商,如果你只说"弄漂�亮一点",最后出来的成果可能完全不是你要的;但如果你先写清楚"我要北欧风、预算多少、客厅要有一个阅读角落",承包商做出来的东西才会真正符合你的期待。

这也让"写清楚的规格文件"这个能力,变成了在agentic coding时代特别珍贵的技能——这呼应了8.6会讲到的"skills shift: judgment over syntax"(判断力比语法知识更重要)这个概念。

## 3. Progressive Explanation:把整个故事串起来

### Phase 1(2024年初):从"补全"到"执行"的技术突破

2024年,几个关键的技术进展,让AI第一次真正具备"agent"的能力,而不只是"预测下一段文字":
- LLM的"推理"(reasoning)能力大幅提升,能够把一个复杂任务拆解成多个步骤
- LLM开始能够"呼叫工具"(tool use / function calling)——也就是说,AI不再只能"说话",它可以实际去执行代码、读写档案、查网路资料,然后把执行结果读回来,再决定下一步
- 这两者结合,才让"agent能自己跑完整个任务流程"这件事变成现实

这个阶段出现的代表性工具,包括能够自主完成整个coding任务的agent产品——你现在正在使用的Claude Code,就是这一波技术演进下的产物。

### Phase 2(2024下半年-2025年):Agentic coding进入主流工程团队

到了这个阶段,agentic coding不再是实验室的demo,而是开始真正进入公司的日常开发流程。工程师开始把整段整段的任务"委托"给agent处理,例如:
- "帮我把这个legacy(遗留)系统里所有用到的旧版函式库都升级到新版"
- "帮我看这个bug report,自己重现问题、找出根因、修好它"

同时,2025年"vibe coding"这个词开始在社群媒体上爆红,也把AI辅助开发的讨论从工程圈扩散到更广泛的大众和创业圈——很多没有传统工程背景的人,开始用这种方式做出真正能用的产品原型(prototype)。

### Phase 3(2025-2026年):行业开始建立"节制"机制

正因为agentic coding的自主性带来了实际的风险(生成的代码没人理解、安全漏洞、方向跑偏),整个行业在2025-2026年间,逐渐发展出一套"节制"agent自主性的最佳实践,主要就是前面讲的两个核心机制:
1. **Human-in-the-loop review** — 关键节点一定要有真人确认
2. **Spec-driven development** — 在动手之前先写清楚规格,减少agent"自由发挥"出错的空间

到2026年(也就是现在),整个行业对agentic coding的态度,已经从"这是天方夜谭"(2023年之前),变成"这是新手小工具的玩具"(2024年),再变成"这是严肃工程团队真正在用、但需要搭配审查机制的日常工作方式"(2025-2026年)。

这也直接带出下一个topic **8.6**要讲的东西:当AI agent能做的事情越来越多,工程团队的组织结构、工程师的角色定位、junior工程师还有没有存在的必要——这些都因为8.5讲的这个转变,而被迫重新被定义。

## 4. 5 个你必须完全理解的核心概念

### ① Agent(代理人)
**一句话定义**:一个能够自主观察现状、做决定、采取行动、检查结果,并重复这个循环直到完成任务的AI系统,而不只是回答单一问题。
**为什么重要**:这是区分"AI工具"(像8.4的Copilot/ChatGPT)跟"AI agent"(像现在的coding agent)最核心的分水岭,理解这个差异,才能理解整个行业2024年之后为什么变化这么快。
**自我测试题**:ChatGPT(8.4)跟一个coding agent最本质的能力差异是什么?
> **答案**:ChatGPT只能生成文字/代码建议,它没办法自己去实际执行这段代码、看执行结果、然后根据结果自己修正;coding agent可以完成"感知→决策→行动→检查"这一整个循环,自主执行多个步骤直到任务完成,不需要人在每一步都介入。

### ② Agentic Workflows(代理式工作流程)
**一句话定义**:AI agent处理任务时,实际跑过的那一套"感知、决策、行动、检查"反复循环的流程。
**为什么重要**:理解这个流程,能帮你判断什么样的任务适合委托给agent(可以拆解成清楚步骤、有明确检查方式的任务),什么样的任务还是需要人自己动手(需要复杂判断、没有明确对错标准的任务)。
**自我测试题**:为什么agentic workflow里"检查(check)"这一步特别重要,如果拿掉这一步会发生什么事?
> **答案**:检查这一步让agent知道自己刚刚采取的行动到底有没有成功、有没有引入新问题,如果拿掉这一步,agent会没有办法从错误中修正,可能会在错误的方向上一直往下做,累积出更多问题,最后交出一个表面完成、实际上充满bug的结果。

### ③ Vibe Coding(2025)
**一句话定义**:使用者只凭自然语言描述想要的"感觉",完全依赖AI agent生成、执行、部署代码,自己不深入理解底层代码的开发方式。
**为什么重要**:这代表AI大幅降低了"做出软件"的门槛,让更多没有工程背景的人能创造产品,但同时也带来了"没人真正理解的代码"这个风险,是理解这个时代最有代表性、争议也最大的一个现象。
**自我测试题**:为什么资深工程师普遍对"vibe coding"用在正式商业产品上感到担忧?
> **答案**:因为vibe coding产出的代码,使用者自己并不理解其中的逻辑,一旦这个产品需要长期维护、出现复杂bug、或存在安全漏洞时,没有人具备足够的理解去安全地修改或排查问题,风险会随着产品规模扩大而急速累积。

### ④ Human-in-the-Loop Review
**一句话定义**:即使AI agent能自主完成一连串工作,业界仍坚持在关键节点上,必须有真人工程师检查并批准,改动才能真正生效。
**为什么重要**:这是目前整个行业用来平衡"agent的高效率"跟"AI仍可能犯错"这两者之间风险的核心机制,理解这个机制能帮你判断一个团队使用AI的方式是否负责任、成熟。
**自我测试题**:"Human-in-the-loop review"跟8.4提到的"prompt-assisted development"里,人类扮演的角色有什么本质上的不同?
> **答案**:在prompt-assisted development里,人类是"每一步都在场的驾驶",几乎是逐行审查、逐行接受AI的建议;在human-in-the-loop review里,人类退到"关键节点才出现的审查者",让agent自主完成一整串步骤后,才在重要的检查点介入确认,人类介入的密度大幅降低,但介入的"节点"变得更关键。

### ⑤ Spec-Driven Development(规格驱动开发)
**一句话定义**:在让AI agent动手写代码之前,先把需求、边界情况、成功标准写成一份清楚的规格文件,再让agent照着执行的开发方法。
**为什么重要**:agent的自主性越高,一开始的指令模糊度造成的偏差就会被放大得越严重,写清楚spec的能力,变成了agentic coding时代里,判断力比单纯写程式语法更重要的具体体现。
**自我测试题**:为什么"spec-driven development"在agentic coding时代,比在8.4的prompt-assisted development时代更加关键?
> **答案**:因为在prompt-assisted development里,人类每一步都在场,就算指令有点模糊,人类马上就能发现方向不对并即时修正;但在agentic workflow里,agent会依据最初的指令自主执行一大串没有人监督的步骤,如果一开始的spec不够清楚,agent可能会在错误的方向上越走越远,等到人类介入检查时,已经产生大量需要重做的偏差工作。
