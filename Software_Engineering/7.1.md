# 7.1 — Everyday Work Vocabulary

## 1. 这个主题在大局中的位置 (High-Level Map)

想象你已经拿到 offer，第一天上班，坐进了工程师团队的 Slack channel 和 daily standup。

前面 1-6 章教你的是**技能** — Git、测试、CI/CD、系统设计、团队协作原理 (Conway's Law 等)。这些是你"能做什么"。

但从 Chapter 7 开始，我们教的是**社交生存技能** — 也就是"听得懂、说得出"。工程师团队有一套自己的黑话 (jargon)，这些词不是技术概念，而是**职场沟通的默认语言**。如果你不懂，你会在会议里一脸问号；如果你懂，你听起来就像"已经在这个圈子待了三年的人"。

7.1 是这套黑话的入门篇：**日常工作词汇**。之后的 7.2 (面试黑话) 和 7.3 (薪酬黑话) 会建立在这个基础上，越往后越"社会"。

---

## 2. 基础打底 (Fundamentals First)

为什么工程师需要一套自己的词汇？因为写代码的人每天都在做**权衡决策 (trade-off decisions)** — 要不要现在把代码写好、还是先求有再求好、要不要现在发布。这些决策太常见了，团队就发明了简短的词来快速表达复杂的意思。

打个比方：医生说"BP 120/80"，不会说"你的血压收缩压是一百二十毫米汞柱"。工程师也一样，会用简短的黑话来节省沟通时间。

我们从五个最核心的词开始，按照它们在真实工作流程里出现的**先后顺序**来讲，这样你会看到它们之间的因果关系。

## 3. 循序渐进讲解 (Progressive Explanation)

### 3.1 MVP (Minimum Viable Product) — 最小可行产品

**故事开场：** 假设你们公司想做一个"打车 App"。工程师理论上可以做到：多语言支持、深色模式、100 种支付方式、AI 客服... 但如果你要等这一切都做完才上线，可能要三年，而竞争对手一年就抢走了你的用户。

**MVP** 就是：只做能验证"这个产品到底有没有人要用"的**最小版本**。打车 App 的 MVP 可能只有：地图 + 叫车按钮 + 一种付款方式。没有花哨功能，但核心闭环能跑通。

这个概念最早由 Eric Ries 在《The Lean Startup》里系统化，现在几乎是所有科技公司的默认工作方式 (你在 Uber、Airbnb 的产品发展史里都能看到 MVP 的影子)。

> 工作中你会听到：*"Let's just ship the MVP first and iterate later."*（先上线最小版本，后面再迭代。）

### 3.2 Shipping / Launching — 发布上线

MVP 做好了，下一步就是 **"ship it"**。

"Ship" 这个词字面意思是"运输"，但在软件工程里，它的意思是：**把代码部署到生产环境 (production)，让真实用户能用上**。这个词其实源自硬件时代 — 以前软件是刻录在光盘上，装箱"运输 (ship)"到商店的，这个说法就一直保留到了云时代。

"Launching" 意思更接近，通常指一次**比较大、比较正式**的发布（比如全新产品线上线），而 "shipping" 可以指任何大小的发布，包括每天可能发生好几次的小更新。

> 工作中你会听到：*"We shipped the fix this morning."* / *"When are we launching this feature?"*

这个词很重要，因为**工程师文化用"有没有 ship"来衡量你的产出**，而不是"你写了多少行代码"或者"你花了多少小时"。这也是为什么求职时你要讲"我 shipped 了什么"，而不是"我学了什么"。

### 3.3 Tech Debt (技术债务)

**故事延续：** 你为了赶时间把 MVP shipped 出去了，用的是一些"能跑就行，但不够优雅"的写法 — 比如把逻辑硬编码 (hardcode) 在一个地方，本该拆成好几个函数的代码堆在一起。

这就像信用卡消费：现在用得爽（上线快），但以后要还利息（以后改代码会更痛苦、更容易出 bug）。这种"欠下的、还没还的代码质量"，就叫 **Tech Debt (技术债务)**。这个比喻是 1992 年 Ward Cunningham（Wiki 发明人之一）提出的，现在是全行业通用词。

Tech debt 不是"坏"的同义词 —— 很多时候**故意欠一点技术债是合理的商业决策**（比如为了赶上市场窗口）。关键是团队要"记账"，之后找时间"还债"。

> 工作中你会听到：*"This works, but it's tech debt — we'll need to refactor it later."*

### 3.4 Refactoring (重构)

那"还债"的动作，就叫 **Refactoring**。

Refactoring 的精确定义是：**在不改变代码外部行为（也就是功能、输出）的前提下，改善代码内部结构**。注意这个定义很重要 —— 如果你改代码的同时也改了功能，那不叫 refactor，那叫加新功能或者改 bug。

举例：把一个 500 行的函数拆成 5 个 100 行的小函数，功能完全不变，只是更好读、更好维护 —— 这才是 refactor。

> 工作中你会听到：*"I'm going to refactor this module before adding the new feature."*

这里正好呼应了 tech debt 的比喻：**tech debt 是"欠的债"，refactoring 是"还债的动作"**。这两个词几乎总是成对出现。

### 3.5 Bandwidth / Blockers — 带宽与阻塞

最后两个词，是团队协作中最常用的"状态汇报"词汇。

**Bandwidth（带宽）** 原本是网络工程术语，指网络能传输数据的容量。工程师借用它来形容"一个人手头能同时处理多少工作的精力/时间容量"。

> *"I don't have bandwidth to review your PR today."*（我今天没空审你的代码。）— 不是技术上做不到，是"精力容量"不够。

**Blockers（阻塞）** 指**卡住你、让你没法继续往前推进任务的障碍**。这个词在 Daily Standup 会议里几乎每天都会出现（还记得 6.4 讲过的 Scrum / Agile 流程吗？Standup 三个问题之一就是"你有没有 blocker"）。

> *"I'm blocked on the API keys — waiting for DevOps to provision them."*

这两个词的共同点：它们让你能**用一个词快速、精准地汇报状态**，而不需要长篇解释。这正是黑话存在的意义——效率。

## 4. 五个最重要的概念小结

**1. MVP (Minimum Viable Product)**
- **定义：** 只包含核心功能、能验证产品是否值得做的最简版本。
- **为什么重要：** 避免团队浪费几个月时间做没人要的功能，让产品能尽早接触真实用户反馈。
- **自测问题：** 如果你们要做一个"记账 App"，MVP 应该包含什么，不应该包含什么？
  **答案：** 应该包含：手动记一笔收支、查看列表。不应该包含：自动识别账单、多币种换算、社交分享——这些是后续迭代加的。

**2. Shipping / Launching**
- **定义：** 把代码部署到生产环境，让真实用户能够使用。
- **为什么重要：** 工程师文化以"有没有 ship 东西"来衡量产出和影响力，而不是投入的时间。
- **自测问题：** "Shipping" 和 "Launching" 有什么区别？
  **答案：** Launching 通常指较大、较正式的一次性发布（新产品/大功能上线）；Shipping 泛指任何规模的发布，可能一天发生多次。

**3. Tech Debt (技术债务)**
- **定义：** 为了求快而牺牲代码质量，导致以后需要花更多时间修复或维护的"隐性成本"。
- **为什么重要：** 让团队能诚实地讨论"现在走捷径的代价"，而不是假装代码质量没有问题。
- **自测问题：** 欠技术债是不是永远都不该做的坏事？
  **答案：** 不是。为了赶上重要的市场窗口或验证 MVP，故意欠一些技术债常常是合理的商业决策，关键是之后要计划好如何"还债"（refactor）。

**4. Refactoring (重构)**
- **定义：** 在不改变代码外部功能的前提下，改善代码内部结构的过程。
- **为什么重要：** 是"偿还技术债"的具体行动，让代码在长期内更容易维护、扩展，减少未来的 bug。
- **自测问题：** 如果你在改代码结构的同时，也顺便修了一个 bug，这还算是 refactor 吗？
  **答案：** 不算。Refactor 的定义要求功能（外部行为）完全不变；修 bug 属于改变了行为，应该算作 bug fix，最好和 refactor 分开提交（separate commit）。

**5. Bandwidth / Blockers**
- **定义：** Bandwidth 指一个人当前能投入工作的时间/精力容量；Blocker 指阻止任务继续推进的障碍。
- **为什么重要：** 让团队能用简短、精准的词快速同步状态，尤其在 Daily Standup 这种高频率的沟通场合中提升效率。
- **自测问题：** 如果同事说"I have no bandwidth this week"，代表什么意思？
  **答案：** 代表这位同事本周时间/精力已经被排满，没有额外容量接手新任务或帮忙审代码，不是说他技术上做不到。
