# 1.2 — The Software Development Lifecycle (SDLC)

## 1. Big Picture：这个知识点在哪里？

回忆一下 [[1.1 — Software Engineering vs Programming]]：我们说 Software Engineering 跟 Programming 最大的差别，就是它要考虑**时间**、**规模**、**团队**、**真实世界的约束**。

那么问题来了——如果一个软件项目不是"写完就结束"，而是要经历"从无到有、再到长期维护"的完整过程，那这个过程到底长什么样？有没有一套固定的步骤？

答案是：有。这套步骤，就叫做 **SDLC —— Software Development Lifecycle（软件开发生命周期）**。

这是整个第 1 章 Foundations 的第二块地基：
- 1.1 回答了"什么是 software engineering"
- **1.2 回答了"一个软件项目的一生，会经过哪些阶段"**

⚠️ 注意：这一节我们只学**阶段的名字和它们的意思**（vocabulary）。至于"具体怎么排这些阶段的顺序、要不要一次做完、能不能来回跳"——这些是**方法论（methodology）**的问题，比如 Waterfall、Agile，会在后面的 6.2、6.3 详细讲。现在先别管"怎么做"，只需要认识"有哪几步"。

---

## 2. Fundamentals First：为什么需要一个"生命周期"？

想象你今天要做一个手机 App。你会怎么开始？

直接打开电脑开始写代码？—— 大概率会写到一半发现："诶，用户到底想要什么功能？" "这个页面要长什么样？" "写完了，怎么知道它是对的？" "写完之后呢，就不管了吗？"

**SDLC 的存在，就是为了回答这些问题，给"做软件"这件事一个结构化的顺序，确保不会有环节被遗漏。**

这跟盖房子很像：你不会一边打地基一边才想"诶这房子到底要盖几层"。你会先规划、再设计、再施工、再验收、最后才是入住和维修。

软件也是一样的逻辑，只是换了个说法。

---

## 3. Progressive Explanation：五个阶段，一个个来

SDLC 通常被拆成 5 个核心阶段。我们按顺序、从头到尾走一遍。

### 阶段 1：Requirements Gathering（需求收集）

**这是什么：** 在写任何一行代码之前，先搞清楚——"我们到底要做什么？为谁做？解决什么问题？"

举例：如果要做一个"外卖 App"，你需要先搞清楚：
- 用户要能做什么？（下单、付款、追踪订单？）
- 商家要能做什么？（上架菜单、接单？）
- 有没有平台限制？（比如预算、上线时间？）

**为什么这一步不能跳：** 如果你没搞清楚需求就直接写代码，很可能做出一个"技术上很完美，但没人要用"的东西。这是软件项目失败最常见的原因之一。

### 阶段 2：Design（设计）

**这是什么：** 需求确定后，工程师要开始规划"这个系统要怎么搭"。

包括：
- 整体架构长什么样？（比如：前端跟后端怎么沟通？数据存在哪里？）
- 数据库要存哪些资料？
- 各个模块（module）之间怎么分工？

**类比：** 这就像盖房子前画的建筑蓝图——你要先知道承重墙在哪、水电怎么走线，才能开始动工。跳过这一步直接施工，房子大概率会歪。

### 阶段 3：Implementation（实现，也就是"写代码"）

**这是什么：** 终于到了大家最熟悉的部分——工程师根据设计蓝图，把逻辑写成实际能跑的代码。

**重要认知：** 注意它在 SDLC 里只是 5 个阶段中的**第 3 个**，而不是全部。这也呼应了 1.1 讲的——"写代码"（programming）只是"软件工程"整个流程里的一环，不是终点。

### 阶段 4：Testing（测试）

**这是什么：** 代码写完了，不代表它是对的。Testing 阶段就是系统性地检查："这段代码，在各种情况下，真的按照需求在运作吗？"

包括检查：
- 正常情况下功能对不对（比如下单流程走得通吗）
- 极端情况会不会出错（比如网络突然断了怎么办）
- 会不会不小心弄坏了其他本来正常的功能

**为什么重要：** 这一步是在正式把东西交给用户之前的"最后一道防线"。跳过测试，等于让用户帮你当"人肉测试员"——代价通常很高（想象银行 App 少算了利息）。

### 阶段 5：Deployment（部署）

**这是什么：** 把测试通过的软件，正式发布给真实用户使用的过程。

比如：把 App 上架到 App Store，或者把网站程序发布到线上服务器，让全世界的人都能访问。

**一个直觉：** 这一步之后，软件才算"活"了——从工程师电脑里的一堆代码，变成真实世界中被人使用的产品。

### 阶段 6：Maintenance（维护）

**这是什么：** 软件上线后，故事并没有结束。

维护包括：
- 修复上线后发现的 bug
- 根据用户反馈调整功能
- 随着时间更新（比如系统升级、安全补丁）

**呼应 1.1：** 还记得我们在 1.1 讲的 **maintainability（可维护性）** 吗？这就是它派上用场的地方——软件的"维护阶段"往往比"开发阶段"持续得更久。一个软件如果一开始写得难以维护，这个阶段就会变得非常痛苦。

## 4. 一张图看懂整个流程

```
Requirements  →  Design  →  Implementation  →  Testing  →  Deployment  →  Maintenance
   (要做什么)      (怎么搭)      (写代码)         (测试对不对)    (正式发布)      (长期维护)
```

**一个关键提醒：** 这张图看起来像是"一步做完才能做下一步"的直线流程（这其实就是后面 6.2 会讲的 Waterfall 模型的思路）。但现实中很多团队并不是严格按这个顺序走一次就结束，而是会用不同的方式重复、交错这些阶段（这就是 6.3 Agile 要解决的问题）。**现在你只需要记住这几个阶段"叫什么、做什么"，还不需要知道"该怎么组织它们"。**

---

## 5. 为什么这个知识点在现实世界很重要？

- 面试中常被问到"walk me through how you'd build X"，本质上就是在考你有没有 SDLC 的思维框架，而不是让你直接跳去讲代码细节。
- 职场上，PM、设计师、工程师、QA 各自负责 SDLC 里不同的阶段，理解这个框架能让你知道"我现在在流程的哪一步，该跟谁沟通"。
- 很多新手最大的错误，就是把 SDLC 直接压缩成"Implementation"一步——想到什么功能就直接写代码，跳过需求分析和设计，最后往往要推倒重来。

## 6. 五个你必须掌握的核心概念

### ① SDLC（Software Development Lifecycle）
- **一句话定义**：SDLC is the set of stages a software project moves through, from figuring out what to build to keeping it running after launch.
- **为什么重要**：它是理解"软件是怎么被造出来的"最基本的框架，之后学的所有方法论（Waterfall、Agile、Scrum）都是"怎么组织这些阶段"的不同方式。
- **自测问题**：如果有人问你"软件项目除了写代码还包括什么"，你能列出几个阶段？
  **答案**：Requirements gathering、Design、Implementation、Testing、Deployment、Maintenance。

### ② Requirements Gathering（需求收集）
- **一句话定义**：Requirements gathering is figuring out what the software needs to do and for whom, before writing any code.
- **为什么重要**：这是避免"做出没人要的东西"最重要的一步，很多项目失败不是因为代码写得差，而是需求从一开始就搞错了。
- **自测问题**：为什么"技术上完美，但没人用"的软件也算失败？
  **答案**：因为软件的价值在于解决真实需求，如果一开始没搞清楚用户要什么，再好的代码也没有意义。

### ③ Design（设计阶段）
- **一句话定义**：Design is planning how the system will be structured (architecture, data, modules) before implementation begins.
- **为什么重要**：设计决定了系统未来好不好扩展、好不好维护——这跟 1.1 提到的 maintainability 直接相关，设计得不好，后面改起来会非常痛苦。
- **自测问题**：为什么不能跳过设计，直接开始写代码？
  **答案**：因为没有整体架构规划，容易出现模块间逻辑混乱、数据结构不合理，之后要大幅返工，成本比先设计高得多。

### ④ Testing（测试阶段）
- **一句话定义**：Testing is systematically checking that the code actually behaves correctly, including in edge cases, before it reaches real users.
- **为什么重要**：这是保护真实用户不被 bug 影响的最后防线，尤其在金融、医疗这类高风险领域，测试不足的代价可能非常严重。
- **自测问题**：如果代码"看起来能跑"，为什么还需要专门的 Testing 阶段？
  **答案**：因为"看起来能跑"通常只验证了最常见的正常情况，Testing 阶段会系统性检查各种边界情况和潜在错误，这些是肉眼很难全部想到的。

### ⑤ Maintenance（维护阶段）
- **一句话定义**：Maintenance is the ongoing work of fixing bugs, updating, and adapting software after it has been deployed.
- **为什么重要**：大部分软件的生命周期里，"维护"所花的时间往往远超过最初的开发阶段，因此代码是否易维护（参考 1.1 的 maintainability）直接影响长期成本。
- **自测问题**：软件正式上线（deployment）之后，这个项目就算"完成"了吗？
  **答案**：不算。上线只是把软件交给真实用户的开始，之后还会持续进入 Maintenance 阶段，处理 bug、更新和调整。
