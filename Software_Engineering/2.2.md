# 2.2 — The Mainframe Era (1950s–1970s)

## 1. High-Level Map（这个主题在大局中的位置）

我们上一节（[[2.1 — The Software Crisis (1968)]]）讲了 **"software engineering"** 这个词是怎么诞生的——因为软件项目老是延期、超支、写出一堆 bug。

但你有没有想过：**在 1968 年那场危机爆发之前，软件世界长什么样？** 谁在写代码？写给谁用？用什么设备写？

这就是 2.2 要讲的东西。这是整个"软件历史"系列（Chapter 2）的**起点**——第一个时代。之后的每一个时代（PC 时代、互联网时代、移动云时代、AI 时代）都是在"打破"或"颠覆"这个最早的时代所建立的规则。所以理解 Mainframe Era，等于拿到了整条历史时间线的"第一块拼图"。

---

## 2. Fundamentals First（从零开始，每个术语都讲透）

### 2.1 什么是 Mainframe（大型主机）？

想象一下：现在你写代码，用的是一台自己的笔记本电脑，对吧？1950-1970 年代完全不是这样。

**Mainframe** 是一台**巨大、昂贵、占满整个房间**的计算机。它不是给"一个人"用的，而是给**一整个公司或政府机构**共用的。

- 体积：可以塞满一整间空调房（计算机太热，需要专门降温）
- 价格：动辄几百万美元（当年的几百万美元，换算到现在更夸张）
- 使用者：只有大公司（银行、保险公司、航空公司）、政府、军方、大学才买得起

**类比**：如果现在的电脑是"每人一辆车"，那 Mainframe 时代就是"整个城市只有一辆巴士，所有人排队上车"。

### 2.2 谁在"开车"？—— Programmers as Operators（程序员＝操作员）

这里有个关键认知：在这个时代，**"程序员"不是一个体面的、有创造力的职业**，而更像是"机器操作员"。

为什么？因为：
1. 计算机太贵、太稀缺，普通人根本摸不到
2. 写程序更像是在"伺候机器"——按照严格的格式、严格的步骤操作
3. 当时甚至没有"software engineer"这个职称（还记得 2.1 讲的吗？这个词要等到 1968 年才被发明出来）

所以早期的程序员，工作内容更接近今天的"工厂技术员"，而不是今天硅谷工程师那种"设计师+建筑师"的角色。

### 2.3 Punch Cards（打孔卡）——没有键盘,没有屏幕

这是最颠覆你想象的部分：**当时的程序员不是对着屏幕打字的**。

**Punch Card（打孔卡）** 是一张硬纸卡片，程序员要在上面**打孔**来表示指令和数据。每一张卡片，通常代表**一行代码**。

流程是这样的：
1. 程序员先在纸上手写代码
2. 拿去给专门的"打孔员"（keypunch operator），或自己用打孔机，把代码"翻译"成卡片上的孔洞
3. 把一叠卡片（可能几百张、几千张）按顺序排好，交给计算机中心
4. **等待**——通常是几个小时甚至一整天
5. 计算机读取卡片，跑完程序，打印出结果
6. 如果代码有一个小错误（比如打孔打错一个洞），对不起，**重新来一遍**

想象一下：你现在写代码，改一个字母要"等一整天"才知道对不对。这就是当年的现实。这也是为什么"软件危机"（2.1）会发生——因为写代码的**反馈循环极其缓慢**，错误成本极高。

### 2.4 Batch Processing（批处理）——为什么不能"实时"用？

因为 Mainframe 太贵、太稀缺，**不可能一个人独占一整天**。所以当时用的模式叫 **Batch Processing（批处理）**：

- 很多人把自己的"打孔卡程序"交上去
- 计算机中心把这些任务**打包（batch）**，一批一批地排队处理
- 你交上去的程序，可能要等别人的任务先跑完，才轮到你

**类比**：这就像去政府机关办事要"排队叫号"，而不是像现在用手机 App 立刻搞定。你交了申请（程序），然后**回家等通知**（等打印结果）。

这和我们今天"实时"、"即时反馈"的开发体验（写代码、按 Ctrl+S、立刻看到结果）完全是两个世界。

### 2.5 IBM Dominance（IBM 的垄断地位）

这个时代几乎可以说是 **IBM 的时代**。

- IBM 是当时全球最大的计算机制造商，市场份额一度接近垄断
- 当时甚至有句话叫 "**IBM and the Seven Dwarfs**"（IBM 和七个小矮人）——形容 IBM 大到像巨人，其他计算机公司都像小矮人
- 软件当时甚至不是"单独卖钱"的产品——**软件是跟着硬件"免费赠送"的**！你买了 IBM 的大型主机，软件是配套服务，而不是像今天这样单独开发、单独销售

这点很重要：**"软件产业"作为一个独立的商业模式，在这个时代还没有真正形成**。要等到下一个时代（[[2.3 — The Personal Computing Era (1980s)]]），软件才开始"独立"成为可以打包销售的商品。

## 3. Progressive Explanation（把这些概念串起来）

好，现在把这几个概念串成一个完整的故事：

> 1950-1970 年代，计算机是**极度稀缺、极度昂贵**的资源，只有大机构（政府、银行、大学）买得起一台 **Mainframe**。因为这么贵，**IBM** 几乎垄断了整个市场，软件也只是硬件的"附赠品"，还没成为独立产业。
>
> 因为一台机器要服务一整个机构的所有需求，大家只能用 **Batch Processing** 的方式排队使用——你把程序交上去，等着轮到你。
>
> 而"把程序交上去"这个动作，具体是通过 **Punch Cards（打孔卡）** 完成的——没有键盘、没有屏幕，写错一个字母都要重新打孔、重新排队、重新等待。
>
> 在这种环境下，写程序的人（Programmer）更像是伺候机器的**操作员（Operator）**，而不是今天我们认知中那种"设计系统、做决策"的工程师。这也解释了为什么"software engineering"这个专业术语，要等到 1968 年（[[2.1 — The Software Crisis (1968)]]）才被正式提出——因为在此之前，写代码根本还没被当成一门"工程学科"来看待。

## 4. 5 个你必须完全掌握的核心概念

### ① Mainframe（大型主机）
- **定义**：一台巨大、昂贵、由整个机构共享使用的计算机，是 1950-70 年代计算的核心载体。
- **为什么重要**：它解释了为什么早期软件是"机构级"而非"个人级"的产物——这个假设直到 PC 时代（2.3）才被打破，理解这一点你才能看懂后续所有历史章节的"转折点"。
- **自测题**：为什么 1970 年代之前，一般人不可能拥有自己的电脑？
  **答案**：因为 Mainframe 体积巨大、造价高达数百万美元，只有政府、银行、大学等大机构才负担得起，个人根本买不起也用不到。

### ② Batch Processing（批处理）
- **定义**：把多个程序任务打包排队、依次交给计算机处理，而不是让用户实时互动的运作方式。
- **为什么重要**：它导致了极慢的反馈循环（写代码到看结果可能要等一整天），这正是后来"软件危机"爆发的根源之一——错误发现得太晚，成本极高。
- **自测题**：如果你在 Batch Processing 时代写程序时打错一个字，你要多久才会知道？
  **答案**：可能要等几个小时到一整天，等你的批次被处理完、打印出结果后才会发现错误，然后必须重新排队。

### ③ Punch Cards（打孔卡）
- **定义**：一种在硬纸卡片上打孔来表示程序指令和数据的输入方式，是当时"写代码"的物理载体。
- **为什么重要**：它体现了当时开发环境的原始程度——没有键盘、没有屏幕、没有"撤销"，这和今天在 VS Code 里秒级看到语法错误提示形成极端对比，帮你理解"开发者体验（DX）"是如何一步步进化的。
- **自测题**：一张打孔卡通常代表程序的什么单位？
  **答案**：通常代表一行代码（one line of code）。

### ④ IBM Dominance（IBM 的垄断）
- **定义**：IBM 在这个时代几乎垄断了大型主机市场，软件当时是随硬件"免费附赠"，还不是独立产业。
- **为什么重要**：它标志着"软件产业"作为独立商业模式尚未诞生——理解这点，才能看懂 2.3 (PC 时代) 里"软件开始能单独打包销售"这个巨大转折的意义。
- **自测题**：在 Mainframe 时代，软件是怎么"卖"给客户的？
  **答案**：软件不是单独销售的商品，而是买硬件（Mainframe）时随附赠送的服务，还没有形成独立的软件产业。

### ⑤ Programmers as Operators（程序员＝操作员）
- **定义**：早期的"程序员"角色更接近伺候机器、严格按流程操作的技术员，而非今天拥有设计权和决策权的工程师。
- **为什么重要**：这解释了"software engineering"作为一门正式学科为什么迟至 1968 年才被命名（见 [[2.1 — The Software Crisis (1968)]]）——因为在此之前，这份工作的社会地位和性质都更接近"操作"而非"工程"。
- **自测题**：为什么当时的程序员更像"操作员"而不是"工程师"？
  **答案**：因为计算机极度稀缺昂贵，写程序的工作被限定在严格的操作流程内（打孔、排队、等待），缺乏今天工程师那种自主设计系统、做架构决策的空间。
